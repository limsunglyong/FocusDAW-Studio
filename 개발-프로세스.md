## 1. 목표

파일 기반 트랙과 실시간 입력 녹음 트랙의 역할을 분리한다.

- 파일 트랙은 프로젝트 시작점 `00:00`에 고정된 Stem/Source 중심으로 유지한다.
- `Audio In` 트랙은 여러 녹음 source와 이동 가능한 clip, Take, Comp를 다루는 별도 트랙으로 확장한다.
- Settings 화면에 `Device Setup`을 추가해 mic 입력, line 입력 등 audio interface의 input source를 선택한다.
- 장기적으로 `Join Edits`, `Consolidate Clips`, `Merge Tracks`를 명확히 구분한다.
- 녹음은 메모리 버퍼가 아니라 JUCE 네이티브 엔진의 스트리밍 WAV 기록을 기준으로 설계한다.

## 3. 작업 원칙

1. 기존 프로젝트는 손실 없이 열려야 한다.
2. 파일 트랙은 기본적으로 `00:00` 고정이며 이동 불가로 유지한다.
3. Audio In과 Bounce 트랙만 clip 이동, trim, delete range를 허용한다.
4. 녹음 원본 WAV는 즉시 삭제하지 않고 `Unused` 관리 후 별도 정리 명령에서 삭제한다.
5. 오디오 콜백에서는 파일 쓰기, JSON 처리, 동적 메모리 할당, mutex 대기를 하지 않는다.
6. **웹 엔진은 모델·파형·Export fallback 계층이고, 가청 출력은 네이티브가 담당한다. 두 엔진은 같은 clip layout을 공유해야 한다.** (2026-07-15 재서술 — 배포 방침 확정 반영)
   - 배포는 **네이티브 Electron 기준**이며 별도 웹 배포판은 없다. 네이티브 연결 시 웹 출력은 `LocalDAW.setOutputMuted(true)`로 음소거되므로, **웹 재생 경로의 타이밍 정밀도(루프 되감기 지터 등)는 요구 사항이 아니다.**
   - 그러나 `audio-engine.js`(`LocalDAW`)는 "웹 버전"이 아니라 앱 내부의 필수 계층이므로 **제거 대상이 아니다**: ① `window.DAW`는 `DAWProxy`로 Bridge에 없는 모든 속성을 `LocalDAW`에 위임한다(UI 전체가 의존), ② `track.buffer`/`peaks`/`sources`/`clips` 등 데이터 모델과 파형 소스가 여기 있다, ③ Phase 5 레이아웃 베이크(`_bakeTrackLayout`)를 수행하는 주체가 웹 엔진이다(네이티브는 평탄화 결과를 받아 재생만 한다), ④ 네이티브 부재·크래시 시 `switchToLocal()`이 웹 출력을 되살리는 안전망이다.
   - **`switchToLocal()` 안전망은 배포본에서도 유지한다.** 네이티브가 유일한 출력인 만큼 이게 없으면 엔진이 죽는 순간 무음 앱이 된다. 코드가 이미 있어 유지 비용은 없다.
   - 따라서 이 원칙이 요구하는 것은 **clip layout 일치**(웹 `renderMix` fallback과 파형 표시가 네이티브 재생과 같은 배치를 보여줄 것)이지, 두 재생 경로의 샘플 단위 동일성이 아니다.
7. 입력 장치와 input source 선택은 트랙별 임시 UI가 아니라 Settings의 `Device Setup`에서 먼저 관리하고, Audio In 트랙은 그 설정을 기본값으로 가져온다.

## 4. 단계별 계획

### Phase 1 — Source/Clip 모델 정리

목표: Audio In 녹음 전에 트랙 데이터 구조를 확장한다.

상태: 완료(v1.17.6). 구현 기록은 [앱개발.md](앱개발.md)의 `Audio In 개발 Phase 1 — Source/Clip 모델 기반 정리` 항목에 보관한다.

- `track.kind`: `file | audioIn | bounce` 도입
- `track.sources[]`와 `track.clips[]`를 표준 구조로 정리
- 기존 프로젝트의 `buffer + clips`를 `sources[0] + sourceId` 형태로 migration
- 프로젝트에 `schemaVersion` 추가
- 프로젝트 길이 계산을 모든 clip의 `start + duration` 기준으로 변경
- `file` 트랙은 `lockedToZero: true`와 시작점 0 규칙 유지

완료 기준:

- 기존 `.focus` 프로젝트가 손실 없이 열린다.
- 기존 파일 트랙의 재생, seek, repeat, export 결과가 변경 전과 동일하다.
- migration 이후 저장한 프로젝트를 다시 열어도 파일 경로와 clip 참조가 유지된다.

### Phase 2 — File Tracks 그룹 기능 마무리

목표: 이미 구현된 접기/펼치기 기능을 계획 범위에 맞춰 안정화한다.

상태: 완료(v1.17.16). File Tracks 그룹 헤더 상태 표시, 파일 트랙 다중 선택, 단일 선택 해제, `Merge Tracks...` 진입점 설계를 반영하고 B-216 사용자 확인을 완료했다.

- `File Tracks` 그룹 헤더의 표시 정보 점검: 트랙 수, Mute/Solo 상태, 그룹 레벨
- 접힌 상태에서 재생, Export, Mute/Solo 동작 무회귀 확인
- 파일 트랙 2개 이상 선택 시 `Merge Tracks...` 진입점 설계
- 다중 선택 규칙: `Ctrl/Shift + Click`
- 선택 상태가 track reorder, collapse, delete와 충돌하지 않도록 정리

완료 기준:

- 접힌 파일 트랙은 파형 렌더링 비용을 줄이면서도 상태 정보를 충분히 보여준다.
- 접기/펼치기와 오디오 상태가 분리되어 프로젝트 사운드가 바뀌지 않는다.

### Phase 3 — Merge Tracks / Bounce

목표: 여러 파일 트랙을 비파괴적으로 새 Bounce 트랙으로 합친다.

상태: 완료(v1.17.17). 선택 트랙 오프라인 렌더, `kind: "bounce"` 트랙 생성, 첫 번째 선택 트랙 원본 폴더의 `Bounces` WAV 저장, 전체 프로젝트 범위, Mono/Stereo, 원본 유지/Mute/삭제 옵션의 1차 구현을 반영했다. `Loop Range` 옵션은 제거하고, `Create Bounce` 즉시 Rendering 로딩 화면으로 전환되도록 보완했다. Merge Tracks 모달에는 Bounce 트랙 이름 입력칸과 버튼 영역 구분선을 추가했다. 비활성 Merge 버튼은 금지 커서 없이 기존 안내 말풍선만 표시하도록 정리했으며, Undo/Redo 복원과 마지막 1개 트랙 삭제 Undo 비활성화까지 B-217에서 사용자 확인을 완료했다.

- `mergeTracks(trackIds, options)` API 신설
- 기존 `joinClips()`와 명확히 분리
- Offline render 옵션 설계:
  - 전체 프로젝트 범위
  - Mono/Stereo
  - Gain/Pan/automation/FX 포함 여부
  - 원본 처리: 유지, Mute, 삭제
- 결과 파일을 첫 번째 선택 트랙 원본 폴더의 `Bounces`에 저장
- 결과 트랙은 `kind: "bounce"`로 추가
- 기본값은 `원본 유지 + Mute`

완료 기준:

- Bounce 결과가 동일 조건의 Export 결과와 청감/레벨상 일치한다.
- 원본 트랙 유지, Mute, 삭제 옵션이 Undo/Redo와 충돌하지 않는다.

### Phase 4 — 단일 Audio In 녹음

목표: 루프/Take 이전에 안정적인 단일 녹음 파이프라인을 만든다.

상태: 완료(핵심 파이프라인, v1.18~v1.20). Device Setup, 입력 장치/채널 선택, `+ Audio In` 트랙, 트랙별 input gain·ARM·MON, JUCE `AudioDeviceManager` 입력 확장, 입력 미터·limiter GR 미터, writer thread + lock-free FIFO 스트리밍 WAV 기록, start/stop/cancel 녹음, 녹음 후 source/clip 추가까지 구현·사용자 확인 완료. 잔여 2건: **① Input source(mic/line/Hi-Z) 선택 → WASAPI 한계로 skip**(사용자 협의; ASIO 드라이버 필요), **③ `.wav.part`→`.wav` 확정 스테이징 → Phase 7 정리 도구로 이관 보류**(현재는 `.wav` 직접 기록 + 정상 Stop 확정 / Cancel 삭제로 안전). ②Input latency 표시(v1.20.7)·④GR 미터(v1.20.6)는 완료. 상세 [앱개발.md](앱개발.md) "Phase 4 점검".

- 기존 Settings 화면에 `Device Setup` 섹션 추가
- Audio input device 선택: 내장 마이크, USB 마이크, audio interface 등
- Input source 선택: mic 입력, line 입력, instrument/Hi-Z 입력 등 장치가 제공하는 source (⛔ WASAPI가 per-source를 노출하지 않아 skip — 사용자 협의. 인터페이스 구분은 입력 장치 선택으로 대체)
- Input channel 선택: Mono Input 1, Mono Input 2, Stereo 1-2 등
- Sample rate, buffer size, input latency 표시 및 필요 시 설정
- 선택한 device/source/channel을 Audio In 트랙 생성 시 기본 입력으로 사용
- `+ Audio In` 트랙 생성
- 트랙별 input gain, Arm, Monitor UI
- JUCE `AudioDeviceManager` 입력 설정 확장
- 입력 미터와 limiter gain reduction 미터 추가
- Limiter 기본 On, ceiling `-1.0 dBFS`
- 녹음은 `.wav.part`로 쓰고 정상 종료 후 `.wav`로 확정 (⏸ 확정 스테이징은 Phase 7 정리 도구로 이관 보류 — 현재는 최종 `.wav`에 직접 스트리밍 기록 + 정상 Stop=헤더 확정 / Cancel=삭제)
- writer thread와 lock-free FIFO 구성
- `startRecording`, `stopRecording`, `cancelRecording` 명령 도입
- 녹음 완료 후 source/clip을 Audio In 트랙에 추가

완료 기준:

- 30분 이상 녹음에서 드롭아웃과 지속적인 메모리 증가가 없다.
- 녹음 중지 후 즉시 프로젝트 타임라인에서 재생된다.
- Settings에서 선택한 mic/line/audio interface 입력이 Audio In 트랙에 정확히 반영된다.
- 입력 장치 없음, 장치 분리, 디스크 오류가 사용자에게 명확히 표시된다.

### Phase 5 — Audio In 클립 편집

목표: Audio In과 Bounce clip을 시간축에서 편집할 수 있게 한다.

상태: 완료(v1.21.0). **전략 B — 레이아웃 베이크** 채택: `track.clips[]`를 비파괴 진실원본으로 두고, 편집 시 clips를 평탄화한 새 `t.buffer`를 즉석 렌더해 `버퍼시간 == 타임라인시간` 불변식을 유지 → 기존 단일소스 재생/Export/네이티브 `loadTrack` 경로를 그대로 재사용. Stage 1(엔진 코어: 편집 API + 겹침 clamp + 파일트랙 lock)·Stage 2(네이티브 동기화 + Export 정합)·Stage 3(UI: 드래그 이동/trim/삭제/복사, 드래그당 Undo 1개)·Stage 4(`deleteRange` Punch 훅) 구현 완료, 사용자 클립 이동 시험에서 발견된 2건(baked 버퍼를 소스 인덱싱으로 샘플링해 생긴 빈 파형, 재시작 후 미복원) 수정 반영. ⚠️ **베이크 도입으로 `track.buffer`에 인덱싱 이원성이 생겼다**(`track._layoutBaked` 플래그로 구분) — 새 파형/재생/피크 코드 작성 전 [앱개발.md](앱개발.md) "상시 개발 노트" 필독. 잔여 검증 2건(웹/native Export A/B, 재생 중 native 교체)은 [시험.md](시험.md) T-1.21.0-*. 상세 [앱개발-아카이브.md](앱개발-아카이브.md) v1.21.0.

- `moveClip(trackId, clipId, newStart)`
- `trimClipStart`, `trimClipEnd`
- `deleteClip`, `deleteTake`
- `deleteRange(trackId, from, to)` — 구간을 비우고 그 자리에 새 clip을 배치할 수 있게 설계(Phase 6 Punch Recording 재사용 훅)
- clip 복사/붙여넣기/duplicate (Q3 결정, 2026-07-14): 같은 `sourceId` 참조 새 clip을 다른 `start`에 배치하는 비파괴 방식. 붙여넣기는 playhead/대상 위치 삽입
- clip gain을 트랙 volume과 분리
- clip 경계에 기본 3~5ms micro fade 적용 (crossfade 아님 — crossfade는 Phase 7 Comp, Q1 결정)
- drag 종료 시 Undo를 한 번만 생성
- 같은 트랙 내 clip 겹침은 1차 버전에서 방지

완료 기준:

- 이동/trim/split/delete 후 seek, loop, export 결과가 일치한다.
- clip 경계에서 click/pop이 들리지 않는다.
- 파일 트랙은 여전히 이동 불가이고, Audio In/Bounce만 이동 가능하다.

### Phase 6 — 루프 Take 녹음

목표: 반복 구간 녹음 시 회차별 Take를 sample-accurate하게 생성한다.

상태: **완료(v1.35.0, 2026-07-19)**. 착수 전 조사·결정 완료(2026-07-15) 후 **Stage 0(v1.24.0, loop range)** · **Stage 1(v1.25.0, 상태머신)** · **Stage 2(v1.26.0, 메트로놈)·2b(v1.27.0, 프리롤)** · **Stage 3a(v1.29.x, Take 모델)·3b(v1.30.0, 루프 Take)** · **Stage 4(v1.31.0, Take Lanes UI)** · **Stage 5(v1.33.0/1, 입력 지연 수동 오프셋)+5-B(v1.34.0~2, Recording Offset Cal.)** · **Stage 6(v1.35.0, Punch Recording — Repeat 구간 재사용 + 구간 교체)** 순으로 구현. Stage 0~5-B는 전부 사용자 시험 통과, Stage 6은 시험 대기(T-1.35.0-1). 상세 [앱개발.md](앱개발.md) "[Phase 6] 착수 전 코드 조사 결과" F1~F6 / Q1~Q4 / 예상 Stage 구성 + 각 버전 항목.

주요 결정(2026-07-15): **Q1** 루프 정밀도 = 네이티브만 sample-accurate(웹은 현행 폴링 유지, 녹음이 native 전용이므로 충분) · **Q2** count-in = **메트로놈 클릭 신설**(native 생성, 모니터 전용·녹음 파일 미포함) · **Q3** Take 모델 = `activeTakeId` + **활성 Take clip만 bake**(겹침 금지 판정은 활성 레인 내부로 한정) · **Q4** 지연 보정 = 드라이버 추정치 + **수동 오프셋 UI**.

- **✅ Stage 0 (선행 과제) — 네이티브 sample-accurate loop range** — **완료(v1.24.0, 2026-07-17 · 사용자 시험 통과)**: `LoopAudioSource`를 `mixerSource → LoopAudioSource → Gain → MasterEffects` 사이에 삽입해 오디오 콜백에서 루프 끝을 샘플 단위로 감지·블록 분할·되감기하고, `setLoopRange` 명령을 신설했으며, JS 폴링 되감기(30ms tick + position 핸들러)를 제거했다. 매 회차 길이 = `floor((end-start)×출력레이트)`로 고정 → 62초 연속 루프 드리프트 −0.1ms 실측. 상세 [앱개발.md](앱개발.md) v1.24.0 · 시험 [시험.md](시험.md) T-1.24.0-*. 아래는 착수 전 문제 기록이다. 현재 Repeat 구간 반복은 **두 엔진 모두 JS 폴링 + seek 되감기**이고(웹 = rAF `_loop()` 재생 재시작, 네이티브 = bridge `setInterval` 30ms 폴링 후 `seek`), **네이티브 C++에는 loop range 개념 자체가 없다**(곡 전체 `setLooping`만). 이 구조로는 아래 완료 기준 3개(Take 길이 = 루프 sample 수, 누적 밀림 없음, 블록 중간 경계 분할)를 **원리적으로 충족할 수 없다**. 오디오 콜백에서 루프 끝을 샘플 단위로 감지·분할·되감기하도록 네이티브를 확장하는 것이 Phase 6의 실질적 1단계다. 녹음이 네이티브 전용(웹에 `getUserMedia` 없음)이므로, 같은 콜백에서 녹음 버퍼를 경계 분할하면 Take 분할이 자연히 sample-accurate가 된다.
- ✅ 상태 흐름 정리: `idle -> armed -> countIn -> recording -> stopping -> finalizing -> armed` — **Stage 1로 완료(v1.25.0, 2026-07-17 · 시험 대기)**. `idle`/`armed`는 ARM에서 파생하고 머신은 바쁜 4개 phase만 소유하며, 전이는 `enterRecordPhase()` 한 곳을 지나 **불법 전이를 거부**한다. 진입점은 `startRecordFlow({ trackId, start, end, loopEnabled, countIn })`([app.jsx](app.jsx)). 상태를 `await` 뒤의 `track.recording`으로 추론하던 구조가 만들던 경합 3건(이중 시작·이중 finalize·시작 중 Stop) 차단. 상세 [앱개발.md](앱개발.md) v1.25.0.
- 루프 시작/끝을 sample index로 변환
- 오디오 블록이 루프 끝을 걸치면 경계에서 분할 기록
- 회차별 `Take A`, `Take B`, `Take C` source 생성 (⚠️ 현재 `attachRecording`은 `sources[]`/`clips[]`를 배열째 재대입해 Take 누적 불가 — **append 기반 + `activeTakeId` 모델로 재구성 필요**)
- Stop 시 불완전한 마지막 회차는 `Partial` Take로 보존
- **✅ 입력 지연 보정값을 clip `start`에 반영 — 완료(v1.33.0, 2026-07-19 · 시험 대기)**: **전역 오프셋(ms)** 을 localStorage(`focusdaw-record-offset-ms`, Auto=드라이버 추정치 추종 / 수동 override)로 두고, 녹음 finalize에서 Take clip을 그만큼 앞으로 슬라이드(non-loop = `min(offset,start)`로 truncation 방지, loop = `loopStart/loopEnd` 동일 시프트로 pass 길이 보존). Device Setup 패널에 Auto 체크박스 + ms 입력 UI. 상세 [앱개발.md](앱개발.md) v1.33.0 · 시험 [시험.md](시험.md) T-1.33.0-1. (v1.20.7 latency는 WASAPI 드라이버 **추정치**라 Auto 기본 + 수동 미세조정으로 처리 — Q4 결정.)
- Take Lanes 펼치기/접기 UI (⚠️ 전략 B 베이크는 트랙당 버퍼 1개 전제 — **활성 Take의 clip만 bake** 대상으로 두고 겹침 금지 판정을 활성 레인 내부로 한정해야 함)
- **✅ count-in용 메트로놈 클릭**(Q2 결정) — **완료(v1.26.0, 2026-07-17 · 사용자 시험 5건 통과)**: `MetronomeClick`(1568/784Hz 사인 버스트, 장치 열 때 미리 렌더)을 **독립 device callback**으로 등록해 BPM 기반 클릭을 네이티브에서 생성하고, `startCountIn`/`stopMetronome` 명령과 트랜스포트 On/Off 버튼을 붙였으며, count-in을 **1초×3 고정 → 1마디 4박(60/BPM)**으로 전환했다(BPM 없으면 버튼 disabled + 기존 무음 유지). **모니터 전용은 구조로 보장** — 재생 체인의 노드가 아니라 독립 콜백이므로 `InputRecorder`(입력만 기록)에도 `exportMix`(소스 체인 오프라인 렌더)에도 닿지 않는다(**T-1.26.0-2로 확인**). 기존 취소 규칙·ARM 잠금(v1.20.11)·Repeat 복원 유지. 상세 [앱개발.md](앱개발.md) v1.26.0.
  - ⚠️ **Stage 2b(v1.27.0)에서 방향 전환**: 사용자 지적으로 **고정 BPM 클릭의 한계**(SUNO 스템은 템포가 흔들리고, BPM이 맞아도 곡의 박자 위상을 클릭이 알 수 없어 sync가 어긋난다)가 확인되어, 정지 상태 녹음의 기본 진입을 **프리롤**(곡 자체를 N초 리드인으로 재생 후 제자리 punch-in)로 바꿨다. 메트로놈/카운트인은 **들려줄 음악이 없을 때**(빈 프로젝트·곡 맨 앞·Repeat 구간)의 폴백으로 남는다. 프리롤은 분석·BPM이 필요 없고 Stage 6 Punch가 그대로 재사용한다. 상세 [앱개발.md](앱개발.md) v1.27.0.
- **Punch Recording (Q4 결정, 2026-07-14 편입)**: 같은 인프라(구간 지정 녹음 `startRecording({start,end})`, count-in/pre-roll, 상태머신, sample-accurate 경계 분할) 위에서 "회차별 Take 누적" 대신 "단일 구간 클립 교체 + 자동 punch-in/out"으로 동작하는 변형. pre-roll 재생 → in 지점 자동 punch-in → out 지점 자동 punch-out → 해당 구간 clip만 비파괴 교체(Phase 5 `deleteRange`+클립 삽입 재사용). 루프 Take 녹음 안정화 후 마무리.

완료 기준:

- 각 Take 길이가 루프 sample 수와 일치한다.
- 반복 녹음이 길어져도 누적 밀림이 없다.
- 루프 경계가 오디오 블록 중간에 있어도 Take가 정확히 나뉜다.
- Punch: pre-roll 후 in/out 지점에서 정확히 punch-in/out되고, 교체된 구간 외 오디오는 변하지 않는다.

- 원본 WAV를 변경하지 않는 비파괴 comp가 가능하다.

- 기존 Settings 화면에 `Device Setup` 영역을 추가한다.

## 6. 프로젝트 파일 관리

권장 구조:

```text
My Project.focus
My Project Audio/
  Recordings/
  Bounces/
  Consolidated/
  Unused/
```

- `Consolidated/`는 `Consolidate Clips` 결과(v1.37.0). `Bounces/`(트랙 합치기 결과)와 분리한다 — 성격이 다르고, 정리 도구가 폴더 단위로 판정하기 쉽다.
- 프로젝트가 아직 저장되지 않았으면 consolidate 결과는 temp(`FocusDAW Consolidated`)에 기록하고, Save As 복사 규칙에서 프로젝트 폴더로 옮긴다. **경로 없는 소스는 재열기에서 복원 불가**이므로 "저장 전에는 기록하지 않음"은 선택지가 아니다.

- `.focus`에는 프로젝트 기준 상대 경로를 우선 저장한다.
- 저장 전 프로젝트는 앱 임시 녹음 폴더를 사용하고 첫 저장 시 이동한다.
- source에는 파일 크기, 수정 시간, 필요 시 hash를 기록한다.
- Autosave는 편집 메타데이터만 저장하고 WAV는 writer가 즉시 디스크에 확정한다.
- 녹음, Merge, Consolidate 결과 파일은 세션 동안 보존해 Undo/Redo를 가능하게 한다.

## 8. 우선순위 요약

| 우선순위 | 작업 | 이유 |
|---|---|---|
| 1 | Source/Clip 모델 정리 | Audio In, Take, 이동 clip의 선행 조건 |
| 2 | File Tracks 그룹 안정화 | v1.17.5 구현분을 새 구조와 연결 |
| 3 | Merge Tracks / Bounce | 파일 트랙 작업 흐름 완성 |
| 4 | 단일 Audio In 녹음 | 가장 작은 녹음 MVP |
| 5 | Audio In 클립 편집 | 녹음 결과를 실제로 편집 가능하게 함 |
| 6 | 루프 Take 녹음 | 다중 Take workflow |
| 7 | Comp / Consolidate / 정리 | 장기 프로젝트 운용 안정화 |

가장 중요한 선행 작업은 `트랙당 단일 버퍼` 관점을 `source/clip layout`으로 바꾸는 것이다. 이 기반 없이 녹음 기능부터 붙이면 Take, 저장, Undo, 네이티브 동기화가 모두 임시 구조에 묶이게 된다.
