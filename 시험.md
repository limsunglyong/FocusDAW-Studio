## v1.21.1 — 수정요청 3건 + Bounce 클립 편집 버그

> 배경: [수정요청.md](수정요청.md) 1~3번 처리 + 인앱 검증 중 발견된 Phase 5 구멍(Bounce 트랙 클립 편집 무동작) 수정. 상세 [앱개발-아카이브.md](앱개발-아카이브.md) v1.21.1.
> 참고: ①의 이름 로직과 ③의 메뉴 동작(Copy→Paste 활성), Esc 해제는 개발자 인앱 검증을 마쳤다. 아래는 **사용자 실사용 관점 확인**이다.

### T-1.21.1-1. 프로젝트 이름이 저장 파일명을 따르는지
- 시험 내용: ① 새 프로젝트에서 **Save As**로 파일명을 `mysong001.focus`처럼 지정해 저장한다. ② 화면 우상단 프로젝트 이름과 Project ▸ Open Project ▸ **Recent Saved** 목록의 이름을 본다. ③ 앱을 재시작해 그 파일을 다시 연다. ④ **기존에 문제였던 `untitled123.focus`도** 열어본다.
- 판정 기준: ②에서 이름이 **`mysong001`**(파일명과 동일)로 표시된다. ③ 다시 열어도 `mysong001`. ④ 구버전이 저장해 내부에 `untitled`가 박혀 있는 파일도 이제 **`untitled123`**으로 열린다.
- 비고: **의도된 동작 변경** — 경로가 있는 프로젝트를 우상단에서 rename한 뒤 저장하면 이름이 **파일명으로 되돌아온다**("이름 = 파일명" 규칙). 다른 이름을 쓰려면 Save As로 저장할 것. 이 동작이 불편하면 알려주면 규칙을 재검토한다.

### T-1.21.1-2. 클립 선택 해제 (Esc / 바깥 클릭)
- 시험 내용: Audio In 클립을 클릭해 선택한 뒤 ① **Esc**를 누른다. ② 다시 선택하고 **빈 레인·다른 트랙·화면 여백**을 클릭한다. ③ 다시 선택하고 **다른 클립**을 클릭한다.
- 판정 기준: ①② 선택 테두리(앰버)가 사라진다. ③ **새로 클릭한 클립이 선택된 채로 남는다**(선택이 해제돼 버리면 실패). 선택 해제 후 Delete를 눌러도 아무 클립도 지워지지 않는다.

### T-1.21.1-3. 클립 우클릭 메뉴 — Audio In + **Bounce** 양쪽
- 시험 내용: **Audio In 클립**과 **Bounce 트랙 클립**(Merge Tracks로 생성) 각각에서 ① 클립을 우클릭해 메뉴를 연다. ② Copy → 다른 위치에서 Paste at playhead. ③ Duplicate. ④ Delete. ⑤ 화면 **가장자리 클립**을 우클릭해 메뉴가 창 밖으로 잘리지 않는지 본다. ⑥ **Bounce 클립을 드래그로 이동·trim**한다.
- 판정 기준: 메뉴에 Copy/Paste/Duplicate/Delete가 단축키와 함께 뜨고 각 동작이 실제로 수행된다. 클립보드가 비어 있으면 Paste는 흐리게 비활성, Copy 후에는 활성. ⑤ 메뉴가 항상 창 안에 보인다. ⑥ **Bounce 클립도 Audio In처럼 이동·trim된다**(v1.21.1 이전에는 조용히 무시됐음 — 이번 수정의 핵심).
- 비고: 클립 편집 UI는 **select 툴 + Automation Off**일 때만 뜬다. 파일 트랙(Stem)에는 의도적으로 뜨지 않는다.

## 추후 확인 항목 (보류·장기 관찰) — 버전 무관

### T-1.20.10-2(b). 입력 미사용 프로필에서 시작 시 마이크 미개방
- ⏰ **시험 시점: Phase 6 Stage 5 착수 직전**(위 "예약된 시험 알림" 참조). 그 전까지는 회귀 위험이 없어 대기.
- ⚠️ 사전 고지: 준비 단계에서 `focusdaw-audio-input` 키를 지우므로 **그동안 맞춰둔 입력 장치 선택이 초기화된다.** 시험 후 Settings ▸ Audio Devices에서 다시 선택하면 된다.
- 시험 환경 준비: `npm start`로 실행 후 DevTools(F12 / Ctrl+Shift+I) 콘솔에서 `localStorage.removeItem('focusdaw-audio-input')` 실행 → 앱을 완전히 종료 후 콜드 재시작한다. (= 입력을 한 번도 쓴 적 없는 프로필 재현. 이 키는 `+` Audio In·arm·녹음·Settings 입력 선택 시 생성된다.)
- 시험 내용: 재시작 직후 Audio In 추가·ARM·녹음을 하지 않은 상태에서 OS 마이크 사용 표시등(Windows 11: 작업 표시줄 마이크 아이콘 / 설정 ▸ 개인정보 및 보안 ▸ 마이크 ▸ 최근 활동)을 관찰한다.
- 판정 기준: 입력 이력이 없는 프로필에서는 시작 시 마이크가 **열리지 않는다**. (대조: Audio In을 한 번 추가해 키를 재생성한 뒤 콜드 재시작하면 이번엔 시작 시 마이크가 열려야 정상 — 이력 있는 프로필의 유휴 워밍업.)
- 비고: T-1.20.10-1 및 T-1.20.10-2(a) arm/녹음은 시험 완료(시험-아카이브.md).

### T-1.20.8-1. 앱 종료 시 엔진 비정상 종료 코드 재발 관찰 (장기 tracing)
- 시험 내용: 평소처럼 사용하며 앱을 시작/종료할 때(믹서 창을 열어 둔 채 종료 포함) 콘솔/로그의 `[AudioEngine] Process exited with code ...` 메시지를 관찰한다. 단발성 경합이므로 일정 기간 반복 관찰이 필요하다.
- 판정 기준: 종료 코드가 항상 정상(1 = Electron kill, 또는 0)이고, `3221225477`(0xC0000005) 등 비정상 코드가 재발하지 않는다. 재발 시 `%LOCALAPPDATA%\CrashDumps`의 새 덤프 유무를 함께 확인해 보고한다.
