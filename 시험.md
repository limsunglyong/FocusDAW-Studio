### 2-A. 브라우저에서 직접 실행 (빠른 시험)

3. DevTools → **Performance** 탭도 열어두기 (성능 측정용)

```bash
# 별도 Windows Terminal / PowerShell 에서 실행 (Claude 내부 터미널 ❌)
npm start
```

> **주의**: `npm start`는 반드시 별도 터미널에서 실행할 것.  
> Claude Code 내부 터미널에서 실행하면 `ELECTRON_RUN_AS_NODE=1` 상속으로 실패한다.

## 7. 알려진 한계 / 시험 시 유의사항

- **Loop OFF 정지 타이밍**: `_loop()` 콜백은 ~100ms 간격으로 실행되므로 duration 도달 후 최대 100ms 후에 `stop()`이 호출될 수 있음. 즉각 정지가 아닌 ~100ms 이내 정지는 정상 동작.
- **IntersectionObserver 최소 지연**: 옵저버가 비동기로 동작하므로 트랙이 뷰포트에 들어오는 순간 극히 짧은 빈 프레임이 보일 수 있음. 200ms 이내라면 정상 동작.
- **scrollLeft 클램핑 timing**: `requestAnimationFrame` 기반이므로 줌 변경 후 1프레임(~16ms) 후에 클램핑 적용됨.
- **MP3 익스포트 시작 딜레이 (~13ms)**: LAME 인코더 고유 딜레이(576 샘플 @ 44100 Hz)로 인해 Electron(Chrome) 환경에서 MP3 파일을 `decodeAudioData`로 디코딩하면 약 13ms 무음이 포함됨. WAV 익스포트는 해당 없음. foobar2000·iTunes 등 LAME 태그 지원 플레이어에서는 자동 보정됨. 설계상 제약으로 수용 (`앱개발.md` §7 ISSUE-2 참조).
- **Windows 탐색기의 WAV 길이 내림(floor) 표시**: Windows 속성창은 WAV 길이를 정수 초로 내림 처리함. 따라서 익스포트 길이가 정확히 정수 초여야 올바르게 표시됨. v0.16.18 이전에는 시스템 레이트(96 kHz) 고정, v0.16.18~v0.16.19 사이에는 트리밍 1프레임 단축으로 4.999979초가 되어 4초로 표시되던 문제가 있었음. v0.16.19에서 트리밍 길이 보존으로 길이가 정확히 `this.duration`과 일치하도록 수정하여 해소됨. (Audacity·실제 재생은 항상 정상)
