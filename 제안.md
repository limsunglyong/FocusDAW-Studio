# Key 반음 변경 기능 추가 제안

## 목표

현재 프로젝트의 Key 값을 기준으로 사용자가 반음 단위로 최대 `+6`, 최소 `-6`까지 조정할 수 있게 한다. 조정값은 재생과 Export에 동일하게 반영되어야 하며, 원곡 Key 검출값은 보존하고 적용 Key만 별도로 관리한다.

## 개발 단계(Phase) 구분

기능을 한 번에 모두 구현하지 않고, 정식 경로인 네이티브 엔진을 먼저 완성·검증한 뒤 브라우저 fallback을 붙이는 2단계로 나눈다.

- **Phase 1 — 네이티브(JUCE) 엔진 우선 구현 및 검증 (정식 경로)**
  - SoundTouch 기반 pitch shift를 재생/Export에 정확히 반영하고, 저장/복원까지 일관되게 만든다.
  - 이 단계 동안 브라우저 fallback에서 `Vari Key`를 켜면 "네이티브 엔진 연결 필요" 안내로 차단한다.
  - Phase 1 검증이 끝나야 Phase 2로 넘어간다.
- **Phase 2 — 브라우저용 WebAudio fallback 구현 (확장 경로)**
  - 네이티브 엔진이 없거나 실패한 환경에서도 Key 변조 재생/Export가 동작하도록 한다.
  - SoundTouchJS(SoundTouch의 JS/WASM 포팅)를 "베이크(pre-render)" 방식으로 적용해 네이티브와 음색을 일치시킨다.

## 현재 코드 상태 (실측 검증)

- `app.jsx`에는 이미 Key 표시/검출/적용 UI의 골격이 있다.
  - `KeyIndicator`에서 로컬 `draft` 값을 `-6 ~ +6` 범위로 조정한다(±6 경계 버튼 비활성화 포함).
  - `Apply`는 `shiftKey(detectedKey, draft)`로 **문자열 적용 Key를 계산해 `setKey`로 커밋**한다. 즉 `draft`(반음 오프셋) 자체는 아직 영속 상태로 저장되지 않는다.
  - `Vari Key` 스위치가 별도 존재한다.
- `audio-engine.js`의 `tempo` 상태에는 `key`, `detectedKey`, `variKey`가 있다. **`keyShift`는 아직 존재하지 않는다(신규 추가 필요).**
- `audio-bridge.js`는 `setVariKey`, `setKey`, `setDetectedKey`를 네이티브 엔진으로 전달한다. **`setKeyShift`는 없다.**
- `juce_skeleton/Source/AudioEngine.*`에는 SoundTouch 기반 `setPitchShift()` 경로가 있고, `updateDspParams()`에서 `variKey && detectedKey && currentKey`일 때 `getSemitoneDifference(detectedKey, currentKey)`로 반음을 계산해 트랙에 적용한다(문자열 차이 의존).
- WebAudio 경로(재생/Export 모두 `createBufferSource` + `playbackRate`)는 BPM time-stretch는 있으나, Key pitch-shift는 적용되지 않는다. `playbackRate`만 바꾸면 tempo까지 같이 변하므로 Key 변경에는 부적합하다.
- Export는 네이티브 렌더 경로와 WebAudio fallback 경로가 나뉘므로 두 경로 모두 Key 변경 반영 여부를 맞춰야 한다.

### 검증된 기존 버그 (Phase 1에서 반드시 수정)

1. **`preservePitch=false`에서 피치 강제 0** — `SoundTouchAudioSource::getNextAudioBlock()`의 rate 모드(`else`) 분기에서 `setPitchSemiTones(0.0f)`를 호출해, 사용자가 설정한 `keyShift`가 무시된다. 이 때문에 Vari BPM + Vari Key를 동시에 쓸 수 없다.
2. **Export 시 DSP 미갱신** — `export` 명령 전에 `setVariKey/setKey/setDetectedKey`를 재동기화하지 않고, `exportMix` 진입 시 `updateDspParams()`도 호출하지 않는다. 프로젝트를 열자마자 바로 Export하면 최신 Key 값이 오프라인 렌더러에 반영되지 않을 수 있다.

## 데이터 모델

`tempo` 상태에 문자열 Key만 저장해도 동작은 가능하지만, 유지보수성을 위해 반음 오프셋을 명시적으로 저장하는 방식을 권장한다.

```js
tempo: {
  projectBpm: null,
  playbackBpm: null,
  variBpm: false,
  detectedKey: null,
  key: null,
  keyShift: 0,
  variKey: false
}
```

권장 규칙:

- `detectedKey`: 원곡에서 검출한 기준 Key
- `keyShift`: 사용자가 선택한 반음 이동값, 정수 `-6 ~ +6`
- `key`: `shiftKey(detectedKey, keyShift)`로 계산된 적용 Key
- `variKey`: 실제 오디오 pitch shift 적용 여부

기존 프로젝트 파일 호환을 위해 `keyShift`가 없으면 `detectedKey`와 `key`의 차이를 계산해 복원한다. 둘 중 하나가 없으면 `0`으로 둔다.

설계 원칙: **JS에서 계산한 정수 `keyShift`를 진실 원본으로 삼고 끝까지 그대로 전달**한다. 네이티브에서 문자열 Key 차이를 다시 계산하지 않게 하여 enharmonic/장단조 표기·파싱 오류의 영향을 제거한다.

# Phase 1 — 네이티브(JUCE) 엔진 우선 구현 및 검증

## 사용자 경험

1. 사용자가 Key 영역을 연다.
2. `Detect`로 원곡 Key를 분석한다.
3. 원곡 Key는 `원Key`로 고정 표시한다.
4. `+ / -` 버튼 또는 휠/스텝 컨트롤로 `-6 ~ +6` 반음 범위 안에서 조정한다.
5. `Apply`를 누르면 적용 Key가 저장된다.
6. `Vari Key`가 켜져 있을 때만 실제 재생/Export에 pitch shift를 적용한다.
7. `Vari Key`가 꺼져 있으면 적용 Key 표시값은 유지하되 소리는 원본 Key로 재생한다.

이 구조는 "Key를 어디로 바꿀지"와 "실제로 적용할지"를 분리해, BPM의 `Project BPM / Playback BPM / Vari BPM` 구조와도 잘 맞는다.

## 1-1. 상태 API 정리 (`audio-engine.js`)

- `tempo`에 `keyShift: 0`을 추가한다(초기화 3곳 모두: 초기 상태, 프로젝트 초기화, 빈 프로젝트 리셋).
- `_normalizeKeyShift(value)`: 숫자를 정수로 변환 후 `-6 ~ +6` clamp.
- `setKeyShift(semitones)`: `tempo.keyShift` 저장 → `tempo.key = shiftKey(detectedKey, keyShift)` 재계산 → `variKey`로 재생 중이면 BPM 변경과 동일한 debounce 후 재시작.
- `_keyShiftSemitones()`: `variKey`가 켜져 있고 `detectedKey`가 있으면 실제 적용할 반음값을 반환.
- `setKey(key)`는 외부 호환용으로 유지하되, 내부적으로 `detectedKey` 기준 차이로 `keyShift`도 함께 맞춘다.

## 1-2. UI는 반음 오프셋 중심으로 (`app.jsx`)

`KeyIndicator`는 이미 거의 원하는 구조다. 다음만 정리한다.

- `draft` 초기값을 `tempo.keyShift` 우선으로 읽는다.
- `+` 버튼은 `draft < 6`, `-` 버튼은 `draft > -6`일 때만 활성화한다(이미 구현됨, 유지).
- `Apply`는 문자열 커밋 대신 `DAW.setKeyShift(draft)`를 호출한다.
- 표시 Key는 `shiftKey(detectedKey, draft)`로 즉시 미리보기한다.

이렇게 하면 UI는 "Key 문자열을 직접 저장"하지 않고 "반음 이동값을 저장"하게 되어 경계값과 Export 동기화가 명확해진다.

## 1-3. 브리지 연동 (`audio-bridge.js`)

- `setKeyShift(semitones)`를 추가하고 `{ command: "setKeyShift", semitones }`를 네이티브로 전송한다.
- `setKey`/`setDetectedKey`/`setVariKey`는 UI 표시 및 기존 호환용으로 유지한다.

## 1-4. C++ 엔진 + IPC (`AudioEngine.*`, `WebSocketServer.cpp`)

- `int keyShift` 상태와 `setKeyShift(int)` 핸들러, `setKeyShift` IPC 명령을 추가한다.
- `updateDspParams()`에서 문자열 차이 대신 전달받은 정수를 사용한다.
  - `targetPitchShift = variKey ? clamp(keyShift, -6, 6) : 0`
- `setKey`/`setDetectedKey`는 표시/호환용으로 남긴다.

## 1-5. preservePitch 버그 수정 (버그 ①)

- `SoundTouchAudioSource::getNextAudioBlock()`의 rate 모드 분기에서도 사용자 `keyShift`가 합산 적용되도록 보완한다. 즉 `preservePitch=false`(tempo와 pitch가 함께 변하는 모드)에서도 `setPitchSemiTones(0)`로 강제 리셋하지 말고 현재 pitch 값을 적용한다.
- 이로써 `Vari BPM + Vari Key` 동시 사용이 가능해진다.

## 1-6. Export 반영 (버그 ② 수정)

- Export 시작 전 `audio-bridge.js`에서 `setVariKey`, `setKeyShift`, `setKey`, `setDetectedKey`를 모두 재동기화한 뒤 `export` 명령을 보낸다.
- `AudioEngine::exportMix` 진입 시 `updateDspParams()`를 1회 강제 호출해 최신 tempo/pitch를 SoundTouch에 주입한다.
- `variKey && keyShift != 0`이면 오프라인 렌더에서 SoundTouch pitch shift를 켠다.
- `variBpm`과 `variKey`가 동시에 켜진 경우 SoundTouch에 `tempo`와 `pitch`를 함께 적용한다.

## 1-7. 프로젝트 저장/불러오기

`exportProject()`, undo snapshot, `importProject()`에 `keyShift`를 포함한다.

호환 규칙:

- 새 프로젝트: `keyShift` 저장
- 예전 프로젝트: `keyShift`가 없으면 `detectedKey`와 `key`로 계산
- 계산 불가: `0`
- 빈 프로젝트로 초기화할 때 `detectedKey`, `key`, `keyShift`, `variKey` 모두 초기화

## 1-8. Phase 1 동안의 fallback 정책 (임시 차단)

- 브라우저 WebAudio fallback 상태에서 `Vari Key`를 켜고 재생/Export를 시도하면, 무음 실패나 경고 없는 원본 재생을 막기 위해 다음 안내를 노출하고 예외 처리한다.
  - "Key 변조 재생 및 Export 기능은 네이티브 오디오 엔진(JUCE) 연결 상태에서만 완벽히 지원됩니다."
- 이 차단은 임시 정책이며 Phase 2에서 해제한다.

## Phase 1 검증 항목

- Key 검출 후 `+6`, `-6`, `0` 적용 Key 표시가 맞는지 확인
- `+6`에서 `+` 버튼, `-6`에서 `-` 버튼 비활성화
- `Vari Key OFF`에서는 적용 Key 표시만 바뀌고 소리는 원본 유지
- `Vari Key ON`에서는 재생 pitch가 반음 단위로 변경
- `Vari BPM ON + Vari Key ON`에서 tempo와 pitch가 동시에 적용(버그 ① 수정 확인)
- 프로젝트를 열자마자 바로 Export해도 최신 Key가 반영됨(버그 ② 수정 확인)
- 네이티브 Export 결과가 실시간 재생과 같은 Key인지 확인
- 프로젝트 저장 후 다시 열었을 때 `detectedKey`, `keyShift`, `key`, `variKey` 복원
- fallback 상태에서 Vari Key Export 시 안전하게 안내/중단

## Phase 1 권장 구현 순서

1. `tempo.keyShift`와 `setKeyShift()` 추가 (`audio-engine.js`)
2. `KeyIndicator`를 `keyShift` 중심으로 연결 (`app.jsx`)
3. `audio-bridge.js`에 `setKeyShift` 전달 추가
4. C++ `AudioEngine`에 `keyShift` 상태와 IPC 명령 추가
5. preservePitch 버그(①) 수정 → 네이티브 재생 pitch shift 검증
6. Export 동기화 + `updateDspParams` 강제 호출(②) → 네이티브 Export 검증
7. 저장/불러오기/undo snapshot 호환 처리
8. fallback 임시 차단 안내 적용

# Phase 2 — 브라우저용 WebAudio fallback 구현

Phase 1 검증 완료 후 진행한다. 목표는 네이티브 엔진이 없는 브라우저 환경에서도 Key 변조 재생/Export를 동일 품질로 제공하는 것이다.

## 2-1. 라이브러리 선택: SoundTouchJS (베이크 방식)

- 네이티브 JUCE 엔진이 이미 SoundTouch를 쓰므로, 같은 알고리즘의 JS/WASM 포팅인 **SoundTouchJS**를 쓰면 네이티브 Export와 fallback Export의 음색이 사실상 일치한다.
- 밑바닥부터 phase vocoder/PSOLA를 구현하는 것은 비권장(난이도·품질 튜닝 부담). Tone.js `PitchShift`(그래뉼러)는 품질이 낮고 offline render에 부적합, Rubber Band는 GPL/상용 라이선스 부담으로 오버킬.

### 베이크(pre-render) 전략

이 앱은 재생/Export 모두 `createBufferSource` + `playbackRate` buffer 기반이므로, 실시간 피치 노드를 끼우기보다 **`keyShift`가 바뀔 때 트랙 버퍼를 한 번 피치 시프트해 새 `AudioBuffer`로 캐시**하고, 재생·Export는 그 시프트된 버퍼를 사용한다.

장점:

- Export(`OfflineAudioContext`)에서 실시간 노드 latency/glitch 문제를 회피한다.
- SoundTouchJS의 실시간 노드는 deprecated된 `ScriptProcessorNode` 기반이라 OfflineAudioContext에서 불안정할 수 있는데, 베이크 방식은 이를 우회한다.
- 재생은 시프트된 버퍼를 `playbackRate`(tempo)로만 돌리면 되어 Vari BPM(tempo)과 Vari Key(pitch)가 자연스럽게 분리된다.

## 2-2. 구현 항목

- `_getShiftedBuffer(track, semitones)`: SoundTouchJS로 `track.buffer`를 `semitones`만큼 시프트한 `AudioBuffer`를 생성/캐시한다. `semitones === 0`이면 원본을 그대로 반환.
- 캐시 무효화: `keyShift`, `detectedKey`, 트랙 버퍼가 바뀌면 해당 캐시를 폐기한다.
- 재생 경로: 시프트 캐시가 있으면 `createBufferSource`에 시프트된 버퍼를 물린다.
- Export 경로(`renderMix` / OfflineAudioContext): 동일하게 시프트된 버퍼를 사용한다.
- `variKey`가 꺼져 있으면 원본 버퍼를 사용한다.

## 2-3. UX 처리

- `keyShift` 변경은 곡 전체 재처리(수백 ms~수초, 곡 길이 비례)를 유발하므로 debounce로 묶고, 처리 중 스피너/진행 표시를 노출한다.
- Phase 1의 "네이티브 필요" 임시 차단 안내는 이 단계에서 해제한다.

## Phase 2 검증 항목

- 네이티브 미연결(브라우저) 상태에서 `Vari Key ON` 재생 pitch가 반음 단위로 변경
- 같은 상태에서 Export 결과가 재생과 같은 Key
- `Vari BPM + Vari Key` 조합에서 tempo와 pitch가 함께 적용
- 네이티브 Export 결과와 fallback Export 결과의 Key/음색이 일치
- `keyShift` 변경 시 캐시 무효화·재생성이 정상 동작하고, debounce 중 과도한 재처리가 없는지 확인

## 예외 처리 (공통)

- 원곡 Key가 없으면 `+ / - / Apply` 비활성화
- `keyShift`는 항상 정수로 저장, 범위 밖 값은 `-6 ~ +6`으로 clamp
- `detectedKey`가 변경되면 기존 적용 Key는 초기화한다(권장). 즉 `keyShift`도 `0`으로 리셋.
- `Vari Key`를 켠 상태에서 재생 중 `keyShift`를 바꾸면 짧은 debounce 후 재시작한다. BPM 변경 재시작 로직과 같은 패턴을 재사용한다.

## 결론

가장 안정적인 설계는 "원곡 Key 문자열 + 반음 오프셋 정수"를 분리하는 것이다. UI는 `-6 ~ +6` 정수만 다루고, 오디오는 `keyShift` 값을 그대로 SoundTouch에 전달한다.

개발은 **Phase 1(네이티브 우선 구현·검증) → Phase 2(WebAudio fallback)** 순으로 나눈다. Phase 1에서 정식 경로를 완성하고 검증된 두 버그(preservePitch 강제 리셋, Export DSP 미갱신)를 함께 잡은 뒤, Phase 2에서 SoundTouchJS 베이크 방식으로 브라우저 환경까지 동일 품질로 확장한다.
