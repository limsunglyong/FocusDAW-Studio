## v1.28.0 — 다중 클립 그룹이 이웃을 뛰어넘어 이동 (+ 크래시 계측 제거) (2026-07-18)

> **재빌드: renderer**([audio-engine.js](audio-engine.js)·[audio-bridge.js](audio-bridge.js)는 raw). 시험 대기 1건([시험.md](시험.md) T-1.28.0-1). v1.27.x 크래시 시험(T-1.27.8-1)·프리롤·F2 시험 **전부 통과**([시험-아카이브.md](시험-아카이브.md)).

### 버그 (사용자 보고)

클립 A-B-C-D에서 **가운데 2개(B-C)를 선택해 드래그**하면 **A 앞이나 D 뒤로 못 나간다**(A·D를 벽으로 취급). 그런데 **B 단독 또는 C 단독**은 A 앞·D 뒤로 잘 나간다.

### 원인 — 그룹은 벽 클램프, 단일은 뛰어넘기

단일 클립 드래그는 `_resolveMovePosition`(v1.21.4 방향성 스냅)으로 이웃을 **뛰어넘어** 앞/뒤에 스냅한다. 반면 그룹 이동은 `_clampGroupDelta`가 선택되지 않은 클립을 **넘을 수 없는 벽**으로 클램프한다([audio-engine.js](audio-engine.js)). 그래서 2개 이상 선택하면 이웃을 못 넘었다.

### 수정 — 그룹판 방향성 스냅

- **`_resolveGroupDelta` 신설** — `_resolveMovePosition`의 그룹 버전. 선택 전체를 강체로 유지한 채, 이웃과 겹치면 **그 이웃의 앞(오른쪽 끝이 이웃 시작에 붙음) 또는 뒤(왼쪽 끝이 이웃 끝에 붙음)로 점프**하고 그 자리가 비어 있으면 확정, 아니면 반대쪽, 둘 다 막히면 원위치. **드래그가 0 이전으로 과하게 가면 타임라인 머리에 고정**(단일의 `s0=max(0,start)` 대응) — 이게 빠져서 "A 앞 빈 공간에 못 들어가던" 세부 버그가 유닛 테스트에서 잡혀 수정됨.
- **`moveClipsByResolved` 신설**(드래그 커밋용, 네이티브 sync 포함) — 넛지는 기존 `moveClipsBy`(butt/clamp) 유지. **단일 클립도 넛지는 뛰어넘지 않으므로**(`_clampNudgeStart`) 그룹 넛지도 동일하게 두는 게 일관적.
- 드래그 고스트([ui-tracks.jsx](ui-tracks.jsx))도 `_resolveGroupDelta`로 바꿔 **점프를 실시간 미리보기**.

**검증**: 리졸버 로직을 독립 유닛 테스트(A 앞 진입·D 뒤 진입·공간 없을 때 정지·gap 내 이동·butt)로 확인. 앱 로드 무에러 확인. **실제 드래그 조작은 T-1.28.0-1에서 사용자 확인.**

### 크래시 계측 전부 제거 (T-1.27.8-1 통과 확인)

곡 끝 크래시(v1.27.8)가 해결 확인돼, v1.27.6/7에서 넣은 진단을 전부 제거: `_diag`(콘솔 스팸)·`studio.html` 에러 핸들러·`socket.onclose` 로그·`main.js` 크래시 로그(바탕화면 파일·render/child-process-gone 핸들러). **실제 수정(v1.27.2 bake·v1.27.3 자동정지·v1.27.4 songLength·v1.27.5 rAF 토큰·v1.27.8 muted-skip)은 전부 유지.**

## v1.27.8 — 곡 끝 근처 재생 시 렌더러 크래시 수정 (Web Audio CHECK) (2026-07-18)

> **재빌드: 없음**([audio-engine.js](audio-engine.js)만 수정 — raw 로드). 시험 대기 1건([시험.md](시험.md) T-1.27.8-1). **계측(v1.27.6/7)으로 원인 확정 후의 실제 수정.** 진단 코드는 이번 확인 통과 후 제거.

### 확정된 원인 (바탕화면 `focusdaw-crash.log`)

```
[RENDER_GONE] {"reason":"crashed","exitCode":-2147483645}
```

`-2147483645 = 0x80000003 = STATUS_BREAKPOINT` — **Chromium 내부 `CHECK()` 실패**의 시그니처(JS 예외로 안 잡히고 렌더러 즉사). 3차에 걸친 계측 결론:

- **누수/OOM 아님**(heap 202MB 고정, 상태 불변) — 이벤트가 **오디오 스레드의 C++ 구조**에 쌓여 JS 힙엔 안 보였다.
- **곡 끝 근처(off≈dur)에서 재생 시작 시** 무증상 하드 크래시. **rep=0**(루프 무관).

### 근본 원인 — 음소거된 웹 그래프가 64루프 커브를 미리 스케줄

가청 출력은 **네이티브**이고 웹 그래프는 `masterOutputGain=0`으로 **음소거**돼 있다(들리지 않는다). 그런데 웹 엔진의 `_scheduleAutomation`/`_scheduleFade`는 재생·seek마다 **`loops=64`회분 커브/램프를 미리 스케줄**한다. audioIn 트랙은 **앞부분 전체가 gap(침묵)** 이라 `hasGaps` 경로로 `setValueCurveAtTime`을 타는데, **곡 끝 근처에서 재생을 시작하면** 남은 구간·경계 커브가 Chromium Web Audio의 CHECK를 건드려 **렌더러가 즉사**한다. 반복할수록 스케줄된 이벤트가 쌓여 6~7회째에 터진 것.

**요지: 안 들리는(음소거) 작업이 렌더러만 죽이고 있었다.**

### 수정

`_scheduleAutomation`/`_scheduleFade`가 **`_outputMuted`(네이티브 활성 = 음소거)면 즉시 건너뛴다** — autoGain/fade를 중립(1)으로 두고 대기 이벤트를 지운 뒤 반환. 네이티브가 자동화·마스터 페이드를 **자체 적용**하므로(`sendTrackAutomationToNative` 등) **가청 손실 0**. `_startHotAddedTrack`이 이미 `_outputMuted`로 게이트하는 것과 같은 패턴.

- **안전망 보강**: `setOutputMuted(false)`(네이티브 크래시 → `switchToLocal`로 웹 복귀)가 재생 중이면 자동화/페이드를 **즉시 재스케줄** — 음소거 중 건너뛴 것을 복원해 폴백 출력이 정상 엔벨로프로 들리게.

### 확인 한계 / 잔여

- **개발자가 크래시를 재현 못 함**(마이크 실녹음·타이밍 필요) — 계측으로 원인 계층까지 확정했고 수정은 그 계층(음소거 웹 스케줄링)을 직접 제거한다. T-1.27.8-1이 실제 확인.

## v1.27.6 — ⛑ 반복 seek 크래시 계측 (원인 미확정) (2026-07-18)

> **재빌드: 없음**([audio-engine.js](audio-engine.js)·[main.js](electron/main.js)·version 수정 — raw/main). **기능 수정 아님, 진단용.** v1.27.5의 rAF 수정으로도 **약 4~5회 반복 후 크래시가 남음**(사용자 보고).

v1.27.5의 `_loop` rAF 누수는 실재했으나(수정 유지) **크래시의 주원인은 아니었다.** 여전히 "재생 → 끝으로 seek → loop → 다시 seek"를 **4~5회** 반복하면 까만 화면. **"4~5회"라는 일정한 횟수**는 점진적 누수보다 **어떤 자원 한계**에 가깝다는 신호일 수 있다(예: 렌더러 OOM, 또는 고정 한도).

- **`main.js`**: `render-process-gone` / `unresponsive` 핸들러 신설 — 렌더러가 죽은 **사유**(`oom`/`crashed`/`killed` 등)를 `userData/focusdaw-crash.log`에 기록. 까만 화면의 정체를 이걸로 확정한다.

1. **앱 완전 재시작**(단순 새로고침 아님 — main.js 변경 반영).
2. **F12 콘솔** 열기.

## v1.27.5 — 재생 중 seek 반복 시 `_loop` rAF 체인 누수 → 렌더러 크래시 (2026-07-18)

> **재빌드: 없음**([audio-engine.js](audio-engine.js)만 수정 — raw 로드). 시험 대기 1건([시험.md](시험.md) T-1.27.5-1). v1.27.4 재시험 중 사용자 발견(기능은 정상, 반복 조작에서 크래시).

### 증상 (사용자 보고)

v1.27.4로 백킹 loop·녹음 재생은 **정상**. 그러나 **재생 → 청음 → 끝으로 seek → 전체 loop → 다시 끝으로 seek**를 **몇 번 반복**하니 앱이 **까만 화면**으로 크래시.

### 원인 — seek마다 `_loop` rAF 체인이 하나씩 쌓인다

웹 엔진의 whole-song 재생 루프 `_loop()`([audio-engine.js](audio-engine.js))는 매 프레임 `requestAnimationFrame(() => this._loop())`로 자기를 재예약하고 **`this.isPlaying`만 보고 멈춘다.** 그런데 **재생 중 seek**은 `_stopSources() → isPlaying=false → play()` 순서로 도는데([audio-engine.js](audio-engine.js) `seek`), `play()`가 다시 `isPlaying=true`로 만들고 **새 `_loop` 체인**을 연다. seek 직전에 예약돼 있던 **이전 체인의 rAF**가 뒤늦게 발화하면, `isPlaying`이 (새 play 때문에) 또 true라서 **죽지 않고 살아남아** 새 체인과 나란히 돈다.

→ **재생 중 seek 할 때마다 `_loop` 체인이 +1.** 체인마다 매 프레임 `_emit()`(전체 React 리렌더)을 부르므로, 먼 위치의 큰 파형까지 N배로 반복 렌더된다. 몇 번 쌓이면 렌더러가 포화돼 창이 꺼진다(점진적·반복 후 크래시 — 보고와 일치).

`_playToken`(play/seek/stop마다 증가)이 **이미 있는데** `_loop`가 그걸 확인하지 않은 게 문제였다.

### 수정

`_loop(token)`이 **자기 토큰이 최신 `_playToken`과 다르면 즉시 종료**하도록 했다. 이제 seek/pause/stop이 토큰을 올리는 순간 이전 체인은 **다음 rAF에서 스스로 죽고**, 항상 **정확히 하나**의 `_loop`만 돈다. loop-restart(경계 되감기)는 `play()`가 새 토큰 체인을 열고 현재 `_loop`는 return하므로 그대로 유지.

### 확인 한계

이 rAF 누수는 **명백한 버그**이고 증상(반복 seek → 점진적 과부하 → 까만 화면)과 정확히 맞는다. 다만 **개발자가 크래시를 재현하진 못했다**(마이크 실녹음·타이밍 필요). T-1.27.5-1이 실제 확인이다. 만약 그래도 크래시가 남으면 별개 원인(네이티브·메모리)을 판다.

## v1.27.4 — 곡 길이가 자라면 백킹 트랙이 자기 끝에서 되감기던 버그 (2026-07-18)

> **재빌드: 없음**([audio-bridge.js](audio-bridge.js)만 수정 — raw 로드, 네이티브 무변경). 시험 대기 1건([시험.md](시험.md) T-1.27.4-1). v1.27.3 재시험 중 사용자 발견.

### 증상 (사용자 보고)

① **백킹 트랙 끝을 지나면 백킹이 처음부터 다시 재생**된다(파일 트랙이 자기 끝에서 loop). ② 이전과 마찬가지로 **녹음 위치와 소리가 어긋난다.** (크래시는 v1.27.3으로 사라짐.)

### 원인 — 트랙마다 다른 songLength로 개별 루프

네이티브는 각 트랙을 **로드 시점의 `songLength`로 패딩**해 **트랙마다 개별적으로 loop**한다([AudioEngine.cpp](juce_skeleton/Source/AudioEngine.cpp) `decodeAndInstallTrack` + `setLooping`). 모든 트랙이 **같은 songLength**를 가질 때만 함께 되감긴다. 그런데:

- 파일 트랙은 곡이 2:55일 때 로드되어 **songLength=2:55**로 패딩됨.
- 이후 Audio In 클립을 3:30으로 옮기거나 그 지점에 녹음하면 **프로젝트 길이는 자라지만**, `syncTrackToNative`는 **방금 편집한 트랙 하나만** 새 길이로 밀어 넣는다. 파일 트랙의 songLength는 **2:55로 남는다.**
- → 재생이 2:55를 지나면 파일 트랙이 **자기 끝에서 되감겨 처음부터** 나오고(증상 ①), 나머지 믹스와 어긋나 "녹음 위치와 소리 불일치"로 느껴진다(증상 ②).

이는 **기존 구조적 한계**다. 프로젝트를 **열 때**는 모든 트랙이 최종 길이로 로드돼 안 드러났고, **편집으로 길이가 자랄 때만** 터진다. Stage 2b(프리롤)로 "곡 끝 너머 녹음"이 실제 흐름이 되면서 표면화됐다.

### 수정 — 길이가 바뀌면 모든 트랙의 songLength 재동기화

`ensureSongLengthConsistent()` 신설([audio-bridge.js](audio-bridge.js)): `syncTrackToNative` 끝에서 **프로젝트 duration이 마지막으로 전체 푸시한 값과 달라졌으면 모든 트랙을 새 길이로 재푸시**한다(재접속 경로가 이미 쓰던 것과 같은 방식). 재진입 가드로 한 번의 대량 푸시가 트랙마다 다시 도는 것을 막고, 녹음 중 트랙은 건너뛴다(테이크 중 재디코드 방지).

- 비용: 길이가 자라는 편집(끝 너머 녹음·이동)에서 파일 트랙이 백그라운드 재디코드된다. 이런 편집은 **대개 정지 상태**에서 일어나고, 네이티브가 old 트랙을 유지하다 원자적으로 교체하므로 가청 글리치는 거의 없다. duration이 안 바뀌는 일반 편집(구간 내 넛지 등)에서는 **아무 일도 하지 않는다**(가드).
- ⚠️ 증상 ②가 이 수정 후에도 남으면 별개의 bake/네이티브 배치 문제다 — T-1.27.4-1에서 판별.

### 후속 과제 (근본 해결은 아님)

진짜 근본 수정은 **whole-song loop를 트랙별 패딩이 아니라 전역 한 지점(LoopAudioSource)에서 wrap**하는 것이다 — 그러면 길이 변경 시 재디코드 없이 값 하나만 갱신하면 된다. 다만 잘 검증된 네이티브 루프 경로를 바꾸는 큰 작업이라, 재디코드 비용이 문제되면 그때 착수한다. 지금 수정은 **정확하되 무겁다.**

## v1.27.3 — 백킹 트랙 끝을 지나 녹음하면 자동정지가 테이크를 죽이던 버그 (2026-07-18)

> **재빌드: renderer.** 시험 대기 1건([시험.md](시험.md) T-1.27.3-1). 사용자가 T-1.27.0-3② 시험 중 발견 — **플레이헤드 0 복귀 · 파형/소리 불일치 · 검은 화면 크래시**의 근원.

### 재현 (사용자 보고)

① 3:10 파일 트랙 준비(프리롤 4초) → ② Audio In에 10초 녹음 후 정지 → ③ 그 클립을 3:30으로 이동 → ④ **3:20에 seek 후 Record**. 프리롤 없이 count-in 후 녹음이 시작되는데 **플레이헤드가 0에서 움직이며 소리가 남**. 이후 3:20에서 재생을 몇 번 하면 **loop되며 파형과 소리가 어긋나다 앱이 검게 꺼짐.**

### 원인 — 백킹 끝을 지난 녹음에서 자동정지가 첫 틱에 발동

`startRecordFlow`의 `songEnd`는 **대상(Audio In) 트랙을 뺀** 다른 트랙들의 끝이다([app.jsx](app.jsx)). 이 시나리오에선 파일 트랙만 남아 **songEnd = 190초(3:10)**. 그런데 녹음은 **200초(3:20)**에서 시작한다. 자동정지 감시가:

```js
if (end > 0 && (DAW.getPlayhead() >= end - 0.03 || !DAW.isPlaying)) doStopRecording();
```

첫 120ms 틱에서 곧바로 `200 >= 190` → **즉시 `doStopRecording`**. `DAW.stop()`이 **플레이헤드를 0으로** 되돌리고, 그 찰나에 만들어진 **깨진 초단타 테이크가 200초에 놓여** v1.27.2 bake로 큰 버퍼가 되며 네이티브 duration/loop와 어긋난다 → 파형·소리 불일치 → 반복 재생 시 상태가 꼬여 크래시.

### 왜 지금인가

**기존 버그다.** Stage 1(v1.25.0)의 자동정지는 "녹음이 백킹 트랙 **안**에서 시작한다"를 암묵 전제했다. 백킹 끝을 지나 녹음하는 경우가 드물어 안 드러났을 뿐이다. T-1.27.0-3②("음악이 2:00에 끝나는 곡의 3:00에서 녹음")가 정확히 이 전제를 깼다. (v1.27.2 bake는 크래시를 **키운** 요소지 원인은 아니다 — 원인은 자동정지.)

### 수정

**녹음이 songEnd에서(또는 그 뒤에서) 시작하면 자동정지 기준을 끈다**(`end = 0` → 백킹 트랙이 없을 때와 동일하게 **수동 Stop까지 녹음**). `punchAt`(모든 경로의 테이크 시작 위치)이 `songEnd`보다 앞设일 때만 songEnd를 자동정지에 쓴다:

```js
const recordEnd = (songEnd > 0 && punchAt < songEnd - 0.05) ? songEnd : 0;
```

이러면 3:20 녹음은 즉시 정지되지 않고 정상적으로 200초에 테이크를 놓고 수동 Stop까지 녹음한다. 백킹 안에서 시작하는 일반 녹음(playhead < songEnd)은 **무변경**.

⚠️ **크래시는 재확인 필요.** 근원(즉시 자동정지 → 깨진 테이크)을 없앴으므로 연쇄도 사라질 것으로 보지만, 네이티브 duration/loop 불일치(desync)가 **독립적으로** 남아 있을 가능성은 T-1.27.3-1에서 검증한다.

## v1.27.2 — start>0 녹음이 0:00에서 소리 나던 버그 (2026-07-18)

> **재빌드: 없음**([audio-engine.js](audio-engine.js)만 수정 — raw 로드). 시험 대기 1건([시험.md](시험.md) T-1.27.2-1). **프리롤이 실제로 동작하려면 반드시 필요한 수정.**

### 증상 (사용자 보고)

- 정지 상태(곡 맨 앞)에서 녹음 → 정상.
- **재생 도중 녹음 → 파형은 보이는데 소리가 안 남.**

### 원인 — start>0 테이크가 bake 안 된 채 네이티브로 감

`attachRecording`([audio-engine.js](audio-engine.js))은 `start`(punch-in 위치)에 클립을 놓지만 **bake하지 않았다.** 그래서:

- `track.buffer` = **원본 디코드**(오디오가 샘플 0에 있음), `_layoutBaked = false`.
- 그런데 브리지 `trackIsBakedLayout()`([audio-bridge.js](audio-bridge.js))은 baked 여부를 **`!_isTrivialLayout`로 추정**한다. start>0이면 이미 non-trivial → **"baked"라고 잘못 보고**.
- `syncTrackToNative`가 그 말을 믿고 **원본 PCM을 `startSeconds=0`으로** 네이티브에 넘긴다.

결과: 파형은 clip.start(예: 1:20)에 그려지는데 **소리는 0:00에 실린다.** 그 자리(1:20)엔 소리가 없다 = "파형 있는데 소리 없음". start=0 녹음은 trivial이라 filePath 경로로 정상이었다(그래서 정지 녹음만 됐다).

### 왜 지금 드러났나

**기존 버그다** — start>0 녹음은 원래 이랬다. 다만 흔한 경로가 "곡 맨 앞에서 정지 녹음"(start=0)이라 안 드러났다. **v1.27.0 프리롤은 항상 punch-in 위치(start>0)에서 녹음**하므로, 프리롤을 켜는 순간 이 버그가 상시화된다. 즉 프리롤 기능이 성립하려면 이 수정이 전제다.

### 수정

`attachRecording`이 클립을 만든 뒤 **`_ensureBaked(track)`를 호출**한다(전략 B 불변식: non-trivial 레이아웃은 반드시 bake). start>0이면 `track.buffer`가 **타임라인 인덱싱 baked 버퍼**(앞부분 침묵 + 테이크)가 되고 `_layoutBaked=true` → `trackIsBakedLayout`의 보고가 사실이 되며 네이티브가 baked PCM을 받아 **제 위치에서 소리 난다.** start=0 테이크는 trivial로 남아 기존 filePath 빠른 경로 그대로(무변경).

- 영향 범위: 재생 중 녹음 · **프리롤 punch-in** · 곡 중간 count-in 녹음 — start>0로 끝나는 모든 녹음. 웹 폴백 재생도 함께 고쳐진다(baked 버퍼를 타임라인으로 재생).
- 비용: 중간 지점 녹음은 "앞부분 침묵 + 테이크"를 temp WAV로 밀어 넣는다(전략 B에서 클립을 그 위치로 옮긴 것과 동일한 비용 — 기존 패턴과 일관).

## v1.27.1 — `[rawBuffer]` 경고 오탐 축소 (2026-07-18)

> **재빌드: renderer만.** 시험 대기 없음(진단 문구 범위만 좁힘 — 반환값·동작 무변경). T-1.26.1 시험 중 사용자가 보고한 콘솔 경고 정리.

### 증상

T-1.26.1-2 시험 중 정상 편집인데도 콘솔에 **`[rawBuffer] source not registered — falling back to primary`** 가 반복해서 떴다.

### 원인 — 경고가 정상 경로에서 울렸다 (오탐)

v1.26.1에서 "미등록 source에 폴백하면 남의 오디오가 나갈 수 있다"며 붙인 경고([audio-engine.js](audio-engine.js) `_rawBufferForSource`)가 **너무 넓게** 잡았다. `sourceId`가 있고 미등록이면 무조건 울렸는데, 그중 **완전히 정상인 경로**가 있다:

- `getSnapshot`([audio-engine.js](audio-engine.js))은 **Undo를 쌓을 때마다 모든 트랙**에 대해 `_rawBufferForSource(t, sources[0].id)`를 호출한다 — 그것도 `_captureRawBuffers` 없이. 아직 편집한 적 없는 파일 트랙(예: 불러온 SUNO 스템)은 `_rawBuffers`가 비어 있어 조회가 빗나가고 → **경고**. 하지만 caller는 곧바로 `|| t.buffer`로 **올바른 raw를 쓴다.** 대체(substitution)가 일어나지 않으므로 문제가 아니다.
- 편집을 여러 번 하면 매 Undo 스냅샷이 스템 트랙을 훑어 **경고가 반복**됐다.

### 수정 — 실제로 오디오가 바뀌는 경우에만 경고

경고 조건을 **"clip이 비-primary source를 지목했는데 그게 미등록이고, 그런데 primary는 등록돼 있다"** 로 좁혔다 — 이때만 정말로 **다른 테이크의 오디오가 대신 나간다.** primary를 찾다 빗나가거나(스냅샷 케이스) 아무것도 등록 안 됐으면 대체가 아니라 `null` 반환 → caller가 `t.buffer`로 폴백하는 정상 경로다. **반환값 로직은 그대로**, 언제 `console.warn` 하느냐만 바꿨다. 문구도 `substituting primary audio`로 명확히 했다.

## v1.27.0 — Stage 2b: 프리롤 녹음 (2026-07-17)

> **재빌드: renderer만**(네이티브 무변경). 시험 대기 **5건**([시험.md](시험.md) T-1.27.0-1~5). **Stage 2의 메트로놈 결정을 사용자 지적으로 뒤집은 결과물.**

### 왜 — 고정 BPM 클릭은 이 앱의 콘텐츠에 맞지 않는다 (사용자 지적, 2026-07-17)

Stage 2를 "클릭을 녹음 중에도 울리게 확장할까"로 이어가려던 참에 사용자가 **전제 자체를 반박**했다. 요지:

- 이 앱은 **MIDI가 아니다.** SUNO **스템을 불러와 그 위에 보컬/악기를 덧녹음**하는 도구다. 눈금자를 **시간 기준으로만** 만든 것도 같은 이유다.
- **문제 1 — SUNO 음악은 중간에 박자가 불규칙하게 바뀌거나 미묘하게 움직인다.** 하나의 BPM으로 만든 격자는 곡을 따라가지 못하고 갈라진다.
- **문제 2 — BPM이 맞아도 sync가 안 맞는다.** 클릭 격자는 곡의 **박자 위상**을 알 방법이 없다.

**타당하다. 그리고 문제 2는 이미 v1.27.0 이전 count-in에서 발생하고 있었다** — 4박 클릭이 "재생 위치에서 거꾸로 4박"으로 찍히므로, 녹음 시작 위치가 곡의 위상과 안 맞으면 count-in부터 어긋난다(0초에서 시험하면 드러나지 않는다). 근본 원인은 하나다: **`BPM 숫자 → 격자` 모델이 틀렸다. 진실은 숫자가 아니라 오디오 안에 있다.**

검토한 선택지: **(A) 프리롤** — 클릭 대신 **곡 자체를 리드인으로** 재생. **(B) 비트맵** — 스템에서 비트 시각 배열을 추출해 그 타임스탬프로 클릭(문제 1·2 모두 해결, 그러나 비트 추적 + 저장 + 네이티브 위치 공유가 필요해 비싸다). **(C) 유지.**

→ **A 채택**(사용자 결정). 문제 1·2가 **원천적으로 없다** — 리드인이 곧 그 음악이니 박자도 위상도 **정의상** 일치하고, 분석도 BPM도 필요 없다. Stage 6 Punch Recording이 어차피 프리롤을 요구하므로 그대로 재사용된다. B는 "클릭이 정말 필요하다"가 확인되면 그때 판단한다.

### 한 일

- **프리롤 녹음** — 정지 상태 Record 시 `punchAt - N초`로 되감아 **기존 음악을 재생**하다가 원래 위치에서 녹음을 시작한다. `startRecordFlow`에 `preroll` 옵션 추가.
- **길이 2 / 4 / 8초** — 트랜스포트 프리롤 버튼을 누를 때마다 `2 → 4 → 8 → 끔` 순환(버튼에 현재 초 표시, 기본 **4초**, `localStorage` 저장). **마디가 아니라 초인 이유는 위 문제 1·2 그 자체다** — 알 수 없는 박자를 단위로 쓸 수는 없다.
- **비침습 PRE-ROLL 칩** — 상단에 남은 시간(0.1초 단위)만 띄운다. count-in의 전체화면 딤을 쓰지 않는 이유: 프리롤은 **플레이헤드가 음악을 향해 다가오는 걸 보며 진입을 가늠하는 시간**인데, 오버레이가 바로 그 타임라인을 가린다.
- **폴백** — 프리롤이 성립하지 않으면 **기존 count-in 그대로**. 성립 조건 3가지는 아래.
- Help 매뉴얼(한/영) 갱신 — 프리롤 행 + "왜 프리롤인가" + 폴백 조건.

### 설계에서 걸린 함정 3개

**① 클립 위치를 `punchAt`에 고정하면 테이크가 앞으로 밀린다.** [app.jsx](app.jsx) `toggleRecording`은 `session.start != null ? session.start : DAW.getPlayhead()`로 클립 위치를 정한다. 프리롤에서 `start = punchAt`으로 고정하면 클립은 `punchAt`에 놓이는데, 녹음기는 **폴링 틱 + 장치 open `await`** 뒤에 시작한다. count-in과 달리 **그 사이 트랜스포트가 굴러가고 있으므로** 실제 캡처는 `punchAt + δ`부터다 → 테이크가 δ만큼 앞으로 밀린다. → **`start`를 null로 두어** 녹음기가 실제 시작하는 순간의 플레이헤드를 읽게 했다(캡처와 배치가 자기정합). 기존 주석이 정확히 이 함정을 경고하고 있었다.

**② 카운트다운을 타이머로 세면 안 된다.** 누른 시점부터 N초를 세는 대신 **플레이헤드를 감시**한다(`punchAt - getPlayhead()`). 재생 시작이 늦거나 멈칫해도 **일찍 punch-in 하지 않는다** — 트랜스포트가 시계다.

**③ Repeat 구간과 프리롤은 양립 불가 → 폴백.** [audio-bridge.js](audio-bridge.js) `getPlayhead()`는 Repeat 구간이 켜져 있으면 플레이헤드를 **구간 안에 가두고**(clamp), `play()`는 구간 밖 플레이헤드를 **구간 시작으로 끌어온다**. 즉 구간 앞으로 되감는 프리롤은 즉시 도로 끌려 들어간다. → `!DAW.repeatPlayEnabled`를 성립 조건에 넣고 count-in으로 폴백한다. 루프 + punch 의미는 **Stage 3/6에서 함께 설계**한다.

### 프리롤 성립 조건 (아니면 count-in)

| 조건 | 왜 |
|---|---|
| 프리롤 길이 > 0 · 정지 상태에서 Record | 재생 중 Record는 지금도 즉시 녹음(무변경) |
| **Repeat 구간이 꺼져 있다** | 함정 ③ |
| 플레이헤드 > 0.5초 | 맨 앞에서는 되감을 리드인이 없다 |
| **다른 트랙의 음악이 프리롤 구간에 걸친다**(`songEnd > punchAt - N`) | 음악이 2:00에 끝났는데 3:00에서 녹음하면 침묵만 4초 듣게 된다 |

### 취소

프리롤 취소는 count-in 취소와 달리 **되돌릴 게 있다** — 트랜스포트를 되감아 재생까지 시작했으므로, 정지 후 **플레이헤드를 Record 누른 자리(`punchAt`)로 복원**한다. 그냥 `stop()`만 하면 0초로 떨어져 punch-in 하려던 지점을 잃는다.

### 메트로놈은 남는다 (지우지 않은 이유)

빈 프로젝트나 곡 맨 앞처럼 **들려줄 음악이 없는 경우**엔 클릭이 유일하게 쓸모 있다. 그 자리에서만 쓰인다.

### 검증

- ✅ 정적: 문법 · 전역 심볼 충돌 검사(상시 노트 절차) — 신규 심볼 충돌 없음(기존 기지 중복 `buildInputPortOptions`만 남음).
- ❌ **화면 확인 못 함** — 검증 시점에 전체화면 게임이 포그라운드를 잡고 있어 앱 창을 띄워 스크린샷을 찍으면 사용자를 방해하게 되어 중단했다. **프리롤 버튼 렌더·칩 표시는 미확인**이며 T-1.27.0-1이 첫 확인이 된다.

## v1.26.1 — F2 선행 정리: 다중 source 배관 + 재녹음 시 상태 누수 버그 (2026-07-17)

> **재빌드: renderer만**(네이티브 무변경). 시험 대기 **1건**([시험.md](시험.md) T-1.26.1-1). Stage 3 착수 전 정리.

### 조사하다 실제 버그를 찾았다 — 재녹음 후 편집하면 테이크가 사라진다

F2("`attachRecording`이 배열째 재대입해 Take 누적 불가")를 보러 갔다가, **Take와 무관하게 지금 재현되는 버그**가 나왔다. `attachRecording`이 `_rawBuffers`와 `_layoutBaked`를 **둘 다 건드리지 않는다**(확인: 해당 함수에 두 식별자가 0회 등장). 그래서 이전 테이크가 남긴 상태가 새 테이크에 그대로 붙는다.

**재현 경로**: 녹음 → **클립 이동**(여기서 bake가 일어나며 `_rawBuffers={A:rawA}`, `_layoutBaked=true`) → **재녹음**(`buffer=rawB`, `sources=[B]`, 그러나 `_rawBuffers`는 여전히 `{A:rawA}`, `_layoutBaked`는 여전히 `true`) → **아무 편집**.

두 필드가 **각각** 독립적으로 트랙을 깬다:

| 남은 상태 | 결과 |
|---|---|
| `_rawBuffers`가 **옛 source id**만 매핑 | 새 source가 등록된 적 없으니 재-bake가 조회 → miss → `[bake] clip skipped — no raw buffer` → **테이크가 사라진다** |
| `_layoutBaked`가 **true로 잔류** | `track.buffer`는 raw 디코드로 돌아왔는데 Waveform은 **타임라인 인덱싱**으로 읽는다(상시 노트 "버퍼 인덱싱 이원성") → **파형이 틀리게 그려진다** |

수정: `attachRecording`이 교체 시 `_rawBuffers = {}` · `_layoutBaked = false` · `sources = []`로 **이전 레이아웃 상태를 함께 철거**한다. 덤으로 옛 source의 raw 버퍼가 계속 붙어 있던 **메모리 누수**도 사라진다.

### F2의 진짜 구조적 장벽은 `_captureRawBuffers`였다

`if (track._rawBuffers) return;` — **첫 호출이 이기고 이후로는 아무것도 추가되지 않는다.** 즉 트랙당 **등록 가능한 source가 사실상 1개**였다. Consolidate(v1.21.7)가 이미 이 벽을 만나 `_rawBuffers`를 직접 쓰는 우회로로 살고 있었고(그래서 다중 source 트랙이 지금도 존재한다), **Stage 3의 Take 누적은 정확히 같은 벽에 부딪힐 자리였다** — Take B의 raw가 등록되지 않아 `_rawBufferForSource`가 primary로 폴백해 **Take B 클립에 Take A 오디오가 구워지는** 조용한 사고.

- `_captureRawBuffers`를 **가산적(additive)**으로 바꾸고, `track.buffer`가 **아직 raw일 때만**(`!_layoutBaked`) primary를 시딩하게 했다(baked 결과를 "source"로 저장하면 재-bake가 자기 출력을 먹는다).
- **`_registerSource(track, meta, rawBuffer)` 신설** — sources 추가와 raw 등록을 **한 번에** 한다. 이 둘이 어긋나면 클립이 침묵으로 구워져 사라지므로 갈라놓을 이유가 없다. `attachRecording`·`_consolidateClips`가 이걸 쓴다.
- `_rawBufferForSource`가 **sourceId를 명시했는데 미등록이면 `console.warn`** 한다. 지금까지는 조용히 primary로 폴백해 **남의 테이크 오디오를 대신 내주고도 정상처럼 보였다**. Stage 3 버그를 무음/오음이 아니라 로그로 드러내려는 것이다.

### 이번엔 일부러 안 한 것

- **clip은 여전히 교체(replace)다.** `track.clips = [새 클립]`을 append로 바꾸면 두 번째 테이크가 겹침 규칙(`_reindexClips`)에 밀려 **첫 테이크 뒤로 이동**한다 — Take 레인(Stage 4)도 `activeTakeId`(Stage 3)도 없는 지금은 그게 더 나쁜 중간 상태다. **동작 무변경**을 유지하고, append 능력만 `_registerSource`로 준비해 뒀다.
- **`_ensureBaked` 강제 호출도 안 했다.** `start > 0` 녹음을 bake하면 앞부분이 침묵으로 채워진 긴 버퍼 + temp WAV가 되는데, 지금은 브리지 `syncTrackToNative`가 **`startSeconds`로 위치를 넘겨** 가볍게 처리한다(raw 유지). 이걸 건드리면 검증된 네이티브 경로가 바뀌므로 Stage 3에서 Take 모델과 함께 판단한다.

### Stage 3이 얹힐 자리는 이미 모델에 있다

`clip.takeId`와 `track.takes[]`가 **이미 정규화·저장·스냅샷 경로에 전부 배선돼 있다**(Phase 1이 예약해 둔 것). 다만 **아무도 값을 넣지 않는다.** Stage 3은 `attachRecording`의 교체 3줄을 append로 바꾸고 여기에 값을 채우면 된다.

## v1.26.0 — Phase 6 Stage 2: 메트로놈 클릭 (2026-07-17)

> **재빌드: 네이티브 + renderer 양쪽.** Q2 결정의 구현. **2026-07-17 사용자 시험 5건 전부 통과**([시험-아카이브.md](시험-아카이브.md)) — 핵심인 **T-1.26.0-2(클릭이 녹음·Export에 섞이지 않는다)** 포함. **시험 대기 0건.**

### 한 일

- **네이티브 클릭 생성** — [AudioEngine.h](juce_skeleton/Source/AudioEngine.h)에 `MetronomeClick` 신설. 1568Hz(첫 박 강조) / 784Hz(나머지) **50ms 사인 버스트 + 지수 감쇠**를 장치 열 때 한 번 렌더해 두고, 오디오 콜백은 그 버퍼를 읽기만 한다(작업원칙 5 — 콜백에서 합성·할당 없음).
- **명령 2개** — `startCountIn(bpm, beats)` / `stopMetronome`. 브리지에 `Bridge.startCountIn` / `Bridge.stopMetronome` 추가.
- **count-in 1초 고정 → BPM 박자** — 메트로놈 On + 프로젝트 BPM 있음 → **1마디 4박**(`60/BPM`초 간격, 오버레이 `4 → 3 → 2 → 1`). Off이거나 BPM 없음 → **기존 무음 3×1초 그대로**.
- **On/Off UI** — 트랜스포트 Loop와 Record 사이에 메트로놈 버튼(`metronome` 아이콘 신설). 상태는 `localStorage`(`focusdaw-metronome`, **기본 On**)에 저장.
- **Help 매뉴얼 갱신**(한/영) — "3 → 2 → 1" 고정 문구 제거, 메트로놈 행 추가.

### 왜 클릭이 별도 device callback인가 (핵심 설계)

Q2의 **"클릭은 녹음 파일에 들어가면 안 된다"**(모니터 전용)를 **구조로** 보장한다. 재생 체인(`mixerSource → LoopAudioSource → Gain → MasterEffects`)의 노드로 넣으면 Export에 섞이므로, 대신 `deviceManager.addAudioCallback(&metronome)`으로 **독립 콜백**에 뒀다. 그 결과:

- **녹음 WAV에 못 들어간다** — `InputRecorder`가 쓰는 건 **입력** 샘플이고, 이 클래스는 **출력** 버퍼만 건드린다.
- **Export에 못 들어간다** — `exportMix`는 소스 체인을 **오프라인**으로 렌더하는데 여기 속하지 않는다.
- 마스터 볼륨/페이드의 영향을 받지 않는다(count-in은 그래야 한다).

⚠️ `AudioDeviceManager`는 보조 콜백에 **지워지지 않은** 출력 버퍼를 넘기고 그걸 믹스에 합산한다 — 손 안 댄 영역은 풀스케일 잡음이 된다(`InputRecorder`에 이미 기록된 함정). 그래서 콜백 진입 시 **항상 전체를 clear**한다.

### 박자 격자는 네이티브가, 숫자는 JS가 소유한다

클릭 위치는 콜백 **자기 샘플 카운터**에서 파생되므로 sample-accurate하고 드리프트가 없다(Stage 0과 같은 철학). 화면 숫자는 기존 `setInterval`이 계속 몰고, 간격만 `60/BPM`으로 바뀐다. **시계가 둘이지만** 합의가 필요한 구간이 **한 마디뿐**이라 둘 사이 오차는 클릭 대비 눈에 잡히는 수준 아래다. 녹음 시작은 지금도 장치 open `await` 뒤에 일어나 어차피 샘플 정렬이 아니다 — 그 정렬은 **Stage 3**의 몫이다.

## v1.25.4 — ⛑ 진단 전용: 장치 리스너가 왜 안 울리나 (2026-07-17)

> **기능 변경 없음.** ✅ **역할 완료 → v1.25.5에서 코드 전부 제거됨.** 아래는 무엇을 왜 넣었는지의 이력이다. **결과: 리스너는 정상 발화**(위 v1.25.5 참조).

### 왜 진단이 필요한가

T-1.25.3-4 실패로 **"JUCE 장치 리스너가 발화하지 않는다"** 가 유력해졌지만, 사실 **네 가지가 구분되지 않는다**: ① 리스너 미등록 ② 리스너 미발화 ③ 발화했으나 장치 목록을 갱신 전 값으로 읽음 ④ `openedInputName`이 비어 조기 반환. 코드만 읽어서는 더 못 좁힌다 — **엔진 로그는 `FOCUSDAW_VERBOSE=1` 게이트라 사용자에게 보이지 않으므로** 진단을 WS 이벤트로 브라우저 콘솔에 띄운다.

### 넣은 것

- `engineDiag` (connect 시): 리스너를 **몇 개 타입에 붙였는지**. → **개발자 측정에서 이미 `4`로 확인 — ①은 소거됐다.**
- `deviceListChangedDiag` (리스너 발화 시, **모든 조기 반환보다 먼저**): `openedInput` / `present` / `listenerTypes`. → USB 한 번 뽑으면 ②③④가 갈린다(아무것도 안 찍힘 = ②, `present=true` = ③, `openedInput=""` = ④).

### 참고 — 확인된 JUCE 동작

- `WASAPIAudioIODeviceType::systemDeviceChanged()`는 `scan()`으로 목록을 갱신한 **뒤** `callDeviceChangeListeners()`를 부른다 → 발화만 한다면 ③은 없어야 한다.
- `DeviceChangeDetector`는 **숨은 최상위 윈도우의 `WM_DEVICECHANGE` 브로드캐스트 + 500ms 지연 타이머**로 동작한다. 엔진은 `JUCEApplicationBase`라 메시지 루프가 있는데도 안 울린 것이라 ②라면 원인이 더 흥미로워진다(브로드캐스트 미수신 등).

## v1.25.3 — 장치가 없으면 ARM이 그렇다고 말한다 (2026-07-17)

### 배경 — 사용자가 찾아낸 설계 구멍

| 안 | 결정 |
|---|---|
| 재연결 시 **자동 재오픈** | ❌ **채택 안 함** — `closeAudioDevice()`는 **출력까지 함께** 닫았다 열기 때문에, 마이크를 꽂는 순간 **재생 중이면 소리가 ~3초 끊긴다.** |
| **ARM 토글로 복구** + UI가 상태를 정직하게 표시 | ✅ **채택** — 아래 2건 구현 |

### 구현

1. **ARM 누를 때 장치 유효성 검사 → 실패하면 Record와 똑같이 에러 창 + ARM 자동 해제** ([app.jsx](app.jsx) `reportInputFailure`). ARM 경로가 `.catch(console.warn)`으로 삼키던 것을 표면화했다. **불이 켜졌는데 아무것도 못 듣는 ARM 버튼은 거짓말이다.** 메인 창·믹서 창 양쪽 경로 모두 적용.
2. **장치를 잃으면 모든 ARM 자동 OFF** — 네이티브가 `audioDeviceListChanged`에서 **"우리가 연 그 입력"** 이 목록에서 사라졌는지 확인해(무관한 헤드폰 연결 등은 무시) `audioInputLost` 이벤트를 브로드캐스트 → 브리지가 `focusdaw-audio-input-lost` DOM 이벤트로 전달 → app.jsx가 `disarmAllInputs()`로 전 트랙 ARM 해제. **알림만 하고 아무것도 재오픈하지 않는다**(위 결정대로 재생 보호).

### ⚠️ 알아 둘 점

- **②는 `AudioIODeviceType::Listener` 발화에 의존한다.** 이 리스너가 실제로 발화하는지는 **아직 증명되지 않았다**(v1.25.2에서 복구가 동작한 건 리스너일 수도, 목록 비교 감지기일 수도 있다 — 구분이 안 된다). 리스너가 안 울리면 **②는 조용히 아무 일도 안 하지만 ①은 그대로 동작**하므로, 최악의 경우에도 v1.25.2보다 나쁘지 않다.
- **①은 리스너와 무관**하다(엔진의 이름 검사 결과를 그대로 보여줄 뿐).

### 시험 재설계 — T-1.25.2-2는 측정 불가능한 시험이었다 (사용자 지적)

"첫 녹음이 즉시 시작되는지"를 보라고 했는데, **정지 상태 Record는 무조건 3초 카운트인이 먼저 돈다.** 그 뒤 `setAudioInput`이 4ms였는지 3초였는지 **가려져서 구분할 수 없다.** → **콘솔의 `[AudioInputTiming] setAudioInput ready in Xms` 숫자를 직접 읽거나**(브리지가 이미 찍고 있다), **카운트인이 없는 "재생 중 Record" 경로**로 측정하도록 바꿨다.

## v1.25.2 — USB 재연결 복구 완성: JUCE가 재오픈을 조용히 거부하고 있었다 (2026-07-17)

> **재빌드: 네이티브 + renderer 양쪽.** 시험 [시험.md](시험.md) T-1.25.2-*.

### v1.25.1이 절반만 고친 이유 (시험 결과)

T-1.25.1-1 결과: **② 에러 표시 = 성공**(JS 캐시 제거가 맞았다). 그러나 **③ 재연결 후에도 무음 = 실패.** 즉 `deviceTopologyChanged` 플래그로 **우리 keep-warm은 제대로 건너뛰었는데도** 재오픈이 일어나지 않았다.

**원인: 나는 플래그를 세운 뒤 재오픈 작업을 JUCE에 넘겼는데, JUCE가 아무것도 하지 않았다.**

```cpp
// juce_AudioDeviceManager.cpp — setAudioDeviceSetup()
if (newSetup != currentSetup) sendChangeMessage();
else if (currentAudioDevice != nullptr) return {};   // ← 동일 setup + 열린 좀비 = 즉시 "성공"

const auto needsNewDevice = currentSetup.inputDeviceName  != newSetup.inputDeviceName
                         || currentSetup.outputDeviceName != newSetup.outputDeviceName
                         || currentAudioDevice == nullptr;                 // ← 전부 false
```

USB를 뽑았다 꽂아도 **setup의 모든 필드가 바이트 단위로 동일**하고 좀비 장치 객체가 살아 있으므로, JUCE는 **죽은 WASAPI 클라이언트를 그대로 둔 채 성공을 반환**한다. 우리 빠른 경로를 건너뛰어 봐야 그 다음이 no-op이었으니 아무 소용이 없었다.

### 수정

- **`deviceManager.closeAudioDevice()`를 먼저 호출** — `currentAudioDevice.reset()`으로 **nullptr**이 되면 위 두 관문이 모두 뚫려 JUCE가 `createDevice()`로 **장치를 새로 만든다**.
- **리스너 비의존 감지기 추가** — `AudioIODeviceType::Listener`가 정말 발화하는지는 **증명하지 못했다**(사용자가 본 ②의 에러는 리스너 없이도 나오는 이름 검사 결과다). 그래서 매 `setAudioInput`마다 스캔한 입력 목록을 직전 스냅샷과 비교해 변화를 잡는다. **스냅샷 갱신은 `unknown input device` 조기 반환보다 먼저** 일어나므로 사용자 재현 순서가 그대로 커버된다:
  - ② 분리 상태 녹음 → 목록에 마이크 없음 → 스냅샷 = "마이크 없는 목록" + 에러
  - ③ 재연결 녹음 → 목록에 마이크 있음 → **스냅샷과 다름** → close+rebuild → 정상
  - 리스너는 그대로 두되(중복 감지는 무해) **복구가 리스너에 걸려 있지 않다.**

### 실측 (엔진 직접 구동 — 감지기를 실제로 발동시켜 검증)

장치 타입을 바꾸면 입력 목록이 달라져 **USB 재연결과 동일한 코드 경로**(감지 → close → rebuild)를 탄다.

| 단계 | 결과 |
|---|---|
| Windows Audio로 마이크 열기 | ok=true 933ms |
| DirectSound로 전환(목록 변함) | ok=true 1650ms |
| 다시 Windows Audio 마이크 | ok=true **3245ms** ← 4ms 빠른 경로가 아닌 **진짜 close+rebuild** |
| **rebuild 후 실제 녹음** | 표본 124개 중 **110개가 0 아님**, 최대 0.02658 → **입력 생존 확인** |

> ⚠️ **여전히 미검증**: 실제 USB 물리 분리·재연결. 위는 "목록이 바뀌면 close+rebuild가 일어나고 그 후 입력이 살아있다"를 증명하며, 사용자 재현 순서가 그 조건을 만든다는 것은 코드 순서로 보장된다. 최종 확인은 T-1.25.2-1.

## v1.25.1 — 마이크 USB 분리 후 재연결해도 무음 녹음 (T-1.25.0-5 발견) (2026-07-17)

> **재빌드: 네이티브 + renderer 양쪽.** ✅ **②(에러 표시) 통과** / ❌ **③④(재연결 복구) 실패 → v1.25.2에서 완성.** 아래 원인 분석 ①②③은 유효하나, **수정은 불완전했다**(플래그를 세워도 JUCE가 재오픈을 거부 — v1.25.2 참조).
> ⚠️ **Phase 4 완료 기준 위반이 뒤늦게 드러난 건**: "입력 장치 없음, **장치 분리**, 디스크 오류가 사용자에게 명확히 표시된다"가 완료로 기록돼 있었으나, **장치 분리는 표시되기는커녕 감지도 못 하고 있었다.** Stage 1 시험(T-1.25.0-5)에서 사용자가 USB를 뽑아 보다 발견.

### 증상 (사용자 재현 순서)

1. Audio In 트랙 ARM 상태에서 **마이크 USB를 뽑는다**
2. 녹음 시작 → **에러 없이** 녹음이 진행되지만 **아무 소리도 안 들어온다**
3. 정지 → **USB를 다시 꽂고** 녹음해도 여전히 무음
4. 트랙을 지우고 새로 만들어도 무음
5. **앱을 완전히 껐다 켜야만** 정상

### 원인 — 캐시 2개가 겹쳐 있었다

**① [audio-bridge.js](audio-bridge.js)의 JS 캐시가 엔진에 묻지도 않았다 (주범)**

```js
const key = JSON.stringify(payload);
if (nativeState.audioInputKey === key) return Promise.resolve();  // ← 엔진 호출 자체를 건너뜀
```

`audioInputKey`는 **성공 시 저장되고 어디서도 무효화되지 않았다**(선언·조회·대입 3곳이 전부). 이 캐시는 *"같은 요청이면 같은 답"* 을 전제하는데, **답이 하드웨어에 달려 있다** — 마이크를 뽑으면 똑같은 요청이 이제 **실패해야** 맞다. 캐시가 메모리에서 성공을 돌려주는 바람에 엔진의 장치 존재 검사(`unknown input device`)에 **도달조차 못 했고**, 그래서 **에러가 안 뜬 것**이다(증상 2). 세션 내내 무효화가 없으니 재연결·트랙 재생성도 소용없고(3·4), **앱 재시작만이 `nativeState`를 비워** 고쳐졌다(5). 5단계 전부가 이 한 줄로 설명된다.

**② 그 아래 C++ keep-warm 빠른 경로가 좀비를 재사용한다 (공범)**

JS 캐시를 걷어내도 재연결이 안 낫는다. `setAudioInput`의 keep-warm 판정은 **우리 캐시 setup**과 **장치 객체**만 보는데 **둘 다 USB 분리를 견디고 살아남는다** — 이름도 그대로, `getActiveInputChannels()`도 그대로 → 전 필드 일치 → **같은 죽은 스트림을 그대로 돌려준다.**

**③ JUCE는 이 경우 아무것도 안 해준다 (확인 완료)**

```cpp
// juce_AudioDeviceManager.cpp — audioDeviceListChanged()
auto currentDeviceName = currentAudioDevice->getName();   // WASAPI in+out 장치 = 출력 이름
if (! currentDeviceStillAvailable) { closeAudioDevice(); ... }
```
WASAPI 입출력 통합 장치의 이름은 **출력 엔드포인트**다. 마이크만 빠지면 출력(Realtek/Focusrite)은 멀쩡하므로 `currentDeviceStillAvailable = true` → **JUCE는 장치를 닫지 않고, 입력만 좀비가 된다.**

### 수정 (두 층 모두)

- **JS**: `audioInputKey` 완료-캐시 **제거**. 이제 arm/record마다 엔진에 묻고, 장치가 없으면 **진짜 에러가 UI까지 올라온다**. 엔진에 keep-warm이 이미 있으므로 비용은 **WASAPI 재오픈이 아니라 왕복 4ms**다. (진행 중 중복 요청을 막는 `audioInputPending`은 별개이므로 유지.)
- **C++**: `AudioEngine`이 `juce::AudioIODeviceType::Listener`를 구현 → 핫플러그 시 `deviceTopologyChanged` 플래그 → **keep-warm 빠른 경로를 1회 거부**하고 진짜 재오픈. 성공적으로 열리면 플래그 해제.

### 실측 (엔진 직접 구동)

| 시험 | 결과 |
|---|---|
| 최초 열기 | ok=true **931ms**(실제 open) |
| 동일 요청 반복 | ok=true **4ms** → **keep-warm 무회귀**(v1.20.10 최적화 유지) |
| 없는 장치(= USB 분리와 같은 코드 경로) | **ok=false** `unknown input device: ...` → 에러가 앱까지 도달 |
| 실패 후 정상 장치 재요청 | ok=true 4ms |

> ⚠️ **미검증**: 위는 전부 개발자 측정이며, **실제 USB 분리·재연결은 물리 조작이라 확인하지 못했다.** 특히 **핫플러그 리스너가 실제로 발화하는지**(재연결 복구의 핵심)는 사용자 시험 T-1.25.1-1로만 확인된다.

## v1.25.0 — Phase 6 Stage 1: 녹음 상태머신 (2026-07-17)

> **재빌드: renderer만**(네이티브 무변경). 시험 [시험.md](시험.md) T-1.25.0-*. 변경은 [app.jsx](app.jsx) 한 파일.

### 무엇이 문제였나 — 상태가 `await` 뒤에 있었다

녹음 진행 상태에 **주인이 없었다.** "지금 녹음 중인가?"를 `DAW.tracks.some(t => t.recording)`으로, "카운트인 중인가?"를 `recordCountRef.current != null`로 **추론**했다. 그런데 `track.recording`은 **`await`가 끝난 뒤에야** true/false가 된다:

- **시작**: `prepareRecordingPath`(IPC) → `setAudioInput`(WASAPI 장치 오픈, 수백 ms) **→ 그 다음** `track.recording = true`. 이 사이에 Record를 다시 누르면 `isRecordingActive()`가 아직 false라 **두 번째 레코더가 시작**됐다.
- **정지**: `stopRecording` → `await promise` → `finalizeRecording` → `readAudioFile` → `attachRecording` **→ 그 다음** `track.recording = false`. 이 사이에 Stop을 다시 누르면 같은 테이크를 **두 번 finalize·attach**했다.
- **시작 중 Stop**: `doStopRecording`이 `t.recording`으로 대상을 찾는데 아직 false → **트랜스포트만 멈추고 레코더는 그대로 시작**돼 계속 녹음됐다.

### 무엇을 했나

- **명시적 상태머신**: `idle → armed → countIn → recording → stopping → finalizing → armed`. `idle`/`armed`는 **ARM에서 파생**(트랙 속성이지 흐름 상태가 아니다)하고, 머신은 바쁜 4개 phase만 소유한다(`recordPhaseRef`). 모든 전이는 **`enterRecordPhase()` 한 곳**을 지나며 **불법 전이는 거부 + `console.warn`**(스냅샷 가드와 같은 패턴).
- **위 3개 경합 전부 차단**: phase를 **`await` 이전에 선점**하므로 두 번째 Record가 들어올 자리가 없고, `stopping`/`finalizing` 중의 Stop/Record는 no-op이며, 시작 중 Stop은 phase를 놓아 **시작 경로가 스스로 중단**한다(`recording → null` 중단 엣지).
- **`startRecordFlow({ trackId, start, end, loopEnabled, countIn })`** 신설 — Stage 1이 요구한 구간 지정 녹음 API. 세션 1개(`recordSessionRef`)가 흐름 전체에서 진실원본이 된다. 기존 `beginRecordFlow()`는 `startRecordFlow()`(전부 기본값)로 남겨 **동작 무변경**. Stage 2(countIn=박자)·Stage 3(loopEnabled=Take 분할)·Stage 6(start/end=Punch)이 이 시그니처에 얹힌다.
- **`window.__recordCountdownActive`는 미러로 유지** — ARM 잠금(v1.20.11)이 다른 렌더러에서 이걸 읽는다. 두 번째 진실원본이 되지 않도록 `enterRecordPhase`만 쓴다.
- **`track.recording`은 그대로 뒀다** — app.jsx·ui-mixer·ui-tracks의 16곳이 읽는 UI 플래그이고, 라이브 파형(`_recordingPeaks`)도 여기 물려 있다. 이제 **진실원본이 아니라 머신이 미러링하는 값**이다.

### ⚠️ 이 작업에서 자체 발견·수정한 것

- **전이표에 중단 엣지가 빠져 있었다**: `recording: ["stopping"]`만 두고 `null`을 안 넣어, **실패한 시작 3경로**(장치 오류·엔진 reject·시작 중 Stop)가 전부 `recording`에 **영구히 갇혀 Record 버튼이 죽는** 버그. 전이표를 코드에서 **추출해 도달성(dead-end)·경합 13케이스를 자동 검사**하다 발견 → `recording: ["stopping", null]`로 수정. 검사 스크립트가 없었으면 못 봤을 버그다.
- **`start`를 흐름 진입 시점에 고정하려다 되돌림**: 세션에 `start: DAW.getPlayhead()`를 기본값으로 넣었더니, **재생 중 녹음**에서 장치 오픈(수백 ms) 동안 플레이헤드가 움직이는 만큼 **클립이 앞당겨 배치**된다. → 기본값을 `null`(= "레코더가 실제로 시작하는 순간의 플레이헤드")로 두고, **명시적으로 준 경우에만** seek + 그 값 사용.

### 남은 것 (Stage 1 범위 밖 — 의도적)

- `loopEnabled`는 **세션에 실려만 있고 동작하지 않는다**. 지금도 녹음 시작 시 곡 전체 Repeat(`loopEnabled`)는 끄고 시작한다(기존 동작 유지). 회차별 Take 누적은 **Stage 3**이고, 그 전에 **F2**(`attachRecording`이 `sources[]`/`clips[]`를 배열째 재대입 → Take 누적 불가)를 먼저 풀어야 한다.
- 구간 끝 정지는 여전히 **120ms 폴링**(`startAutoStopMonitor`). 샘플 단위 경계 분할은 Stage 3에서 네이티브 콜백으로 내린다(Stage 0의 `LoopAudioSource`와 같은 콜백).

## v1.24.1 — Repeat 구간을 프로젝트에 저장/복원 (2026-07-17)

- 검증(단위): 정상 구간 통과 / 필드 부재·null·역전·비수치 → `null` / 끝을 걸친 구간 → duration으로 클램프.

## v1.24.0 — Phase 6 Stage 0: 네이티브 sample-accurate loop range (2026-07-17)

> ✅ **2026-07-17 사용자 시험 통과** — T-1.24.0-1(약 **1시간 연속 repeat** 무문제)·1b·2·3 전부 정상. T-1.24.0-4는 **전제가 틀린 무효 항목**으로 판명(위 v1.24.1 참조). 상세 [시험-아카이브.md](시험-아카이브.md) v1.24.0.

> 착수 전 조사의 **F1(최대 리스크·선행 과제)** 해소. **Q1 결정**(네이티브만 sample-accurate, 웹은 현행 rAF 폴링 유지) 그대로 구현. **재빌드: 네이티브 + renderer 양쪽 필요.**

### 무엇이 문제였나

Repeat 구간 반복이 **JS 폴링 + seek 되감기**였다. [audio-bridge.js](audio-bridge.js)가 `setInterval` 30ms로 플레이헤드를 보다가 **이미 루프 끝을 지나간 뒤에야** `seek`를 보냈고, `position` 브로드캐스트(100ms)에서도 같은 되감기를 했다. 그래서 **회차마다 길이가 달라지고**(0~30ms+ 지터) 그 오차가 **누적**됐다. 네이티브 C++에는 loop range 개념 자체가 없었다(곡 전체 `setLooping`뿐). Phase 6은 **회차 1개 = Take 1개**이므로 이 구조로는 완료 기준 3개를 원리적으로 충족할 수 없었다.

### 무엇을 했나

- **`LoopAudioSource` 신설**([AudioEngine.h](juce_skeleton/Source/AudioEngine.h)) — 그래프에 `mixerSource → LoopAudioSource → GainAudioSource → MasterEffectsAudioSource`로 삽입. 오디오 콜백 안에서 **루프 끝에 블록을 분할**하고 그 자리에서 모든 트랙 트랜스포트를 `setPosition(loopStart)`로 되감는다.
  - **누적 밀림이 원리적으로 불가능한 이유**: 분할 지점을 "블록 안에서 어디였나"가 아니라 **경계까지의 절대 거리**로 계산하고, 되감기가 **절대 위치 지정**이다. → 매 회차가 정확히 `floor((end - start) × 출력레이트)` 샘플, 블록 위상과 무관하게 항상 동일.
  - 위치 단위는 **프로젝트 타임라인 초**(JS가 보내는 값과 동일). Vari BPM은 타임라인이 실시간보다 `playbackBpm/projectBpm`배 빠르게 흐르므로 `timelineRate`(= SoundTouch가 늘이는 바로 그 비율, Vari BPM off면 **정확히 1.0**)로 환산한다. 녹음이 필요로 하는 sample-accurate 경로는 Vari BPM off 케이스이고 거기서 이 값은 근사가 아니다.
- **`setLoopRange(start, end, enabled)`** — 엔진 API + `setLoopRange` WS 명령([WebSocketServer.cpp](juce_skeleton/Source/WebSocketServer.cpp)) 신설. play/pause/stop/seek/clearTracks가 루프 소스 위치·구동 상태를 동기화한다.
- **JS 폴링 되감기 제거**([audio-bridge.js](audio-bridge.js)) — `startLocalTickLoop`의 30ms 되감기와 `position` 핸들러의 되감기를 **삭제**했다. 브로드캐스트는 최대 100ms 전 상태라 지금은 엔진이 이미 되감은 뒤이고, 여기서 seek를 또 쏘면 **엔진과 싸우기만 한다.** tick loop은 이제 UI 갱신 전용. `Bridge.getPlayhead()`의 벽시계 외삽은 같은 범위를 모델링하므로 그대로 유지(플레이바 표시용).
- **`pushLoopRangeToNative()`** — 네이티브는 `importProject`가 없어 **개별 명령으로만 상태가 산다**(상시 노트). 그래서 `setLoop`을 밀던 3곳(connect·import·applySnapshot) 전부와 `setLoopRange`/`setRepeatPlayEnabled`에서 함께 민다. 안 하면 **재시작·프로젝트 복원 후 Repeat가 조용히 곡 전체 반복으로 되돌아간다.** 루프 구간 드래그가 mousemove마다 `setLoopRange`를 호출하므로 **동일 값 중복 전송은 dedupe**하되, 재연결 시엔 캐시를 비워(`lastLoopRangeSent = ""`) 새 엔진이 반드시 값을 받도록 했다.
- **Export 무영향**: 오프라인 export는 **같은 그래프**로 렌더하므로(`masterEffects → gain → loop → mixer`) export 중 `setBypassed(true)`. export는 항상 전체 타임라인이다.
- **오디오 스레드 안전**(작업원칙 5·6): 되감기는 juceTracks 벡터를 순회하므로 **`tracksLock` + `ScopedTryLock`** 사용 — 오디오 스레드는 **절대 대기하지 않고**, 트랙 설치와 충돌한 드문 블록에선 되감기를 건너뛴다(다음 블록이 처리). 명령 스레드는 항상 `engineMutex` **안에서** 이 락을 잡고 벡터를 바꾸므로(추가/교체/삭제 4곳) 오디오 스레드가 파괴된 트랙을 밟을 수 없다. 락 순서는 항상 `engineMutex → tracksLock` 한 방향뿐이라 데드락이 없다.
  - 되감기 시점은 **믹서 pull과 pull 사이**라 어떤 트랜스포트도 자기 `getNextAudioBlock` 안에 있지 않다(재진입 없음).

### ⚠️ 알아 둘 한계 (동작 영향 미미 · 재발 시 참고)

`transportSource->setPosition()`을 **오디오 스레드에서도** 호출하게 됐다(기존엔 메시지 스레드의 `seek()`만). JUCE `AudioTransportSource::setNextReadPosition`은 **callbackLock을 잡지 않으므로**(JUCE 원본 확인), 사용자 seek이 **루프 경계와 같은 순간**에 겹치면 두 위치 지정이 경합할 수 있다. 다만 ① 실제 재배치의 무거운 부분인 `ResamplingAudioSource::flushBuffers`는 **자체 락으로 직렬화**되고 ② 나머지는 정렬된 int64 저장이라 **메모리 손상이 아니라 "둘 중 하나가 이긴다"** 수준이며 ③ 창이 수 ns다. 증상이 난다면 "루프 경계와 정확히 겹친 seek이 한 번 무시됨" 정도다. 근본 제거는 seek을 오디오 스레드로 지연 적용(`pending seek`)해야 하는데, 잘 검증된 seek 경로를 흔드는 변경이라 Stage 0 범위 밖으로 둔다.

### 실측 (구현 직후, 엔진 직접 구동)

| 시험 | 결과 |
|---|---|
| 2초 루프(4.0~6.0s) 9초 재생 | 되감기 4회, **JS seek 0회**(엔진 단독 wrap). position 최대 6.0001s = **경계 샘플 1개(0.1ms) 이내** — 구 폴링의 30ms+ 초과와 대비 |
| **62초 연속 루프(31회차) 드리프트** | 이상적 루프 모델 대비 잔차: 첫 20초 1.8ms → 40초 이후 1.7ms = **드리프트 −0.1ms**(측정 바닥 이하). 구 30ms 폴링이면 같은 구간에서 **~300ms+** 밀렸을 값 |
| Repeat OFF 회귀 | min 2.00 → max 9.97, **되감기 0회** — 통과 재생 무변화 |
| **재생 중 Repeat ON(구간 안, seek 없음)** | 아래 버그 수정 확인 — 무루프로 0부터 재생 중 5초 지점에서 `setLoopRange[4,8]`만 전송 → max 7.99s, **끝을 넘지 않고 정상 wrap** |

> 🐛 **구현 중 자체 발견·수정**: 루프가 **꺼져 있는 동안에는 `LoopAudioSource`의 위치가 갱신되지 않는데**, JS는 플레이헤드가 구간 **밖일 때만** seek을 보낸다. 즉 **재생 중 구간 안에서 Repeat를 켜는 흔한 조작**에서 엔진이 직전 play/seek 시점의 낡은 위치로 루프를 무장해 **엉뚱한 지점에서 되감을** 수 있었다. → `setLoopRange`가 무장 전에 **살아 있는 트랜스포트에서 위치를 재동기화**하도록 수정(위 표 4행이 그 확인).

## ⚠️ 상시 개발 노트 — 버퍼 인덱싱 이원성 (전략 B, 반드시 숙지)

> Phase 5에서 도입된 **레이아웃 베이크(전략 B)**의 결과로 생긴 **상시 제약**이다. 완료 기록은 아카이브에 있으나, 이 노트는 이후 모든 파형/재생/피크 코드에 계속 적용되므로 여기에 남긴다.

전략 B에서 `track.buffer`는 두 가지 인덱싱 체계 중 하나이며, **`track._layoutBaked` 플래그가 이를 구분한다.** 이 둘을 혼동하면 **이동한 클립이 빈 파형으로 그려지는 버그**가 재발한다(2026-07-14 실제 발생, 수정 완료).

| 상태 | 버퍼 | 오디오 위치 | 파형 샘플링 기준 | 재생 |
|---|---|---|---|---|
| `_layoutBaked = true` (baked) | 평탄화 렌더 결과 | **타임라인 위치**(`clip.start`) | `clip.start` 기준(타임라인) | `_offset`(플레이헤드)로 그냥 재생 |
| `_layoutBaked = false` (raw/trivial) | 원본 디코드 버퍼 | 소스 원점 | `clip.offset` 기준(소스) | 동일(정렬돼 있어 무관) |

- `_ensureBaked`가 bake하면 `_layoutBaked=true`, trivial로 raw 복원하면 `false`로 설정한다.
- [ui-tracks.jsx](ui-tracks.jsx) `Waveform.bucketAt`는 `track._layoutBaked`로 분기해 `bufPos`를 **타임라인(clip.start) vs 소스(clip.offset)** 로 계산한다. peaks 배열도 `_ensureBaked`가 baked 버퍼에서 재계산하므로 타임라인 인덱싱이다.
- **새 파형/재생/피크 코드를 추가할 때**: baked 버퍼를 `clip.offset`로 접근하면 안 된다(그 위치는 침묵). 반드시 `_layoutBaked`를 확인하고 baked면 타임라인 위치로 접근.
- 재접속(`_assignDecodedToTrack`)은 raw 디코드 후 non-trivial이면 `_ensureBaked`를 호출해 재시작 후에도 이 불변식을 복원한다.
- 관련: `_rawBuffers[sourceId]`(원본 보관), `_isTrivialLayout`(bake 스킵 판정), `clip.gain`은 baked 버퍼에 반영·`clip.params.volume`/automation은 합성커브(이중적용 방지).

## ⚠️ 상시 개발 노트 — 렌더러는 번들이 아니라 **전역 스코프 공유** (반드시 숙지)

> 2026-07-17 세션에서 **실제로 앱을 죽인** 함정이다(v1.22.0). 빌드가 잡아주지 않으므로 노트로 남긴다.

`scripts/build-renderers.js`는 esbuild **`transform`만** 하고 **번들링하지 않는다.** 결과물은 [studio.html](studio.html)에서 각각 `<script>`로 로드되므로 **모든 렌더러(`ui-kit` · `ui-tracks` · `ui-mixer` · `ui-dialogs` · `ui-help` · `app`)가 하나의 전역 스코프를 공유**한다.

- **최상단 `const`/`let` 이름이 겹치면 `Identifier 'X' has already been declared`로 앱 전체가 죽는다.** 빌드는 통과한다. (v1.22.0에서 `EMPTY_CLIP_IDS`를 app.jsx·ui-tracks.jsx 양쪽에 선언해 발생 — 정적 검사로 발견.)
- **최상단 `function`이 겹치면 오류 없이 조용히 덮인다** — 로드가 늦은 쪽이 이긴다. 더 찾기 어렵다. (실제 사례: `buildInputPortOptions`가 ui-tracks·ui-mixer에 중복 정의 → 항상 mixer 버전이 이김. 위 "미처리 발견" 참조.)
- **로드 순서**(studio.html): `ui-kit` → `ui-tracks` → `ui-mixer` → … → `app`. 그래서 app.jsx는 ui-tracks의 전역을 그대로 읽을 수 있다(의도적으로 쓰는 경우 주석으로 명시할 것).
- **새 최상단 심볼을 추가하기 전 이름 충돌을 확인할 것**:
  ```bash
  node -e "const fs=require('fs');const n=f=>[...fs.readFileSync(f,'utf8').matchAll(/^(?:const|function)\s+([A-Za-z0-9_\$]+)/gm)].map(m=>m[1]);const F=['build/app.js','build/ui-tracks.js','build/ui-kit.js','build/ui-mixer.js','build/ui-dialogs.js','build/ui-help.js'];const s={},d=[];for(const f of F)for(const x of n(f)){if(s[x])d.push(x+' ['+s[x]+' vs '+f+']');else s[x]=f}console.log(d.length?d:'없음')"
  ```
- 공용으로 쓸 심볼은 **[ui-kit.jsx](ui-kit.jsx)에 1곳만** 두는 것이 원칙(가장 먼저 로드됨).

## ⚠️ 상시 개발 노트 — Undo 스냅샷 정합성 (반드시 숙지)

> 2026-07-17 세션에서 이 규칙 위반으로 버그 3건(v1.23.2 · v1.23.3)이 나왔다. 새 편집 UI를 붙일 때마다 적용된다.

판정 기준은 하나 — **`getSnapshot()`이 그 상태를 담는가?**

| 상태 | 스냅샷 | `onBeforeChange`/`pushUndo` |
|---|---|---|
| `master`(volume·reverb·echo·fadeIn·fadeOut·room·roomParams·saturation·widener·exciter), `eqBands`, `tempo`, `duration`, 트랙 `params`·`clips`·`sources`·`takes` | **담음** | **필수** — 없으면 그 기능은 Undo 불가(v1.23.3 fade) |
| 플레이헤드, `loopRange`, `repeatPlayEnabled`, `masterFxBypassed` | **안 담음** | **금지** — 100% no-op 항목이 되어 **다음 Ctrl+Z를 먹고 Redo까지 지운다**(v1.23.2 OUTPUT FX / v1.23.3 Repeat 버튼) |

- **드래그 핸들러는 mousedown이 아니라 "첫 실제 변경 직전"에 1회만** 스냅샷(클릭만 했을 때 no-op 항목이 생기지 않게). 값 비교 후 같으면 조기 반환하는 것이 정석 — 각 컴포넌트의 **휠 핸들러가 원래부터 `if (nv === value) return;`으로 이 패턴**을 쓴다. 단 `Fader`·`SleekSlider`는 mousedown에서 클릭 위치로 값을 점프시키므로 그 자리의 스냅샷이 정당하다.
- **편집이 아무것도 바꾸지 않았으면 `cancelUndo(token)`** 으로 되돌린다. `pushUndo()`는 **redo 스택을 비우므로** `undoStack.pop()`만 하면 **Redo가 조용히 죽는다**(v1.23.1).
- **가드**: `undo()`가 복원 전후 `snapshotSig`를 비교해 같으면 `console.warn("[undo] no-op entry ...")`. 정상 동작 시 무출력이므로, **이 경고가 보이면 위 규칙을 어긴 핸들러가 있다는 뜻.**

## [Phase 6] 루프 Take 녹음 — 착수 전 코드 조사 결과 — 2026-07-15

> [개발-프로세스.md](개발-프로세스.md) Phase 6 스펙을 현재 코드와 대조한 결과. **완료 기준을 현 구조로는 충족할 수 없는 항목(F1)이 있어 선행 과제가 필요하다.** 아래 F1~F6은 사실 조사, Q1~Q4는 착수 전 결정 대기 문의.

### F1. 루프 구간 반복이 sample-accurate가 아니다 — **최대 리스크 · 선행 과제** → ✅ **v1.24.0에서 해소(Stage 0 구현 완료)**

> 아래는 **v1.24.0 이전의** 상태 기록이다. 네이티브 경로는 이제 오디오 콜백에서 샘플 단위로 wrap한다(위 v1.24.0 항목). **웹 경로는 Q1 결정대로 rAF 폴링 그대로**이며, 네이티브 연결 시 음소거되므로 요구 사항이 아니다.

현재 Repeat(구간 반복)은 **두 엔진 모두 JS 폴링 + seek 되감기**로 구현돼 있고, **네이티브 엔진에는 loop range 개념 자체가 없다**.

| 경로 | 구현 | 정밀도 |
|---|---|---|
| 웹 | [audio-engine.js](audio-engine.js) `_loop()` — `requestAnimationFrame`으로 `raw >= loopRange.end` 감지 → `_offset = loopRange.start` 후 `_stopSources()` + `play()` **재시작** | rAF ≈ 16 ms + 소스 재시작 지연 |
| 네이티브 | [audio-bridge.js](audio-bridge.js) `startLocalTickLoop()` — `setInterval` **30 ms** 폴링으로 `getPlayhead() >= loopRange.end` 감지 → `Bridge.seek(loopRange.start)`. 추가로 `position` 이벤트 수신 시에도 동일 되감기 | 30 ms 폴링 + WebSocket seek 왕복 |
| 네이티브 C++ | `setLoop`/`isLooping`은 **곡 전체 반복**(`readerSource->setLooping`)뿐. `grep`상 `loopRange`/`loopStart`/`loopEnd` **부재** | 구간 반복 미지원 |

- 게다가 네이티브 playhead는 [audio-bridge.js](audio-bridge.js) `getPlayhead()`에서 **`Date.now()` 벽시계 외삽**으로 추정한다(`nativeState.offset + elapsed * rate`).
- **영향**: Phase 6 완료 기준 3개 — "각 Take 길이가 루프 sample 수와 일치", "반복 녹음이 길어져도 누적 밀림이 없다", "루프 경계가 오디오 블록 중간에 있어도 Take가 정확히 나뉜다" — 를 **현 구조로는 원리적으로 충족 불가**. 매 회차 길이가 폴링 지터(0~30 ms+)만큼 달라지고 회차마다 밀림이 누적된다.
- **결론**: Phase 6는 **"네이티브 엔진에 sample-accurate loop range 도입"을 Stage 0 선행 과제**로 두어야 한다. 오디오 콜백 안에서 루프 끝을 샘플 단위로 감지해 블록을 경계에서 분할·되감기하고, JS 폴링 되감기는 native 연결 시 비활성화한다. 이렇게 하면 **같은 콜백에서 녹음 버퍼도 경계 분할**할 수 있어 Take 분할이 자연히 sample-accurate가 된다. (작업원칙 6 "웹/네이티브 결과 일치"와 Phase 6 완료 기준의 관계는 Q1 참조.)

### F2. `attachRecording`이 source/clip을 통째로 교체한다 — Take 누적 불가

- [audio-engine.js](audio-engine.js) `attachRecording`은 `track.sources = [{...}]`, `track.clips = [...]`를 **배열째 재대입**한다(1 녹음 = 1 source + 1 clip). 기존 Take가 있으면 사라진다.
- Phase 6은 회차별 `Take A/B/C` **누적**이 필요 → `attachRecording`을 **append 기반**으로 재구성하고 `track.activeTakeId`(또는 clip별 `takeId`) 모델을 도입해야 한다.
- 부수: `attachRecording`은 `track.buffer = decoded.buffer`를 raw로 넣고 bake하지 않는다. Take 도입 시 `_ensureBaked` 경로와 함께 정리 필요(상시 노트 참조).

> ✅ **선행 정리 완료(v1.26.1, 2026-07-17)** — Stage 3이 부딪힐 **구조적 장벽은 제거됐다.** ① `_captureRawBuffers`가 `if (_rawBuffers) return;`이라 **트랙당 source 1개만 등록 가능**하던 것을 가산적으로 고쳤다(이게 F2의 진짜 벽 — Take B의 raw가 등록 안 돼 **Take B 클립에 Take A 오디오가 구워질** 자리였다). ② `_registerSource(track, meta, rawBuffer)` 신설(sources 추가 + raw 등록을 한 번에). ③ `_rawBufferForSource`가 미등록 source에 `console.warn`(조용한 오음 대신 로그). ④ 조사 중 발견한 **실제 버그 수정** — 재녹음이 `_rawBuffers`/`_layoutBaked`를 안 지워 "녹음 → 이동 → 재녹음 → 편집"에서 **테이크가 사라지던** 문제. 상세 이 문서 v1.26.1.
>
> **남은 F2 본체는 Stage 3 몫**: `track.clips = [새 클립]` 교체를 **append + `takeId`/`activeTakeId`**로 바꾸는 일. 지금 append만 켜면 두 번째 테이크가 겹침 규칙에 밀려 뒤로 이동하므로 **의도적으로 교체를 유지**했다. `clip.takeId`·`track.takes[]`는 **이미 정규화·저장·스냅샷에 배선돼 있고 값만 비어 있다**.

### F3. Take Lane과 전략 B 베이크 모델의 정합 문제

- 전략 B는 **트랙당 평탄화 버퍼 1개**를 전제한다. 반면 Take는 **같은 시간대에 여러 대체 후보**가 존재한다 → 그대로 bake하면 Take들이 서로 겹쳐 섞인다.
- 필요한 규칙: **활성 Take의 clip만 bake 대상**, 비활성 lane clip은 데이터로만 보존하고 bake·재생·Export에서 제외. Phase 5의 "같은 트랙 내 clip 겹침 금지"는 **활성 레인 내부에만** 적용되도록 판정 범위를 좁혀야 한다(비활성 Take는 겹쳐도 정상).
- 이 규칙이 서면 Phase 7 Comp(활성 Take 선택 + 구간별 조합)도 같은 모델 위에 얹힌다.

### F4. count-in은 이미 있다(무음·박자 무관), 없는 것은 **클릭 소리**다 — 2026-07-15 정정

> ⚠️ 최초 조사에서 "count-in 자체가 없다"고 잘못 적었다. 실제로는 **3→2→1 count-in이 이미 구현·문서화·시험까지 되어 있다.** 아래가 정확한 현황.

- **기존 count-in**([app.jsx](app.jsx) `beginRecordFlow`): 정지 상태에서 Record → 화면 중앙 **3→2→1 시각 오버레이** → 끝나면 `play`+`record` 동시 시작. 재생 중 Record는 count-in 없이 즉시 녹음. count-in 중 Record/Stop/Play 누르면 `cancelCountIn()`으로 취소.
  - 관련 상태: `recordCountRef`(interval id), `recordCount`(오버레이 숫자), `window.__recordCountdownActive`(ARM 잠금 등이 참조), `prevLoopRef`(count-in 중 Repeat off → 취소 시 복원).
  - 문서·시험 존재: [ui-help.jsx](ui-help.jsx) 매뉴얼에 "3 → 2 → 1 count-in" 명시 + 스크린샷 `manual/live-screens/49-record-countin.png`, v1.20.11 시험(카운트인 중 ARM 잠금)에도 등장.
- **없는 것 = 클릭 소리**: 앱 코드(`*.js`/`*.jsx`/`juce_skeleton/Source`)에 metronome/click 구현 **부재**(검색 결과는 전부 `third_party/JUCE`의 AAX SDK 헤더). 기존 count-in은 **완전 무음**이다.
- **한계**: `setInterval(..., 1000)`으로 도는 **1초 간격 3회 고정** → **BPM/박자와 무관**하다. 연주자가 진입 타이밍을 귀로 잡을 수 없다.
- → Q2는 "count-in 신설"이 아니라 **"기존 count-in에 클릭 소리 + 박자 개념을 얹는 확장"**이다.

### F5. 녹음은 네이티브 전용이다 (제약이자 기회)

- 웹 엔진에 `getUserMedia`/`MediaRecorder` **없음**. 녹음은 JUCE `InputRecorder`(writer thread + `ThreadedWriter` FIFO) 전용.
- 즉 **루프 Take 녹음은 native 연결이 전제**이고, 웹 fallback에서는 불가(현재 단일 녹음과 동일한 제약).
- 뒤집어 보면 native가 **recorder이자 transport**이므로, F1을 네이티브에서 해결하면 루프 경계 분할·Take 생성이 전부 한 콜백 안에서 sample-accurate하게 처리된다. 이게 F1을 선행 과제로 두는 근거이기도 하다.

### F7. 웹 엔진의 실제 역할 — "웹 버전"이 아니라 앱 내부 계층 (2026-07-15 확인)

> 배포 방침(네이티브 Electron 기준, 웹 배포판 없음) 확정에 따라 "웹 엔진을 걷어낼 수 있는가"를 조사한 결과. **결론: 출력 경로로서는 이미 포기된 상태이나, 코드는 제거 대상이 아니다.**

- **출력으로서의 웹은 이미 비활성**: 네이티브 연결·핸드오버 완료 시 [audio-bridge.js](audio-bridge.js) `activateNativeOutput()`이 `LocalDAW.setOutputMuted(true)`로 음소거한다. 들리는 소리는 전부 네이티브다. → Q1 결정은 이 현실 위에 서 있다.
- **그러나 `audio-engine.js`는 필수 계층**:
  1. **모델의 진실원본** — `window.DAW`는 `DAWProxy`이고, Bridge에 없는 모든 속성/메서드를 `LocalDAW`로 위임한다. UI 전체(`DAW.tracks`/`duration`/`loopRange`)가 여기 의존.
  2. **파형 소스** — `track.buffer`/`peaks`/`sources`/`clips`가 전부 웹 엔진에 있다. 화면 파형 = 웹 엔진 디코드 버퍼.
  3. **Phase 5 baker** — `_bakeTrackLayout`이 웹 엔진에서 Float32로 bake → 브리지가 temp WAV로 native에 `loadTrack`. **네이티브는 bake를 못 한다**(평탄화 결과 재생만).
  4. **Export fallback** — `Bridge.renderMix`는 `options.forceLocal || !this.isNative`면 `LocalDAW.renderMix`로 떨어진다. 배포본 정상 경로는 native `command: "export"`.
  5. **안전망** — `switchToLocal()`(`socket.onclose`/재시도 소진)이 웹 mute를 풀어 소리를 유지한다.
- **⚠️ `switchToLocal()` 안전망은 배포본에서도 유지**한다. 네이티브가 유일 출력인 만큼 없으면 엔진 크래시 = **무음 앱**. 코드가 이미 있어 유지 비용 0.
- 반영: [개발-프로세스.md](개발-프로세스.md) 작업원칙 6을 위 내용으로 재서술(2026-07-15).

### F6. 입력 지연 보정값이 드라이버 추정치다

- v1.20.7의 latency 표시는 WASAPI **드라이버 보고 기반 근사치**(문구에도 `driver estimate` 명시).
- Phase 6은 "입력 지연 보정값을 clip `start`에 반영"을 요구 → 근사치를 그대로 쓰면 Take마다 수 ms 어긋날 수 있다. 정확도 목표 결정 필요(Q4).

### 🐛 Consolidate source가 저장/재열기에서 소멸 — v1.27.1 경고가 노출 (2026-07-18)

**증상**: 편집 중 콘솔에 `[rawBuffer] source not registered — substituting primary audio`. **오탐이 아니다** — 실제로 어떤 clip이 primary 오디오로 대체되고 있다.

**원인**(코드 확정): `_consolidateClips`([audio-engine.js](audio-engine.js))는 **떨어진 클립을 Join**할 때 두 오디오를 합친 **새 source를 `filePath: null`(메모리 전용 raw 버퍼)로** 만든다(v1.21.7, "Join 후 Trim 시 소멸" 수정의 부산물). 그런데 `importProject`([audio-engine.js](audio-engine.js))는 **primary source의 `filePath`에서만** 오디오를 재로드한다 — `filePath: null`인 consolidated source는 재열기 시 raw 버퍼를 복원할 수단이 없다. 그래서 그 source를 지목한 clip이 `_rawBufferForSource`에서 primary로 폴백 → **엉뚱한 오디오**.

**재현**: 떨어진 두 클립을 Join(Consolidate 발생) → 저장 → 재열기 → 합쳐진 클립이 primary 오디오를 낸다. (라이브 세션에서는 `_rawBuffers`가 살아 있어 정상 — **저장/재열기 또는 네이티브 재연결에서만** 터진다. `registered: Array(1)`이 그 신호.)

**기존 버그다** — v1.26.1 이전에도 대체는 일어났고 **조용했을** 뿐이다. v1.27.1이 경고 범위를 "실제 대체가 일어나는 경우"로 좁히면서 처음 드러났다.

**제대로 된 수정 = Phase 7 "Consolidate Clips: 선택 clip을 새 WAV로 렌더"**([개발-프로세스.md](개발-프로세스.md)). consolidate 시 합친 오디오를 프로젝트 오디오 폴더(Recordings/Bounces)에 **WAV로 써서 `filePath`를 부여**하면 재열기에서 살아남는다. 다만 이는 **Save As 파일 복사 규칙·미저장 프로젝트 임시 스테이징**과 얽혀 Phase 7 범위다. **우선순위·착수 시점은 사용자 결정 대기.** 그 전까지는 "떨어진 클립 Join은 저장 전에만 신뢰"가 현재 한계.
