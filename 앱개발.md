## v1.37.3 — Join과 Consolidate를 `Merge Clips` 하나로 통합 (사용자 결정) (2026-07-20)

> **재빌드: renderer([app.jsx](app.jsx)·[ui-tracks.jsx](ui-tracks.jsx) esbuild) + [audio-engine.js](audio-engine.js)(플레인 로드).** **네이티브 무변경.** 사용자 결정(2026-07-20): "사용자 입장에서 Join과 Consolidate가 혼동된다" → 하나로 통합.

### 문제 — 같은 일을 두 이름으로

사용자에게 보이는 결과가 **동일**했다(N개 클립 → 1개). 진짜 차이는 "새 WAV를 쓰는가"인데 그건 내부 구현이고, 게다가 **Join은 소스 비연속이면 조용히 consolidate로 폴백**했다. 즉 이름은 Join인데 하는 일은 Consolidate. 구분할 수 없는 것을 두 명령으로 노출하고 있었다.

### 통합

- **`Merge Clips` 하나**로 노출. 2개 이상 선택 시 메뉴에서 **즉시 병합**(`Merge N clips`), 선택이 없으면 **병합 도구**를 켜서 클릭으로 이웃과 병합(`J` 유지). 두 경로 모두 `handleConsolidateClips` 하나를 타므로 **거부 메시지·Undo·WAV 기록이 동일**.
- 전략은 **엔진이 판단**: 신규 **`_isHealableRun(clips)`** 이 "같은 source + 타임라인 gap 없음 + sourceOffset 연속"(=split 이후 아무도 안 움직인 상태)을 판정해 **heal**(단일 연속 소스 읽기, 파일 생성·재렌더 없음) / **render**(새 WAV) 중 하나를 고른다. `canConsolidateClips`가 `strategy`로 함께 반환.
- **N개로 일반화** — 3조각으로 split한 것도 한 번에 heal. 기존 Join은 2개씩만 가능했다.
- **`joinClips`는 2줄 위임**으로 축소(병합 도구 진입점으로만 유지, 이름은 내부 API·bridge 호환 위해 보존). 2개 전용 헬퍼 **`_consolidateClips`는 삭제**(사용처 소멸).

### 동반 수정 — Undo 스냅샷 정합성 위반

기존 `handleJoin`이 **거부당하는 경우에도 `pushUndo()`를 먼저** 호출해 no-op 스냅샷을 쌓고 있었다(레인·between 가드 도입으로 거부 경로가 늘며 실제 문제가 됨). "Undo 스냅샷 정합성" 노트 위반 — **다음 Ctrl+Z를 먹고 Redo까지 지운다**. 이제 `canConsolidateClips`로 **먼저 검사하고 통과할 때만** 스냅샷을 뜬다.

### 검증

esbuild OK, `node --check` OK, 전역 심볼 충돌 신규 0건, Electron 부팅 렌더러 에러 0. 전략 판정 **시뮬레이션 6검사 통과**(2조각·3조각 split→heal / 이동·트림·복사붙여넣기→render / 연속이어도 base+take면 거부가 우선). 사용자 시험 T-1.37.0-1·2·3 **전부 통과**(2026-07-20) → [시험-아카이브.md](시험-아카이브.md).

## v1.37.2 — 선택 구간을 가로지르는 클립이 있으면 병합 거부 (사용자 보고) (2026-07-20)

> **재빌드: renderer([app.jsx](app.jsx) esbuild) + [audio-engine.js](audio-engine.js)(플레인 로드).** **네이티브 무변경.** 사용자 보고("3개 중 1번과 3번을 consolidate하면 중간 clip이 그대로 남아 있는데 정상인가?").

### 원인 — 정상 아님. 중간 클립이 **밀려나고 있었다**

병합 클립은 `first.start ~ last.end` 전체를 차지하므로 중간 클립의 자리를 덮는다. 그러면 [`_reindexClips`](audio-engine.js)의 "트랙 내 겹침 금지" 불변식이 그 클립을 **오른쪽으로 밀어낸다** — 화면엔 남아 보이지만 **원래 시간 위치를 잃은** 상태이고, 병합 결과 안에서 그 구간은 무음으로 렌더된다. `joinClips`에는 이미 같은 형태를 막는 `between` 가드가 있었는데 consolidate에 넣지 않은 누락.

### 수정 — 판정 분리 + 사유별 안내

- 신규 **`canConsolidateClips(trackId, clipIds)`** 이 `{ok, reason}`을 반환(`few`/`lane`/`between`/`track`). 구간을 가로지르는 **활성** 클립이 있으면 `between`으로 거부(비활성 take 클립·0길이 placeholder는 렌더되지 않으므로 막지 않는다).
- [app.jsx](app.jsx)가 사유별 안내 메시지를 띄운다 — 종전 "same Take lane..." 한 줄로는 무엇이 문제인지 알 수 없었다(T-1.37.0-3에서 사용자가 실제로 확인 요청). `between`은 "중간 클립을 Ctrl+클릭해 포함하거나 인접한 것끼리 나눠서 합치라"고 다음 행동을 알려준다.

### 검증

거부 규칙 **시뮬레이션 5케이스 11검사 통과**(1번+3번→between / 셋 다 선택→ok / 인접 쌍→ok / 비활성 take 무시 / 0길이 placeholder 무시).

## v1.37.1 — 저장 → 재열기에서 병합 결과가 사라지던 버그 2건 수정 (사용자 시험 T-1.37.0-1 실패) (2026-07-20)

> **재빌드: [audio-engine.js](audio-engine.js)(플레인 로드)·[version.js](version.js).** **renderer·네이티브 무변경.** 사용자 시험 실패 보고 → 사용자 저장 파일 `untitled_20260720_3.focus`를 직접 열어 **원인 확정**.

### 증거

저장된 프로젝트에서 consolidate 소스의 **`filePath`가 `null`** 이었다 — `Consolidated/` 폴더에는 WAV가 정상 기록돼 있는데도.

```
bounce 트랙  src_5s4bnul7  dur=342.54  filePath = null
Audio In    src_7tp0wyg0  dur=97.22   filePath = null
```

### ① 비동기 경합 — `filePath`가 버려진 객체에 기록됨

`persistConsolidatedSources`가 **`await`(WAV 인코딩+IPC) 전에** source 객체를 잡아두고, 완료 후 그 객체에 `filePath`를 썼다. 그런데 **`_normalizeTrackLayout`은 `track.sources`를 새 복사본 배열로 교체**한다(`[primary, ...slice(1).map(s => ({...s}))]`) — 그리고 이건 serialize/snapshot/import마다 실행되며 **autosave가 1.5초마다** 유발한다. 그 사이에 교체가 일어나면 기록이 **고아 객체**에 남고, 프로젝트에는 `filePath: null`이 저장된다 → 재열기 시 `reconnectProjectAudio`의 hydrate 대상(`s.needsAudio && s.filePath`)에서 빠짐 → raw 없음 → 베이크 스킵 → **파형 없음**(사용자 보고한 Audio In 증상).
**수정**: `await` **이후에 source를 id로 재조회**해서 기록한다.

### ② 재접속 휴리스틱이 병합 클립을 되돌림

[`_assignDecodedToTrack`](audio-engine.js)의 "클립 1개 + `start === 0`이면 파일 전체"라는 판정이, **병합 결과가 정확히 그 모양**이라 직격했다. 그 클립을 **primary 소스로 다시 가리키게 하고 원본 길이로 늘려** 재열기 시 병합 이전 상태로 보였다(사용자 의심대로 **bounce 원본을 불러온 것이 맞다**).
**수정**: 그 클립이 **실제로 primary 소스에 속할 때만** 적용(`!c.sourceId || c.sourceId === primaryId`).

### ③ 동반 — Join이 base 클립을 Take에 흡수

`joinClips`가 `first.takeId || second.takeId`로 태그를 정해, **base 클립(`takeId=null`, 항상 렌더)을 Take 안으로 흡수**했다. Loop-Punch Comp 구간 경계를 걸쳐 join하면 흡수된 base 오디오가 **Take 전환 시 사라진다**. Consolidate는 거부하는데 Join은 조용히 수행하던 **불일치**(사용자 질문으로 발견). Join에도 같은 레인 가드를 적용 — ⚠️ **기존 동작 변경**.

### 한계

**v1.37.1 이전에 병합해 이미 저장한** 클립은 `filePath`가 null로 굳어 소급 복구 불가 — 다시 병합해야 한다(원본 WAV는 `Consolidated/`에 남아 있다).

## v1.37.0 — Phase 7 Stage 2: Consolidate Clips (+ Consolidate source 소멸 버그 근본 수정) (2026-07-20)

> **재빌드: renderer([app.jsx](app.jsx)·[ui-tracks.jsx](ui-tracks.jsx) esbuild) + [audio-engine.js](audio-engine.js)·[audio-bridge.js](audio-bridge.js)(플레인 로드) + [electron/main.js](electron/main.js)·[electron/preload.js](electron/preload.js).** **네이티브 무변경**(기존 `loadTrack` 베이크 sync 재사용). 사용자 승인(2026-07-20). ⚠️ 이 버전의 초회 시험에서 **저장/재열기 실패 2건**이 나와 v1.37.1에서 수정했고, 명령 자체는 v1.37.3에서 `Merge Clips`로 통합됐다 — 최종 동작은 위 v1.37.1~3 항목을 함께 볼 것.

### 개요 — "새 WAV로 렌더"가 곧 버그 수정

Phase 7 계획의 `Consolidate Clips`(선택 clip을 새 WAV로 렌더)를 구현하면서, **이월돼 있던 "Consolidate source가 저장/재열기에서 소멸" 버그를 같은 변경으로 해소**했다. 원인은 `_consolidateClips`가 만든 소스가 `filePath: null`(메모리 raw)인데 재열기 복원 경로([app.jsx](app.jsx) `reconnectProjectAudio`)가 **`s.needsAudio && s.filePath`인 소스만** `hydrateSource`로 재로드하는 것 — 경로 없는 소스는 복원 수단이 아예 없어 clip이 primary 오디오로 폴백했다. **소스에 실제 WAV 경로만 생기면 기존 복원 경로가 그대로 살려내므로 재열기 로직은 무변경.**

### ① 디스크 기록 ([electron/main.js](electron/main.js)·[electron/preload.js](electron/preload.js))

- 신규 IPC **`save-consolidated-audio`**(`save-bounce-audio` 패턴) → **`<Project> Audio/Consolidated/`**. Bounce(트랙 합치기 결과)와 Consolidate(클립 편집 확정 결과)는 성격이 달라 폴더를 분리했다(사용자 결정 2026-07-20). 정리 도구가 폴더 단위로 판정하기도 쉽다.
- **미저장 프로젝트 폴백**: 프로젝트 경로가 없으면 첫 트랙 소스 폴더, 그것도 없으면 **temp**(`FocusDAW Consolidated`)에 기록. 경로 없는 소스는 재열기에서 소멸하므로 "저장 전이라 못 쓴다"가 곧 버그 재발이다 — 반드시 어딘가에는 기록한다. temp→프로젝트 폴더 이동은 Phase 7 "Save As 복사 규칙"에서 처리.

### ② 엔진 — N-클립 렌더 + persist 큐 ([audio-engine.js](audio-engine.js))

- `_consolidateClips`(2개 전용)를 **`_renderClipsToSource(track, clips, gaps)`(N개 일반화)** 로 교체하고 기존 함수는 위임만 한다 → **`joinClips` 시그니처·동작 무변경**(비연속 join도 그대로 consolidate 경로).
  - 타임라인 gap은 무음으로 기록해 각 clip이 제자리를 지키고, **이음매마다** micro-fade(양쪽), **바깥 가장자리는 페이드 없음**. `n`(raw가 실제 보유한 길이)과 `dur`(clip 선언 길이)를 분리해 **raw가 짧아도 뒤 clip이 앞으로 밀리지 않는다**.
- 새 소스는 `_pendingConsolidations[]`에 큐잉. 신규 **`async persistConsolidatedSources(saveFn)`** 이 큐를 비우며 `saveFn`으로 WAV를 쓰고 `source.filePath`/`fileName`을 채운다. **best-effort** — 실패하면 메모리 소스로 남아 **v1.37.0 이전과 동일**, 편집 자체는 성립(디스크 문제로 편집이 실패하지 않는다).
- 신규 **`consolidateClips(trackId, clipIds)`**: 선택 clip을 하나로 평탄화. **같은 레인만 허용**. raw 누락 시 `null` 반환하고 **clip을 건드리지 않는다**.
  - ⚠️ 이 가드가 실제로 막는 것은 **"Take A + Take B"가 아니다** — Take Lane 행은 **표시 전용**(클릭=활성화, 선택 가능한 clip rect 없음)이고 메인 레인은 `_activeClips`만 렌더하므로, **서로 다른 Take의 clip을 함께 선택하는 경로 자체가 없다**. 실제 대상은 메인 레인에 나란히 보이는 **base clip(`takeId=null`, 항상 렌더) + 활성 Take clip**의 혼합이며, 이는 정확히 **Loop-Punch Comp(v1.36.0)의 레이아웃**(구간 밖=base, 구간 안=pass별 take)이다.
  - 이 조합을 합치면 어느 쪽으로 태깅해도 깨진다: 결과를 `null`로 두면 그 Take 오디오가 **모든 Take에서 겹쳐** 재생되고, Take id를 붙이면 **base 오디오가 Take 전환 시 사라진다**. 그래서 틀린 답을 고르는 대신 거부한다.

### ③ 브리지·UI

- [audio-bridge.js](audio-bridge.js) `consolidateClips` 래퍼가 재베이크된 레이아웃을 native로 sync(joinClips와 동일 패턴). `persistConsolidatedSources`는 `DAWProxy` 위임으로 도달.
- [ui-tracks.jsx](ui-tracks.jsx) 클립 우클릭 메뉴에 **Consolidate**(Split/Join과 같은 그룹). Split/Join과 달리 **즉시 실행**이며 **선택 전체**에 작용 — 2개 미만이면 비활성 + 안내 title(`ClipContextMenu`에 `title` 지원 추가).

- 그 **위에 "PUNCH" 텍스트**(버튼 아님, `pointerEvents:none`)를 추가: **punch 모드 ON = 진하게(앰버·bold·glow), OFF = 흐리게**(`--faint`, opacity .5). `focusdaw-punch`를 매 렌더 읽어 라이브 반영.
- 결과 동작(사용자 명세대로): ① PUNCH 클릭 불가(=Repeat ON으로 안 바뀜) ② 구간 선택 시 Repeat 버튼 위 PUNCH 글자 ③ Repeat 버튼 누르면 Repeat ON ④ 그 상태로 Play=repeat play ⑤ PUNCH 글자는 punch 켜짐=진하게/꺼짐=흐리게.

### 로직 확인 (사용자 문의)

Punch(**녹음** 동작)와 Repeat-play(**재생** 동작)는 **독립**이라 충돌 없음. 매트릭스: **Punch ON → Record=punch**(repeat 상태 무관, 녹음 중에만 repeat-play 잠시 off 후 복원) · **Punch OFF+Repeat ON → Record=루프 Take** · 둘 다 OFF → 일반 녹음. ⚠️ 단, **Repeat+Punch 동시 켜짐에서 Record는 단일 punch**이며 "여러 pass 중 선택(loop-punch comp)"은 **Phase 7**.

## ⚠️ 상시 개발 노트 — 버퍼 인덱싱 이원성 (전략 B, 반드시 숙지)

> Phase 5에서 도입된 **레이아웃 베이크(전략 B)**의 결과로 생긴 **상시 제약**이다. 완료 기록은 아카이브에 있으나, 이 노트는 이후 모든 파형/재생/피크 코드에 계속 적용되므로 여기에 남긴다.

전략 B에서 `track.buffer`는 두 가지 인덱싱 체계 중 하나이며, **`track._layoutBaked` 플래그가 이를 구분한다.** 이 둘을 혼동하면 **이동한 클립이 빈 파형으로 그려지는 버그**가 재발한다(2026-07-14 실제 발생, 수정 완료).

| 상태 | 버퍼 | 오디오 위치 | 파형 샘플링 기준 | 재생 |
|---|---|---|---|---|
| `_layoutBaked = true` (baked) | 평탄화 렌더 결과 | **타임라인 위치**(`clip.start`) | `clip.start` 기준(타임라인) | `_offset`(플레이헤드)로 그냥 재생 |
| `_layoutBaked = false` (raw/trivial) | 원본 디코드 버퍼 | 소스 원점 | `clip.offset` 기준(소스) | 동일(정렬돼 있어 무관) |

- `_ensureBaked`가 bake하면 `_layoutBaked=true`, trivial로 raw 복원하면 `false`로 설정한다.
- [ui-tracks.jsx](ui-tracks.jsx) `Waveform.bucketAt`는 `track._layoutBaked`로 분기해 `bufPos`를 **타임라인(clip.start) vs 소스(clip.offset)** 로 계산한다. peaks 배열도 `_ensureBaked`가 baked 버퍼에서 재계산하므로 타임라인 인덱싱이다.
- **새 파형/재생/피크 코드를 추가할 때**: baked 버퍼를 `clip.offset`로 접근하면 안 된다(그 위치는 침묵). 반드시 `_layoutBaked`를 확인하고 baked면 타임라인 위치로 접근.
- 재접속(`_assignDecodedToTrack`)은 raw 디코드 후 non-trivial이면 `_ensureBaked`를 호출해 재시작 후에도 이 불변식을 복원한다.
- 관련: `_rawBuffers[sourceId]`(원본 보관), `_isTrivialLayout`(bake 스킵 판정), `clip.gain`은 baked 버퍼에 반영·`clip.params.volume`/automation은 합성커브(이중적용 방지).

## ⚠️ 상시 개발 노트 — 렌더러는 번들이 아니라 **전역 스코프 공유** (반드시 숙지)

> 2026-07-17 세션에서 **실제로 앱을 죽인** 함정이다(v1.22.0). 빌드가 잡아주지 않으므로 노트로 남긴다.

`scripts/build-renderers.js`는 esbuild **`transform`만** 하고 **번들링하지 않는다.** 결과물은 [studio.html](studio.html)에서 각각 `<script>`로 로드되므로 **모든 렌더러(`ui-kit` · `ui-tracks` · `ui-mixer` · `ui-dialogs` · `ui-help` · `app`)가 하나의 전역 스코프를 공유**한다.

- **최상단 `const`/`let` 이름이 겹치면 `Identifier 'X' has already been declared`로 앱 전체가 죽는다.** 빌드는 통과한다. (v1.22.0에서 `EMPTY_CLIP_IDS`를 app.jsx·ui-tracks.jsx 양쪽에 선언해 발생 — 정적 검사로 발견.)
- **최상단 `function`이 겹치면 오류 없이 조용히 덮인다** — 로드가 늦은 쪽이 이긴다. 더 찾기 어렵다. (실제 사례: `buildInputPortOptions`가 ui-tracks·ui-mixer에 중복 정의 → 항상 mixer 버전이 이김. 위 "미처리 발견" 참조.)
- **로드 순서**(studio.html): `ui-kit` → `ui-tracks` → `ui-mixer` → … → `app`. 그래서 app.jsx는 ui-tracks의 전역을 그대로 읽을 수 있다(의도적으로 쓰는 경우 주석으로 명시할 것).
- **새 최상단 심볼을 추가하기 전 이름 충돌을 확인할 것**:
  ```bash
  node -e "const fs=require('fs');const n=f=>[...fs.readFileSync(f,'utf8').matchAll(/^(?:const|function)\s+([A-Za-z0-9_\$]+)/gm)].map(m=>m[1]);const F=['build/app.js','build/ui-tracks.js','build/ui-kit.js','build/ui-mixer.js','build/ui-dialogs.js','build/ui-help.js'];const s={},d=[];for(const f of F)for(const x of n(f)){if(s[x])d.push(x+' ['+s[x]+' vs '+f+']');else s[x]=f}console.log(d.length?d:'없음')"
  ```
- 공용으로 쓸 심볼은 **[ui-kit.jsx](ui-kit.jsx)에 1곳만** 두는 것이 원칙(가장 먼저 로드됨).

## ⚠️ 상시 개발 노트 — Undo 스냅샷 정합성 (반드시 숙지)

> 2026-07-17 세션에서 이 규칙 위반으로 버그 3건(v1.23.2 · v1.23.3)이 나왔다. 새 편집 UI를 붙일 때마다 적용된다.

판정 기준은 하나 — **`getSnapshot()`이 그 상태를 담는가?**

| 상태 | 스냅샷 | `onBeforeChange`/`pushUndo` |
|---|---|---|
| `master`(volume·reverb·echo·fadeIn·fadeOut·room·roomParams·saturation·widener·exciter), `eqBands`, `tempo`, `duration`, 트랙 `params`·`clips`·`sources`·`takes` | **담음** | **필수** — 없으면 그 기능은 Undo 불가(v1.23.3 fade) |
| 플레이헤드, `loopRange`, `repeatPlayEnabled`, `masterFxBypassed` | **안 담음** | **금지** — 100% no-op 항목이 되어 **다음 Ctrl+Z를 먹고 Redo까지 지운다**(v1.23.2 OUTPUT FX / v1.23.3 Repeat 버튼) |

- **드래그 핸들러는 mousedown이 아니라 "첫 실제 변경 직전"에 1회만** 스냅샷(클릭만 했을 때 no-op 항목이 생기지 않게). 값 비교 후 같으면 조기 반환하는 것이 정석 — 각 컴포넌트의 **휠 핸들러가 원래부터 `if (nv === value) return;`으로 이 패턴**을 쓴다. 단 `Fader`·`SleekSlider`는 mousedown에서 클릭 위치로 값을 점프시키므로 그 자리의 스냅샷이 정당하다.
- **편집이 아무것도 바꾸지 않았으면 `cancelUndo(token)`** 으로 되돌린다. `pushUndo()`는 **redo 스택을 비우므로** `undoStack.pop()`만 하면 **Redo가 조용히 죽는다**(v1.23.1).
- **가드**: `undo()`가 복원 전후 `snapshotSig`를 비교해 같으면 `console.warn("[undo] no-op entry ...")`. 정상 동작 시 무출력이므로, **이 경고가 보이면 위 규칙을 어긴 핸들러가 있다는 뜻.**

- ✅ **Consolidate(떨어진 클립 Join) source가 저장/재열기에서 소멸 — v1.37.0~v1.37.1에서 근본 수정, T-1.37.0-1 통과**(2026-07-20). 병합 소스가 `filePath:null`(메모리 raw)이라 재열기 복원 대상에서 빠지던 것을, 생성 즉시 `<Project> Audio/Consolidated/`에 WAV로 기록하고 `source.filePath`를 채워 **기존 `hydrateSource` 복원 경로에 태우는** 방식으로 해소(v1.37.0). 초회 시험에서 **경로가 실제로는 저장되지 않는 비동기 경합**과 **재접속 휴리스틱이 병합 클립을 되돌리는** 두 결함이 드러나 v1.37.1에서 마저 수정했다.
  - ⚠️ 남은 한계 ①: **v1.37.1 이전에 병합해 이미 저장한** clip은 `filePath`가 null로 굳어 소급 복구 불가 — 다시 병합해야 한다(렌더된 WAV 자체는 `Consolidated/`에 남아 있다).
  - ⚠️ 남은 한계 ②: 디스크 기록이 실패하면 종전처럼 메모리 전용 소스로 폴백하므로, 그 세션 한정으로 같은 증상이 날 수 있다(설계상 의도 — 디스크 문제로 편집 자체가 실패하지는 않는다).
- `clip.gain`은 baked WAV로 native에 정상 반영됨(문제 아님).

### 사용자 결정으로 종결한 항목 (구현 안 함)

- **clip 단위 volume / automation UI** → **전체 트랙 automation으로 대체(사용자 결정 2026-07-20)**. clip 단위 조작 UI는 신설하지 않는다. 엔진의 `setClipParam`/`clip.params`는 미사용 상태로 유지(호출 UI 없음), `clip.gain`만 baked로 반영. 이에 따라 **clip-automation의 native 미표현 문제도 함께 무효**(발동 경로 없음).
- **micro fade 시각화** → **불필요(사용자 결정 2026-07-20)**. 엔진 bake에 3~5ms로 이미 반영되어 소리에는 나타나므로 UI 표시는 하지 않는다.
- **Input source 선택(mic/line/Hi-Z)** → Skip(WASAPI 한계, ASIO 필요 — 사용자 협의).

### 향후 개발 후보

**우선순위 높음**
- **성능**: 대용량/다트랙 peak 캐시·Web Worker 디코딩·automation 곡선 memoize·스크롤/줌. 전략 B bake가 편집마다 트랙 전체를 재렌더하므로 긴 트랙 편집 반응 속도 계측 필요. 장시간 세션 메모리·Export 캐시 무효화 정책 점검.
- **장시간 녹음 실측(30분+)**: 설계상 충족(ThreadedWriter+FIFO)하나 실측 기록 없음 — 필요 시 스트레스 시험.

**중간**
- **트랙 재정렬**: 트랙 헤더 드래그로 순서 변경(클립 드래그 이동은 Phase 5 완료).
- **자동 업데이트 + CI**: `electron-updater` + GitHub Releases, `electron-builder --publish`.
- **리버브 IR 개선**: 절차적 노이즈 IR → 실측 IR 로딩/프리셋.

**검토**
- **EQ 밴드 표현 통일**: OUTPUT FX 3밴드 vs Mixer Master/Advanced EQ 9밴드.
- **VST3 / 고급 미터링**(True Peak, LUFS) 장기 조사.
- **Cloud Sync**(.focus + stem을 클라우드 동기화) 장기.
- **Auto-tune(피아노 롤 피치 편집)**: 대형 신규 모듈, 착수 전 결정 필요(모노포닉 범위·피치 엔진·오프라인 렌더). 상세 아카이브 v1.21.0 Q5.
