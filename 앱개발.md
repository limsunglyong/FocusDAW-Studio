# FocusDAW — Stem Studio · 앱 개발 문서 (v1.0.0)

> 웹브라우저 기반(Electron) Stem 믹싱 DAW  
> 하나의 곡에서 분리된 Stem을 불러와 구간별로 볼륨·이펙트를 적용하고, 하나의 MP3로 최종 출력하는 경량 DAW.

## 2. 새로운 앱의 목표

v1.0.0 단계의 핵심 목표는 오디오 트랙별 **시간(Tempo/BPM) 및 주파수(Pitch) 제어 기능**을 고도화하여, Stem 믹싱 중 템포 매칭과 조바꿈(Key Change) 편집을 지원하는 것입니다.

### 2.1 BPM 측정 기능 (BPM Detection) — ✅ 구현 완료 (v1.2.6, TAP 개선 v1.2.9)
- 임포트된 개별 오디오 파일(Stem)의 BPM을 자동으로 분석하여 검출하고 표시합니다. → **구현됨**: B 트랙 선택 후 Detect, STFT(Meyda) Spectral Flux 온셋 + biased 자기상관 + Tempo Prior + 옥타브 해소로 측정(상수 BPM).
- 사용자가 기준 템포를 맞출 수 있도록 수동 탭 템포(Tap Tempo) 기능을 함께 제공합니다. → **구현됨**: 선형회귀 기반 TAP, 실시간 BPM/탭수 표시.
- ⛔ **현재 범위 불필요**: 변동(variable) BPM / Tempo Map 분석 — 전체 곡 속도 조정은 비례(proportional) 스케일링이라 상수 배율을 모든 시점에 동일 적용하면 원곡 템포 굴곡이 자동 보존되므로 변동 BPM 측정이 필요 없음. 비트 그리드 정렬·템포 곡선 시각화·스템 워프 기능을 도입할 때만 필요(제안.md §제안 D 참조).

### 2.2 ~~트랙별~~ Pitch-preserving BPM 변경 편집 기능 (Time Stretching)
- 각 오디오 트랙의 음정(Pitch)을 원래대로 유지하면서, 재생 속도(BPM / Tempo)만 독립적으로 늘리거나 줄일 수 있는 편집 기능을 구현합니다.
- 이를 통해 서로 다른 템포의 Stem 파일들을 하나의 프로젝트 BPM에 동기화할 수 있게 합니다.

### 2.3 ~~트랙별~~ BPM-preserving Pitch 변경 편집 기능 (Pitch Shifting)
- ~~각 오디오 트랙의 재생 속도(BPM)를 일정하게 유지하면서,~~ 음정(Pitch)~~만~~ 을 반음(Semitones) 단위로 조절할 수 있는 편집 기능을 구현합니다.
- Key 값을 상단 BPM 옆 (VariBPM 스위치 오른쪽) 표시.
- 전체 곡의 Key를 찾는다.
- 반음 단위로 Key를 up/down 하여 플레이하고 변주된 상태로 녹음하는 기능.

## 3. 개발 방향 및 세부 계획 (Plan)

Web Audio API의 기본 `AudioBufferSourceNode.playbackRate` 조절은 템포와 피치가 동시에 변하는 한계가 있습니다. 이를 해결하기 위해 DSP(디지털 신호 처리) 알고리즘을 도입하고, 실시간성 및 오프라인 렌더링(Export) 무결성을 확보하기 위해 다음과 같이 단계별 개발 계획을 수립합니다.

### Phase 1: BPM 측정 엔진 및 UI 구현 — ✅ 완료 (v1.2.6 / TAP v1.2.9)
> **구현 결과**: 초기 계획(로우패스 + 에너지 엔벨로프)과 달리, 최종적으로 **Meyda STFT Spectral Flux 온셋 → biased 자기상관(ACF) + 로그정규 Tempo Prior → harmonic-comb 옥타브 해소** 방식으로 구현해 실제 곡에서 정확한 상수 BPM을 산출함. BPM 표시기/설정 패널(2열), Detect 진행 표시(Analyzing…), 선형회귀 TAP까지 완료. 자세한 경위는 아래 §6 Changelog #38·#41 참조.
- **알고리즘 구현**:
  - ~~`OfflineAudioContext`를 사용하여 임포트된 오디오 데이터를 약 150Hz 이하로 로우패스 필터링(Low-pass filter)합니다.~~ → 미채택. 대신 Meyda STFT Spectral Flux 온셋 곡선 사용.
  - 오디오 신호의 온셋 곡선에서 자기상관 분석(Auto-correlation) + Tempo Prior로 가장 지배적인 비트 간격(BPM)을 산출합니다. → **구현됨**
- **UI 연동**:
  - 검출된 BPM을 상단 BPM 표시기(프로젝트 BPM | 재생 BPM)에 표시합니다. → **구현됨**
  - 수동 `TAP` 버튼으로 박자에 맞춰 클릭하면 BPM이 실시간 계산되도록 지원합니다. → **구현됨**(선형회귀 + 실시간 표시)

### Phase 2: Time-stretching / Pitch-shifting 오디오 프로세서 구현
- **기술 스택 선정**:
  - **AudioWorklet**: 실시간 오디오 처리를 위해 오디오 스레드에서 동작하는 `AudioWorkletNode`를 생성하고, 여기에 DSP 처리 코드를 탑재합니다.
  - **DSP 알고리즘**:
    - **WSOLA (Waveform Similarity Overlap-Add)**: 타임 도메인에서 오디오 신호의 유사성을 기반으로 조각을 이어 붙이는 방식. 연산량이 적고 실시간 처리에 적합하며 단성음(보컬/단일 악기)에 왜곡이 적음.
    - **Phase Vocoder**: 주파수 도메인에서 FFT/IFFT를 통해 위상을 보정하며 재합성하는 방식. 다성음(드럼 믹스, 전체 Stem 믹스)에 적합함.
    - **라이브러리 이식 검토**: Pure JavaScript로 작성된 경량 DSP 라이브러리(예: `soundtouchjs` 등)의 코어를 `AudioWorkletProcessor`로 포팅하여 연산 효율과 안정성을 높입니다.
- **현재 채택 방향(2026.06.14 / v1.2.40 기준)**:
  - 단기 안정 경로는 현행 Electron Export의 `ffmpeg atempo` 후처리를 유지합니다. 이미 `renderMix({ applyTempo:false })`로 원속도 믹스를 만든 뒤 `processAudio()`에서 `atempo`와 `loudnorm`을 한 필터 체인으로 처리하므로, BPM 중복 적용과 Normalize 볼륨 저하 문제는 해결된 상태입니다.
  - 브라우저 fallback의 JS `timeStretchBuffer()`는 품질 기준을 만족하지 못한 실험 경로로 간주하고, 새 기능의 기준 엔진으로 확장하지 않습니다.
  - v1.2.33~v1.2.40에서 기존 JS `timeStretchBuffer()`를 재사용한 **캐시형 실시간 Time Stretch 프리뷰**를 붙이고, 준비 표시/입력 대기/계산 중 입력 정리까지 보정했습니다. B-123~B-130 수동 확인 결과, 현재 UX 기준에서는 Vari BPM 재생 중 피치 보존 요구와 응답성 기준을 만족했습니다.
  - 실시간 AudioWorklet 적용은 현재 즉시 착수하지 않습니다. 실제 곡에서 캐시형 프리뷰의 음질, 지연, 메모리, 다중 트랙 성능 한계가 다시 확인될 때 다음 단계로 검토합니다.
  - SoundTouch/Rubber Band 계열을 채택하기 전에는 WASM 빌드 가능성, Electron 패키징 크기, 브라우저 fallback 필요성, 라이선스/배포 조건을 별도 확인합니다.
  - 참고 구현으로 [BPM_SHIFTER.html](BPM_SHIFTER.html)을 확인했습니다. 해당 파일은 `HTMLAudioElement.playbackRate` + `audio.preservesPitch = true`와 `captureStream()`/`MediaRecorder` 녹음 Export를 사용합니다. 브라우저가 제공하는 피치 보존 재생 품질을 빠르게 확인하는 A/B 데모로는 유용하지만, FocusDAW 본류의 `AudioBufferSourceNode` 믹싱 그래프·`OfflineAudioContext` Export·트랙별 FX/Automation에는 그대로 이식할 수 없습니다.
- **엔진 통합 (`audio-engine.js`)**:
  - 트랙별 신호 체인에 `AudioWorkletNode`를 삽입합니다:
    `source → timeStretchPitchShiftNode → fader → panner → masterBus`
  - 재생 속도 비율(Stretch Ratio)과 피치 시프트 양(Semitones) 파라미터를 실시간으로 노드에 매핑합니다.

### Phase 3: 트랙별 편집 UI 적용 및 내보내기(Export) 연동
- **편집 UI 구현**:
  - 트랙 헤더(M/L 크기)에 `Pitch (Semitones, -12 ~ +12)` 및 `BPM / Tempo Stretch (Ratio 0.5x ~ 2.0x)`를 조절할 수 있는 노브 또는 슬라이더 컨트롤을 추가합니다.
  - 프로젝트 기준 BPM에 맞추어 개별 트랙의 템포가 자동으로 스트레치되도록 하는 "Sync to Project BPM" 토글 스위치를 추가합니다.
- **오프라인 렌더링 연동**:
  - `renderMix()` 함수 내 `OfflineAudioContext` 구성 시 동일하게 `AudioWorkletNode`를 생성 및 동기화하여, 내보낸 MP3/WAV 파일에도 Pitch/BPM 편집 내용이 무결하게 반영되도록 보장합니다.

### Phase 0: BPM 변화 기능 1차 구현 방향 (JUCE 전환 전) — 🟡 1차 구현 완료 + v1.2.33 실시간 프리뷰 시작
> **구현 결과**: `AudioBufferSourceNode.playbackRate` 기반 **전역 템포 변경**(프로젝트 BPM 대비 재생 BPM 비율을 모든 트랙에 동일 적용)은 구현 완료. 단, 아래 "트랙별 원본 BPM 저장 / Sync to Project BPM 토글"은 **제거됨** — 목표가 "전체 음악 BPM 조정"으로 정리되어 전역 Project/Playback BPM만 유지(§6 Changelog #42). v1.2.33부터는 Vari BPM 재생에서 캐시형 Time Stretch 프리뷰를 사용해 피치 보존을 우선 적용합니다.
- **결론**: BPM 변화 기능은 JUCE 기반 오디오 엔진 전환보다 먼저, 현재 Web Audio 기반 엔진에서 1차 프로토타입을 구현합니다.
- **이유**:
  - JUCE 전환을 먼저 진행하면 BPM 기능 구현과 오디오 엔진 전체 교체가 동시에 얽혀 개발 범위와 위험도가 크게 증가합니다.
  - 먼저 Web Audio 기반에서 BPM 기능의 데이터 모델, 사용자 흐름, UI 배치, Export 반영 요구사항을 확정하는 편이 안전합니다.
  - 현재 구조에서 BPM 변화 기능의 최소 기능을 구현하면 이후 고품질 DSP 또는 JUCE 전환 시 필요한 요구사항을 더 명확히 판단할 수 있습니다.
- **1차 구현 범위**:
  - 트랙별 원본 BPM 값을 저장합니다. 자동 검출 전이라도 수동 입력값을 먼저 지원할 수 있습니다.
  - 프로젝트 BPM 값을 저장하고, 트랙별 `Sync to Project BPM` 토글을 추가합니다.
  - v1.2.0~v1.2.32에서는 `AudioBufferSourceNode.playbackRate` 기반으로 템포 변화 동작을 검증했습니다.
  - v1.2.33에서는 Vari BPM 재생 시 트랙별 스트레치 프리뷰 버퍼를 준비하고, 재생 중에는 해당 버퍼를 원속도(`playbackRate = 1`)로 재생해 속도 변경과 피치 보존을 분리합니다.
- **2차 구현 검토**:
  - Pitch-preserving Time Stretch는 `AudioWorklet` 또는 JS DSP 라이브러리 기반으로 별도 프로토타입을 구성합니다.
  - 실시간 재생 품질, CPU 사용량, 다중 트랙 안정성, Export 결과 일치성을 기준으로 채택 여부를 판단합니다.
- **JUCE 전환 판단 기준**:
  - Web Audio / AudioWorklet 기반에서 다중 트랙 실시간 Time Stretch 품질이 부족한 경우. (충족 - 캐시형 프리뷰로 임시 우회했으나 궁극적 실시간 고품질 연산 불가)
  - 실시간 재생과 오프라인 Export 결과를 안정적으로 일치시키기 어려운 경우. (충족 - 재생은 JS 캐시, 내보내기는 ffmpeg atempo로 이원화 상태)
  - VST3, LUFS/True Peak Meter, 고성능 네이티브 DSP 등 데스크톱 DAW 수준 기능이 실제 필수 요구사항으로 확정되는 경우.
  - **결정 (2026.06.15)**: 위의 조건들이 충족되어 Web Audio API의 한계를 극복하고 실시간 Key 변경(+1/-1 반음) 및 고품질 Time Stretch를 완벽히 연동하기 위해 **JUCE 기반의 C++ 네이티브 오디오 엔진 마이그레이션 단계로 즉각 돌입**하기로 결정함.

### Phase 4: JUCE C++ 네이티브 오디오 엔진 마이그레이션 — ✅ 완료 (2026.06.15)
- **배경**: 현재 Web Audio API 기반 엔진은 브라우저 및 스레드 성능 제약으로 고성능 DSP 실시간 연산에 한계가 있음. 특히 반음 단위 Key 변경(Pitch Shift) 및 고품질 Time Stretch를 재생과 내보내기에서 동일한 품질로 제공하기 위해, C++ 네이티브 오디오 엔진 프레임워크인 JUCE로 오디오 코어를 전환함.
- **세부 개발 계획**:
  1. **1단계: JUCE 엔진 아키텍처 및 IPC 프로토콜 설계 — ✅ 완료 (2026-06-15)**
     - C++ JUCE 프레임워크 기반의 독자적인 네이티브 오디오 엔진 프로세스(Daemon) 설계 및 CMake 스켈레톤 생성.
     - Electron 메인 프로세스(Node.js)에서 C++ 서브프로세스 관리 구현 (`spawn` 및 `kill`).
     - 하이브리드 오디오 브릿지(`audio-bridge.js`) 설계/구현 및 Web Audio Fallback 검증 통과.
     - 오디오 재생(Play, Stop, Seek), 트랙 파라미터(Volume, Pan, Mute, Solo), FX 및 볼륨 미터 값을 주고받을 JSON 메시지 포맷 정의.
  2. **2단계: 오디오 디코더 및 트랙 믹싱 그래프 이식 — ✅ 완료 (2026-06-15)**
     - WAV/MP3 오디오 파일 로드 및 디코드 기능을 JUCE 코어 모듈로 이식.
     - `juce::AudioProcessorGraph`를 활용해 트랙 노드, 볼륨 페이더, 팬 노드, 마스터 버스로 이어지는 믹싱 시그널 체인 구현.
     - 재생 시간 및 재생 헤드 위치 정보를 React UI단과 마이크로초 단위로 동기화.

### 114) Key Detection 항상 "C" 버그 재발 수정 (Meyda 크로마 필터뱅크 캐싱) — v1.5.14 · 2026.06.17
- **증상**: Key Detect가 입력과 무관하게 항상 `C`로 검출됨(재발).
- **원인 규명(Node 재현)**:
  - 격리 환경에서 `estimateKey`는 G장조 신호에 대해 정상적으로 `G` 반환 → 로직/Meyda 자체는 정상.
  - 그러나 **BPM 검출이 먼저 실행**되면(`computeOnsetEnvelope`가 `amplitudeSpectrum`을 `bufferSize=2048`로 추출) 이후 Key 검출의 첫 크로마 추출이 깨짐. Meyda는 `chromaFilterBank`를 **처음 만든 bufferSize로 캐싱하고 bufferSize 변경 시 재생성하지 않음**. Key 검출이 `bufferSize=4096`로 바꿔도 필터뱅크가 2048(1025열) 기준으로 만들어져 스펙트럼(2049열)과 차원 불일치 → 크로마가 `NaN`/평탄 → 모든 상관이 0 → 폴백에서 평탄 크로마 → 항상 `C`.
  - 재현 순서: `amplitudeSpectrum@2048` → `bufferSize=4096` → 첫 `chroma` 추출 = `bankCols 1025, sum NaN, peak 0(C)`. 추출 전 `Meyda.chromaFilterBank=undefined` 부여 시 `bankCols 2049, 정상 peak`.
- **수정 결과**:
  - [audio-engine.js](audio-engine.js) `accumulateChroma`: `Meyda.bufferSize/sampleRate/windowingFunction` 설정 직후 `Meyda.chromaFilterBank = undefined`를 추가해, 현재 frameSize(4096)/sampleRate 기준으로 크로마 필터뱅크를 강제 재생성. (BPM→Key 순서로 실행해도 정상. Node 재현 테스트에서 `G` 정상 복구 확인.)
  - audio-engine.js는 직접 로드되어 `build:renderers` 불필요.
  - [version.js](version.js), [package.json](package.json): 앱 버전을 `1.5.14`로 갱신.
- **검증**:
  - 실제 앱에서 BPM Detect 후 Key Detect 시 곡의 실제 키가 나오는지(더 이상 무조건 C가 아닌지) 수동 확인 필요.

  - [studio.html](studio.html): `variBpmBlink` keyframes(opacity + box-shadow를 스무스하게 왕복, 1.6s ease-in-out)와 `.vari-bpm-tag` 클래스 추가.

  - 변동 BPM/Tempo Map은 **비트 그리드 스냅·변속곡 워프(warp)·스템 정렬·템포 곡선 시각화·메트로놈/MIDI 동기·DJ 비트매칭** 같은 기능을 도입할 때에만 필요.

### 38) BPM 측정 알고리즘 전면 재설계 (Onset Strength → ACF + Tempo Prior → 옥타브 해소) — v1.2.6 · 2026.06.13
- **문제점**:
  - 특정 곡(실제 158 BPM)을 studio.html에서 측정하면 145 BPM으로, `bpm-analyzer-code2.html`에서 측정하면 158 BPM으로 서로 다르게 나옴.
  - 원인 분석: 기존 [audio-engine.js](audio-engine.js) `estimateBpm`은 ① 시간영역 진폭 엔벨로프 온셋, ② 홉=윈도우(오버랩 없음, ~40fps)의 거친 프레임 해상도, ③ `(length-lag)` 정규화로 인한 저-BPM 편향 탓에 158을 분해(resolve)하지 못하고 145로 떨어짐.
  - 중간 검토에서 "10초 구간 분할 + 상·하위 2개 trim + RMS" 집계 방식도 폐기: RMS는 템포 집계에 물리적 의미가 없고(상향 bias), 구간 분할은 periodicity 증거를 약화시키며, 서로 다른 옥타브의 BPM 숫자를 평균/RMS하면 무의미한 값이 됨.
- **수정 결과**:
  - [studio.html](studio.html): `audio-engine.js` 로드 직전에 Meyda(`unpkg.com/meyda`) 스크립트를 추가.
  - [audio-engine.js](audio-engine.js): 표준 onset detection + tempo estimation 흐름으로 재설계(구간 분할/RMS 제거, 단일 global 추정):
    - `estimateBpm()`: 곡 **중앙의 연속 구간(최대 75초)** 1개를 분석 대상으로 선택(periodicity 증거 보존).
    - `computeOnsetEnvelope()`: STFT(Meyda `amplitudeSpectrum`, frameSize 2048 / hop 512, ~86fps) spectral flux를 **로그 압축(log1p)** + half-wave rectify로 계산하고, **국소 평균 차감(~0.5초 이동평균)** 으로 DC·드리프트를 제거한 onset strength 곡선 생성.
    - `estimateTempoFromOnset()`: 곡선 **전체에 biased 자기상관(ACF, `/n`)**(40~240 BPM)을 적용하고, **로그정규 tempo prior(중심 120 BPM, σ=0.9 in log2)** 로 가중. ACF의 국소 피크를 후보로 모은 뒤 **harmonic comb score(`Σ r[m·L]`, m=1..4) × prior**로 fundamental을 선택해 **옥타브를 해소**하고, **포물선 보간**으로 lag를 소수점 정밀화.
  - **옥타브 오류 수정(1차 검증)**: 초기 비편향 ACF(`/(N−lag)`)는 큰 lag(저BPM)를 과대평가해 158의 절반인 **79 BPM이 검출됨**. ① biased ACF(`/n`)로 전환해 저-BPM 편향 제거, ② 옥타브 해소를 단일 ½×/2× 비교 → **harmonic comb score**로 교체(기본 템포가 배음 피크 지지를 흡수해 빠른 fundamental 선택)하여 해결.
  - [package.json](package.json) 및 [version.js](version.js): 앱 버전을 `1.2.6`으로 갱신.
- **검증**:
  - `node --check version.js`, `node --check audio-engine.js`로 문법 확인.

### 24) Reverb/Echo 토글 — 슬라이더로 0을 만든 경우엔 on 눌러도 복원하지 않도록 수정 — v1.1.12 (버전 불변) · 2026.06.13
- **문제점**: #21/#22에서 `<key>Stored`는 **버튼 bypass 시에만** 갱신됨. 그래서 사용자가 슬라이더를 0으로 내려 off된 경우에도 Stored엔 이전 양수값이 남아, on을 누르면 그 값으로 복원돼 버림. "내가 0으로 만들어 껐는데 다시 켜면 옛 값이 돌아온다"는 비직관적 동작.
- **수정 방안**: "슬라이더로 0" vs "버튼 bypass"를 구분.
  - 슬라이더 값이 0(≤0.001)이 되면 `<key>Stored`도 0으로 초기화 → 복원할 값 없음.
  - on 토글은 저장값이 양수(또는 미정의→기본 0.4)일 때만 복원, **저장값이 0이면 no-op**(켜지지 않음).
- **수정 결과([ui-mixer.jsx](ui-mixer.jsx))**:
  - `FxCard`(Mixer): 슬라이더 `onChange`를 `setAmount`로 교체 — 값 적용 후 `v≤0.001`이면 `onMaster(storedKey, 0)`. 토글에 `canEnable`(stored 미정의 or >0.001) 도입, off & !canEnable이면 no-op. bypass/복원 시에만 `onBeforeChange`(undo) 기록. 버튼 툴팁/커서를 "raise the slider to enable"/default로 분기.
  - `FxChip`(메인 OUTPUT FX 버튼) + `OutputTrack.toggleFx`: 동일 로직 반영. 메인 창은 슬라이더가 없으므로 stored가 0이면 버튼이 no-op이 됨 → 툴팁을 "set amount in mixer to enable"로 안내. `fxCanEnable(key)`로 커서/툴팁 분기.
  - 기본값(미정의) 처리: 엔진 master 기본 `reverbStored:0.4`/`echoStored:0.35`라 신규/기존 프로젝트는 stored가 정의돼 버튼이 정상 enable됨. stored===0은 오직 "슬라이더로 명시적 0" 경로에서만 발생.
- **검토한 논리적 이슈**:
  - **(주의) 메인 창 버튼이 비활성처럼 보이는 경우**: 메인 창 OUTPUT FX의 REV/ECHO는 슬라이더가 없어, Mixer 슬라이더로 0을 만든 뒤(=stored 0) Mixer를 닫으면 메인 창 버튼만으로는 다시 켤 수 없음(Mixer 슬라이더를 올려야 함). 단, 메인 창에서 버튼 bypass로 끈 경우엔 stored에 값이 남아 정상 복원됨. 즉 "막힌 버튼"은 'Mixer 슬라이더로 0' 한 좁은 경우에 한함. 툴팁으로 안내함.
  - undo: no-op일 땐 `onBeforeChange`를 호출하지 않아 빈 undo 항목이 생기지 않음.
  - 동기화: `storedKey` 0 초기화도 `setMaster`→broadcast 및 SYNC 직렬화(#21 포함)로 메인·Mixer 양쪽에 반영됨.
- **검증**: `npx babel` 무빌드 구조라 별도 빌드 없음. 사용자 수동 시험 결과 정상 동작 확인(2026.06.13) — (1) 슬라이더 0→off→on 시 켜지지 않음, (2) 버튼 bypass→on 시 이전값 복원, (3) 신규/로드 프로젝트에서 버튼 enable 정상 동작.

### 17) Mixer 창 크기 누적 증가(Drift) 버그 최종 해결 및 크기 조절 유지 기능 구현 · 2026.06.13
- **문제점**: 사용자가 Mixer 창을 이동시키거나 Y방향 등으로 크기를 조절할 때, 창 크기가 자동으로 조금씩 늘어나며(Drift) 트랙과 MASTER 패널 사이의 빈 공간이 계속 벌어지는 심각한 버그 발생. 이 버그를 피하기 위해 이전 버전에서는 믹서 창의 크기 조절 기억을 포기하고 강제 고정 크기로 오픈했으나, 사용자는 조절한 창 크기가 정상 유지되길 원함.
- **원인 분석**: Windows 프레임리스 창(`frame: false` + `thickFrame: true`) 환경에서 `getBounds()` 또는 `getContentBounds()` 조차도 OS 인셋 영역을 완전히 배제하지 못하고 특정 리사이즈 방향(특히 Y축)이나 창 재오픈/프로그램적 바운드 설정 시점에 추가 테두리를 포함해 리턴할 수 있음. 이로 인해 리사이즈 이벤트를 메인 프로세스에서만 받아서 저장하면 크기가 미세하게 계속 증가하는 피드백 루프가 형성됨.
- **수정 결과**:
  - **콘텐츠 영역 기준(Content Bounds) + 렌더러 기반 델타(Delta) 하이브리드 제어**:
    - **크기(너비/높이)**: OS 및 윈도우 프레임의 간섭을 일절 배제하고자, 창 생성 후 300ms(배치 및 인셋 할당이 완전히 완료되는 시점) 이후에만 크기 정보 보고를 허용합니다. 사용자가 드래그하여 움직인 **순수 변화량(Delta)**만 계산하여 의도한 가로/세로 크기에 덧셈하는 방식으로, 가로 크기 누적 증가 버그를 차단했습니다.
    - **위치(X/Y)**: 어떠한 딜레이나 무력화 윈도우 없이 창 생성 초기부터 메인 프로세스의 `moved` 및 `resized` 이벤트가 발생하면 `x` 및 `y` 좌표를 즉시 갱신하도록 구성하여, 사용자가 창을 빠르게 움직이거나 드래그 후 곧바로 창을 온/오프하더라도 원래 위치로 돌아가지 않고 마지막 위치가 정확히 저장 및 복원되도록 처리했습니다.
    - `BrowserWindow` 옵션에 `useContentSize: true`를 지정하여 초기 생성 및 복원 시 콘텐츠 크기 기준으로 완벽히 동기화.
  - 이를 통해 창을 Y방향이나 다른 방향으로 수동 조절하고 닫았다가 다시 여는 온/오프 동작을 수없이 반복하더라도 단 1px의 크기 증가(Drift)도 발생하지 않으며, 사용자가 지정한 크기와 위치가 그대로 고정 복원됨.
  - **검증**: 창 빠른 드래그 이동 및 즉각적인 온/오프 시 위치 기억 복원, 그리고 Y축 리사이즈 시 가로 크기 동결 검증 완료.

### 16) Mixer 창 왼쪽 가장자리 리사이즈 후 재오픈 시 화면 밖으로 이동하는 버그 수정 · 2026.06.13
- **문제점**: Mixer 창을 화면 오른쪽에 두고 **왼쪽 가장자리를 끌어 폭을 넓힌 뒤** 닫았다 다시 열면, 창이 오른쪽으로 크게 이동해 화면 밖으로 벗어남.
- **원인 분석**: `electron/main.js`가 크기와 위치를 **서로 다른 두 캐시에 따로** 저장하고 있었음 — `'resized'` 핸들러는 `mixerWinSize`(width/height)만, `'moved'` 핸들러는 `mixerWinPos`(x/y)만 갱신. 그런데 **왼쪽 가장자리 리사이즈는 x와 width가 동시에 바뀌는 동작**이고 Windows에서 이때 `'resized'`만 발생(`'moved'`는 미발생). 결과적으로 **새 width는 저장되지만 x는 리사이즈 이전(오른쪽) 값으로 남아** 둘이 어긋남 → 재오픈 시 `옛 x + 늘어난 width`가 화면 오른쪽 끝을 수백 px 초과(실측: 오른쪽 끝 예상 3302 vs 화면 2752).
- **수정 결과**:
  - 분리돼 있던 `mixerWinSize`/`mixerWinPos`를 **단일 `mixerBounds`(x,y,width,height)** 로 통합하고, `'resized'`·`'moved'` 양쪽에서 **동일한 `getBounds()` 스냅샷 한 번**으로 함께 저장 → x와 width가 항상 일관됨.
  - 추가 방어: `clampMixerBounds()` 도입(`screen.getDisplayMatching` 사용). 재오픈 시 복원 좌표/크기를 해당 디스플레이 **작업영역 안으로 클램프** → 모니터 변경·엣지 리사이즈 등 어떤 경우에도 창이 화면 밖으로 나가지 않음.
  - **검증(실측)**: 대량 off-screen 케이스(오른쪽 끝 3302) → 클램프로 x를 당겨 오른쪽 끝 2752(정확히 화면 안). 왼쪽 가장자리 리사이즈 케이스 → 재오픈 후 화면 안(잔여 +2px는 항목 14-1의 #51679 인셋 WONTFIX 수준). `node --check` 통과, 실제 앱 부팅 정상.
  - 참고: 이 수정은 항목 14-1)의 "이동 후 1회 ~2px 증가"(#51679, WONTFIX)와는 별개의 위치 동기화 버그를 잡은 것. ~2px 잔여는 동일하게 남음.
- **▶ 회귀 수정(16-1, 2026.06.13)**: 위 통합(`mixerBounds`) 직후 테스트에서 **"창을 이동→닫기→열기 반복 시 크기가 계속 누적 증가"** 회귀 발생. 원인: `'moved'` 핸들러가 **전체 `getBounds()`(크기 포함)** 를 저장하도록 바뀌었는데, Windows 프레임리스 창은 이동 시 #51679 인셋으로 **live 크기가 +2px 부풀고**, 그 값이 캐시→재오픈 시 x/y로 또 부풀→재캐시되는 **피드백 루프** 형성. (항목 14에서 막았던 것과 같은 부류의 루프가 'moved' 경로로 부활.)

- **원인 분석**:
  - **근본 원인(플랫폼 버그)**: Windows의 frameless(`frame:false`) + `thickFrame` 창에서 `getBounds()`가 설정한 값과 **가역적이지 않음**. 보이지 않는 리사이즈 테두리(`SM_CXSIZEFRAME + SM_CXPADDEDBORDER`) 때문에 `setSize(X)` 직후 `getBounds()`가 `X±α`를 돌려줌. - **문제점**: `SettingsDialog` 내부의 `Mixer Console Window` 리셋 카드 박스에 테두리(`border`)와 `background: var(--surface)` 스타일이 잔존하여 어두운 카드 형태로 보임.
- **수정 계획 및 결과**: `ui-dialogs.jsx`의 `SettingsDialog` 리셋 영역 인라인 스타일에서 테두리(`border`)와 배경색(`background`), 모서리 깎임(`borderRadius`)을 삭제하고 `padding`을 `8px 0`으로 조절하여 창 내부에 일체형으로 녹아들도록 스타일을 다듬음.

### 2) Settings 창 믹서 초기화 미작동 버그 수정
- **문제점**: 리셋 버튼을 클릭하여 `localStorage` 및 Electron 메인 프로세스 전역 `lastMixerBounds` 캐시를 비우더라도, 이미 믹서 창이 열려 있는 상태라면 믹서 창이 닫힐 때(`beforeunload` / `close` 이벤트) 믹서 창의 현재 위치와 크기를 다시 읽어와 `localStorage` 및 `lastMixerBounds`에 **다시 덮어씌워버려** 실제 초기화가 적용되지 않음.
- **수정 계획**:
  - **Electron**: `reset-mixer-bounds` 핸들러 호출 시 `isMixerBoundsReset = true` 플래그를 설정하고, 열려 있는 `mixerWindow`가 있다면 즉시 `close()` 처리함. `mixerWindow`의 `close` 이벤트 리스너에서 `isMixerBoundsReset`가 참인 경우 bounds 기록을 스킵하고 캐시를 `null`로 강제 유도함.
  - **웹 브라우저**: `SettingsDialog` 리셋 버튼 클릭 시 `localStorage.removeItem("focusdaw-mixer-bounds")`를 수행하며, 열려 있는 팝업 객체(`window.mixerPopup`)가 있다면 즉시 `window.mixerPopup.close()`를 수행함. 팝업이 닫힐 때 트리거되는 `saveBounds` 리스너에서는 `localStorage.getItem("focusdaw-mixer-bounds")`가 이미 비어있다면(`null`인 경우) 저장을 건너뛰고 return 하도록 예외 처리하여 캐시 덮어쓰기를 원천 방어함.

### 3) 사용자 매뉴얼 내용 일치화 및 도움말 매뉴얼 F3 단축키 추가
- **문제점**: `manual/사용자 메뉴얼.html` 파일과 앱 도움말의 `ui-help.jsx` 한글 설명 문구 간에 미세한 자구 불일치(예: 믹서 페이드 핸들 표시 관련 텍스트 차이 등)가 있었으며, 앱 도움말 매뉴얼의 단축키 표에서 F3 (믹서 콘솔 열기 / 닫기) 단축키 설명이 누락되어 있었음.
- **수정 계획 및 결과**: 
  - `ui-help.jsx` 의 한글 및 영문 단축키 목록 테이블에 F3 단축키 설명을 신규로 보완 추가함.
  - `manual/사용자 메뉴얼.html` 의 본문 내용을 `ui-help.jsx` 에 구성된 한글 도움말 매뉴얼 텍스트와 정확히 100% 동일하게 일치시켜 사용자 경험의 일관성을 확보함.
