## v1.43.23 — take 개수 배지 클릭 시 play bar가 이동하던 버그 수정 (2026-07-23)

> **재빌드: renderer([ui-tracks.jsx](ui-tracks.jsx) esbuild) + version metadata.** **엔진·네이티브 무변경.** 사용자 요청(2026-07-23): Audio input 트랙 좌측상단의 take 개수 버튼("2 takes")을 누르면 play bar가 클릭 지점으로 원치 않게 이동. take 개수 버튼을 누를 때는 play bar에 영향이 없어야 함.

### 원인
- take 배지 버튼([ui-tracks.jsx](ui-tracks.jsx))은 `onClick`에서 `e.stopPropagation()`을 했으나, play bar seek는 **부모 레인 div의 `onMouseDown`**(`laneClick` → `onSeek(sec)`)에서 발생한다.
- `mousedown`은 `click`보다 먼저 발생하므로, 버튼을 누르는 순간 mousedown이 부모 레인으로 버블링되어 이미 seek가 실행된다. 그 뒤에 오는 `onClick`의 `stopPropagation`은 지나간 mousedown을 취소할 수 없어 무력.

### 수정 내용
- [ui-tracks.jsx](ui-tracks.jsx): take 배지 버튼에 `onMouseDown={(e) => e.stopPropagation()}` 추가 → mousedown이 부모 레인의 seek(및 트랙 선택) 핸들러에 도달하지 못하게 차단. 버튼의 toggle 동작(`onClick`)은 그대로 유지. 부수효과로 배지 클릭이 트랙 선택을 바꾸지 않게 되는데, 토글 버튼 성격상 바람직.
- [version.js](version.js)·[package.json](package.json)·[ui-help.jsx](ui-help.jsx)(릴리즈 노트 range·fix 항목) 버전 `1.43.23`.

### 검증
- `node scripts/build-renderers.js` 빌드 통과. 정적 분석으로 mousedown 버블링 차단 확인. ⚠️ T-1.43.23-1 사용자 시험 대기.

## v1.43.22 — 메뉴 핸들러 stale 클로저 수정 (저장된 프로젝트에서 메뉴 Save가 Save As처럼 동작) (2026-07-23)

> **재빌드: renderer([app.jsx](app.jsx) esbuild) + version metadata.** **엔진·네이티브 무변경.** T-1.43.18-3(미저장 프로젝트 안내) 시험 중 사용자가 보고: 저장된 프로젝트를 File▸Open으로 열고 아무 편집 없이 **메뉴 ▸ Save Project**를 누르면 Save As처럼 저장 다이얼로그가 뜨고, 터미널에 `[collect] failed ... Invalid source audio path`가 찍혔다. Clean Up Unused Recordings도 "먼저 저장" 안내가 계속 떴다. **키보드 Ctrl+S는 정상**이라 "키보드와 메뉴가 다르게 동작"으로 좁혀졌다.

### 원인
- `App`이 렌더 시점에 `const H = handlersRef.current`로 메뉴 핸들러 객체를 캡처했다. 그런데 `Studio`가 핸들러를 등록하는 `useEffect`는 **렌더 커밋 이후** 실행되고, ref 갱신(`handlersRef.current = h`)은 리렌더를 유발하지 않는다. 그래서 프로젝트 로드로 `saveProject`가 새 `projectPath`를 담아 재생성돼도, 렌더에서 캡처된 `H`는 **한 세대 뒤처진 초기 마운트 `saveProject`(projectPath=null, projectName="untitled")**를 가리켰다.
- 키보드 Ctrl+S는 keydown `useEffect` deps에 `saveProject`가 있어 클로저가 최신이라 정상. **메뉴만 stale** → 저장된 프로젝트인데 `projectPath=null`로 동작해 Save As 다이얼로그 + collect가 상대 source 경로를 null 기준으로 풀어 `Invalid source audio path` 실패, Clean Up 게이팅(`!projectPath`)도 오작동.
- 사용자 로그로 확정: `[SaveDBG] saveProject ... projectPath="…dfghjkl.focus"`(Ctrl+S, 정상) 직후 `[SaveDBG] saveProject ... projectPath=null projectName="untitled"`(메뉴, stale) 2회 호출.

### 수정 내용
- [app.jsx](app.jsx): `const H = handlersRef;`로 바꾸고 모든 MenuBar 핸들러를 **호출 시점에** `H.current.onXxx`로 ref를 읽도록 변경 → 메뉴가 키보드와 동일하게 항상 최신 핸들러 사용. (신규 메뉴 항목 추가 시에도 이 call-time-ref 패턴 유지.)
- [app.jsx](app.jsx) `collectProjectAudioForSave`: 상대 source 경로를 `projectPath || targetPath` 기준으로 해석(방어) — `projectPath` 상태가 순간적으로 비어도 저장 대상 폴더 기준으로 풀린다.
- 버전 `1.43.22`.

### 검증
- `node scripts/build-renderers.js` 전체 빌드 통과. 사용자 재확인: File▸Open → 메뉴 Save Project 시 다이얼로그 없이 제자리 저장, Clean Up 정상. 상세 메모리 [[menu-handlers-stale-ref]].

### 결정 — 이전 "구현 안 함"을 뒤집음

- [audio-bridge.js](audio-bridge.js) `setClipGain` 래퍼: 웹 엔진 적용 후 native면 `syncTrackToNative`.
- [app.jsx](app.jsx) `handleSetClipGain`: 드래그당 **undo 1개**(변화 없으면 `cancelUndo`). 두 `TrackRow` 렌더 지점에 `onSetClipGain` 전달.
- [ui-tracks.jsx](ui-tracks.jsx): 클립 박스 안에 **드래그 가능한 수평 볼륨 라인**(select 모드, editable 트랙). `startClipDrag`에 세로 드래그 `gain` 모드 추가 — 드래그 중에는 ghost 라인 + dB 라벨만 표시하고 mouse-up에 1회 커밋(move/trim과 동일 granularity, 매 mousemove bake 방지). 라인은 **선택된 클립 + 이미 감쇠된 클립**에 표시(trim 핸들 피해 좌우 8px inset).

## v1.43.7 — 일반 clip 이동 시 비활성 Take clip이 함께 움직이던 문제 수정 (사용자 보고 버그) (2026-07-21)

> **재빌드: [audio-engine.js](audio-engine.js)(플레인 로드) + renderer 번들.** **UI·네이티브 무변경.** 사용자 보고(2026-07-21): Audio In 트랙에 일반 clip 3개가 있고, 3번 clip에 punch-repeat로 Take A/B/C/D 4개를 만든 뒤 Take B가 active인 상태에서 1번 또는 2번 clip을 이동하면 **비활성 Take A/C/D가 같이 이동**했다. 3번 active Take B clip을 이동할 때 A/B/C/D가 함께 이동하는 것은 정상. T-1.43.7-1 사용자 시험 통과.

### 원인 — take sibling 이동 보정이 모든 clip 이동에 무조건 걸렸다

Stage 4/Comp 흐름에서 active Take clip을 움직이면 inactive Take clip도 같은 delta로 따라 움직여야 한다. 이를 위해 [audio-engine.js](audio-engine.js)의 `moveClip`/`moveClipsByResolved`/`moveClipsBy`/`nudgeClip`이 이동 후 `_shiftInactiveTakeClips(t, delta)`를 호출했다.

문제는 `_shiftInactiveTakeClips`가 **이동한 대상 clip이 active Take clip인지 확인하지 않고**, 현재 active take가 아닌 **모든 take clip**을 이동시켰다는 점이다. UI는 [ui-tracks.jsx](ui-tracks.jsx)에서 `takeId=null` 일반 clip과 active take clip만 메인 레인에 표시하므로 사용자는 1번/2번 일반 clip만 움직였지만, 엔진 커밋 단계에서 비활성 Take A/C/D가 추가로 밀렸다.

### 수정 — 이동 대상에 active Take clip이 포함된 경우에만 sibling Take를 이동

- `_shiftInactiveTakeClips(track, delta, movedClipIds)`로 호출 계약을 좁혔다.
- 함수 내부에서 `movedClipIds` 중 **현재 active takeId를 가진 clip이 실제로 있는 경우에만** inactive take clip을 이동한다.
- 일반/base clip(`takeId=null`) 이동은 sibling Take 이동을 트리거하지 않는다.
- 만약 다중 선택/미래 경로에서 이미 이동 대상에 inactive take clip이 들어온 경우를 대비해, 보조 이동 루프는 `movedClipIds`에 포함된 clip을 다시 움직이지 않게 했다.
- 적용 경로: `moveClip`, `nudgeClip`, `moveClipsBy`, `moveClipsByResolved`. 따라서 마우스 단일 드래그, 다중 선택 이동, 키보드 nudge 모두 같은 판정을 쓴다.

### 기대 동작

### 수정 — 순서를 바로잡는다

`saveProject`에서 **`onProjectPathChange(result.path)`를 `onRenameProject` 뒤로** 옮겼다(File>Open과 동일 순서). 이제 rename이 null로 만든 뒤 pathChange가 저장경로로 되살려, projectPath 상태·autosave 스냅샷이 **모두 올바른 절대경로**로 유지된다.

> **재빌드: renderer([app.jsx](app.jsx) esbuild).** **엔진·네이티브 무변경.** 사용자 보고(2026-07-21): "Audio In 트랙 1차 녹음 후, **Repeat 구간을 지정하지 않은 채** Punch를 켜고 30초 부근에서 ~3초 녹음 → **1차 녹음 전체가 사라지고** 새 3초만 남았다"(스크린샷 2장). 사용자 결정(2026-07-21): **메시지 창 대신 버튼 자체를 비활성화** · 게이트 조건 = **"Repeat 구간이 있으면"(ON/OFF 무관)** — 단일 Punch(Repeat OFF) 보존을 위해(대안 "Repeat ON일 때만"은 Loop-Punch만 남고 단일 Punch가 UI에서 사라짐). 시험 대기 1건([시험.md](시험.md) T-1.43.1-1).

### 원인 — Punch가 구간 없이는 "일반 녹음"으로 조용히 강등되고, 일반 녹음은 v1.40.2부터 전체 교체다

이 앱의 Punch는 **Repeat 구간(`DAW.loopRange`)을 [in→out] 스팬으로 재사용**한다([app.jsx](app.jsx) `startRecordFlow`). 그래서 punch 판정이 `region`(구간 유효)에 걸려 있다:

```js
const region = !!(lr && (lr.end - lr.start) > 0.05);
const punch  = !!(isPunchOn() && !DAW.repeatPlayEnabled && region);
```

**구간이 없으면 `region=false` → Punch 버튼이 켜져 있어도 `punch`/`loopPunch`가 둘 다 false** → punch 경로를 타지 않고 **일반 `attachRecording`** 으로 흐른다. 그런데 일반 녹음은 **v1.40.2에서 "트랙 전체 교체"로 바뀐 파괴적 경로**다(`track.clips = [새 클립 하나]`). 결과적으로 1차 녹음이 통째로 지워지고 새 클립만 남는다 — 스크린샷과 정확히 일치. Punch 버튼이 **구간 없이도 눌러졌던 것**이 근본 함정이었다.

### 수정 — 버튼을 구간에 게이팅(모달 없음)

- **Punch 버튼 `disabled={!punchRegion}`** — 유효한 Repeat 구간이 없으면 버튼 자체를 비활성화(기존 메트로놈 버튼의 `disabled`+`title` 패턴과 동일). 무효-구간 툴팁을 **"To use Punch, set a Repeat region first — drag one on the timeline."** 로 교체.
- **`MenuTransport` 본문 auto-heal** — `if (!punchRegion && isPunchOn()) setPunchOn(false)`. `useTick()`로 매 프레임 리렌더되므로, 구간을 **나중에 지운 경우**([ui-mixer.jsx](ui-mixer.jsx)의 `setLoopRange(null)`)에도 Punch 플래그가 즉시 꺼진다 → **"구간 없이 Punch ON 고착" 상태가 원천 봉쇄**되어, 녹음이 파괴적 전체 재녹음으로 새지 않는다.
- **Punch 기본값 off** — 이미 그렇다(`focusdaw-punch` 미설정 시 false).
- 버튼 게이트 + auto-heal이 모든 진입을 막으므로 `startRecordFlow`에 별도 record-time 가드(모달)는 두지 않는다.

**게이트 조건은 "구간이 있으면"(Repeat ON/OFF 무관)** 이라 **단일 Punch(Repeat OFF·구간)** 와 **Loop-Punch(Repeat ON·구간)** 가 **둘 다 유지**된다. 일반(전체 교체) 녹음을 원하면 Punch를 끈다(구간이 있어도 버튼은 자유롭게 토글 가능).

### 검증

`node scripts/build-renderers.js` 전체 빌드 통과(`built build\app.js`), 번들에 새 툴팁 반영 확인. ⚠️ **실제 녹음 흐름은 T-1.43.1-1(사용자)** — 마이크 입력이 필요해 자동화 불가.

## v1.43.0 — Save As 오디오 수집 + 상대경로 (Phase 7) (2026-07-21)

> **재빌드: renderer([app.jsx](app.jsx) esbuild) + [electron/main.js](electron/main.js)·[electron/preload.js](electron/preload.js)·[audio-engine.js](audio-engine.js)(플레인).** **네이티브 무변경.** 사용자 결정(2026-07-21): 수집 범위 = **앱 생성 오디오만**(import 스템은 제자리 참조) · 경로 = **상대경로(이식가능)**. 시험 대기 1건([시험.md](시험.md) T-1.43.0-1).

### 무엇 / 왜

종전 `save-project`는 **JSON만 쓰고 오디오는 복사하지 않았다.** 그래서 ① 미저장 프로젝트의 녹음·consolidation은 temp(`FocusDAW Recordings`/`Consolidated`)에 남아 **OS가 temp를 청소하면 재열기에서 소실**되고, ② `.focus`가 자기완결적이지 않아 **폴더를 옮기면 깨졌다**. Save As가 프로젝트의 앱 생성 오디오를 자기 폴더로 수집하고 경로를 상대로 바꿔 **자기완결·이식가능**하게 만든다.

### 흐름 (경로 선택 → 수집 → JSON → 기록)

`save-project`가 다이얼로그+기록을 한 번에 하던 것을 분리했다. `saveProject`는:
1. **경로 확보** — Save As/최초 저장이면 `choose-project-path`(다이얼로그만, 기록 안 함)로 경로를 먼저 받는다. 기존 저장 프로젝트의 일반 Save는 현재 경로 사용.
2. **수집** — `collectProjectAudioForSave` → `collect-project-audio` IPC가 앱 생성 소스(녹음/Bounce/Consolidated)를 `<Project> Audio/<category>/`로 복사(충돌 시 유니크 이름)하고, 각 소스를 **`.focus` 기준 상대경로**로 되돌려준다. 이미 프로젝트 폴더 안이면 복사 없이 상대경로만(멱등). 실패는 best-effort(그 소스는 옛 경로 유지).
3. **JSON export** — 수집 **후**에 export해 새 상대경로가 담기게 한다.
4. **기록** — `save-project`가 JSON을 쓴다.

- **`DAW.setSourcePath(trackId, sourceId, filePath, fileName)`** 신설 — 수집 결과 경로를 라이브 소스에 반영(+primary면 `track.filePath`도). 로드된 버퍼는 불변, 재열기 메타만 갱신.
- **import 스템은 수집 안 함**(사용자 결정) — `track.kind==="file"`이고 consolidated 아님 → skip. 절대경로 그대로 참조.
- **카테고리 판정**: `kind==="bounce"`→Bounces, consolidated(경로/이름에 "Consolidated")→Consolidated, `kind==="audioIn"`→Recordings.

### 상대경로 재열기 (하위호환)

- `resolveSourcePath(filePath, projectPath)`: **절대경로**(import 스템·구프로젝트)는 그대로, **상대경로**(수집된 파일)는 프로젝트 폴더 기준으로 해석. `read-audio-file`의 `path.resolve`가 혼합 구분자를 정규화.
- ⚠️ **`reconnectProjectAudio(pathForOpen)`** — 여는 프로젝트 경로를 **인자로 명시 전달**한다. open 흐름(`loadProjectJson`)에서 `projectPath` **prop은 아직 리렌더 전**이라, 클로저의 낡은 경로로 상대해석하면 깨진다. 그래서 `nextPath`를 직접 넘긴다.
- 절대경로 분기가 그대로라 **기존 프로젝트(절대경로 저장분)는 무영향**.

### 검증

`node --check`(main/preload/engine) + 전체 빌드 통과 · 격리 부팅 정상(에러 없음). 스탠드얼론 검사: **경로 헬퍼 13검사**(Windows 드라이브/UNC/POSIX 절대 판정·상대해석·`path.resolve` 정규화) + **collect 로직 11검사**(temp→폴더 복사·상대경로·멱등 재저장·Save As 새 폴더·이름 충돌 유니크·원본 불변) 전부 통과. ⚠️ **실제 다이얼로그 Save As·재열기 복원은 T-1.43.0-1(사용자)** — 네이티브 다이얼로그라 자동화 불가.

## v1.42.0 — Comp Lane 구성: Take 레인 스와이프 (Phase 7) (2026-07-21)

> **재빌드: renderer([app.jsx](app.jsx)·[ui-tracks.jsx](ui-tracks.jsx) esbuild) + [audio-engine.js](audio-engine.js)·[audio-bridge.js](audio-bridge.js)(플레인 로드).** **네이티브 무변경**(comp는 기존 bake→baked buffer 경로로 흘러 native는 평탄화 결과만 받는다). 사용자 결정(2026-07-21): 조작 방식은 **스와이프(대표 DAW 방식)**. ✅ **T-1.42.0-1 통과**(2026-07-21 → [시험-아카이브.md](시험-아카이브.md)). **Phase 7 "Comp Lane 구성" 완료 — Phase 7의 마지막 comp 조각.**

### 무엇

Take 레인 위에서 **마우스로 구간을 드래그(스와이프)**하면 그 구간이 해당 Take로 comp된다. 구간마다 다른 Take를 골라 하나의 테이크를 합성하는 것(vocal comping). 종전에는 `activeTakeId` **하나**가 트랙 전체에 적용돼 "구간별로 다른 Take"를 담을 그릇이 없었다.

### 모델 — `track.comp` 구간 맵 (비파괴)

`track.comp = [{ start, end, takeId }]` — 정렬·무겹침·gap 없이 take 구간을 타일링하는 세그먼트 목록. 각 구간은 지정된 Take의 오디오를 그 구간에 맞게 **슬라이스**해 재생한다.

- **단일 진실원본 `_renderClips(track)`**: comp가 있으면 base 클립(takeId=null, 항상 렌더) + 각 세그먼트의 Take를 슬라이스한 클립을, 없으면 기존 `_activeClips`(단일 활성 레인)를 반환. **bake·볼륨 커브·Flatten이 모두 이걸 읽으므로** comp가 재생/Export/native/커밋에서 동일하게 나온다.
- **crossfade 공짜**: comp 경계(다른 Take가 맞닿는 지점)는 `_planClipEdges`가 **seam**으로 분류해 v1.38.0 equal-power crossfade를 그대로 탄다. ⚠️ 루프 Take는 **같은 소스**를 `sourceOffset`으로만 구분하는데, seam 판정이 sourceId만이 아니라 **offset 연속성**(`b.srcStart === a.srcStart + a.n`)까지 보므로 A→B 경계(offset이 passDur만큼 점프)가 continuous로 오판되지 않는다.
- **seed + interval-replace**: 첫 스와이프 때 comp를 **현재 활성 Take로 전 구간 시드**해 gap 없이 타일링하고, 이후 스와이프가 구간을 분할·재지정. 전 구간이 한 Take로 collapse되면 comp=null(그 Take를 active로) — 되돌아갈 escape hatch.
- **Flatten 연동**: comp가 있으면 `flattenComp`가 **comp된 오디오**(구간별 Take)를 확정. 없으면 종전처럼 단일 활성 레인. 커밋 후 comp=null·takes=[].
- **hygiene**: `_normalizeTrackLayout`이 매 serialize/undo/import마다 comp를 sanitize — 삭제된 Take를 가리키는 세그먼트는 활성 fallback으로 재지정, 구간 클램프, 한 Take/무 Take로 degenerate 시 null. `setActiveTake`/`deleteTake`는 comp를 폐기(레인 전체 선택이 per-region 선택을 대체).
- **직렬화·undo**: `comp`를 `_addTrack`·serialize 2곳·getSnapshot·applySnapshot·importProject·app.jsx `snapshotSig`에 추가 → 저장/재열기·undo/redo 정합.

### UI — [ui-tracks.jsx](ui-tracks.jsx) `TakeLaneRow`

- **스와이프**: 레인에서 4px 이상 드래그하면 구간 선택 사각형이 뜨고, 놓으면 `setCompRegion`. 4px 미만은 클릭으로 간주해 종전처럼 **Take 활성화**(= 그 Take로 comp 리셋).
- **오버레이**: comp가 있으면 각 레인을 어둡게 덮고, **그 Take가 comp에 기여하는 구간만 밝게**(앰버 하이라이트) 뚫어 한눈에 comp 구성을 보여준다.
- **Flatten 행**: comp 활성 시 `Clear` 버튼 추가(comp 폐기 → 단일 Take), 안내 문구를 스와이프 설명으로 교체.
- 메인 레인은 baked buffer를 그리므로 **comp 결과가 자동 반영**(별도 코드 없음).

### 검증

`node scripts/build-renderers.js` 전체 빌드 통과 · `node --check` 통과. 인터벌 로직(setCompRegion interval-replace + coalesce + renderClips 슬라이스) **스탠드얼론 8검사 통과**(A/B/A 타일링·재지정 merge·전구간 collapse·클램프·슬라이스 sourceOffset·base 보존). ✅ **실제 comp 재생/Export/재열기/Flatten/Take삭제 경계까지 T-1.42.0-1 통과**(2026-07-21, 사용자) — 판정 기준 ①~⑦ 전부 정상.

## v1.41.4 — 성능 계측 계장: `[perf bake]` 로그 (2026-07-20)

> **재빌드: 불필요** — [audio-engine.js](audio-engine.js)는 플레인 로드(esbuild 대상 아님), 네이티브 무변경. 시험 대기 2건([시험.md](시험.md) T-1.41.4-1·2).

### 왜

"향후 개발 후보 · 우선순위 높음"에 **전략 B bake의 편집 반응 속도 계측**과 **30분+ 장시간 녹음 실측**이 올라 있었는데, **둘 다 실측 기록이 없어 최적화가 필요한지조차 판단할 수 없는 상태**였다. 계측 수단이 없으면 시험이 "느낌상 느린가"가 되므로, 먼저 숫자를 볼 수 있게 만들었다.

### 수정

`_ensureBaked`에 **bake / peaks / total 3구간 타이밍**을 넣고 한 줄로 출력한다:

```
[perf bake] <트랙명> len=63.2s clips=2 bake=18.4ms peaks=25.1ms total=47.3ms
```

- **`bake`와 `peaks`를 분리한 것이 핵심.** 어느 쪽이 지배적이냐가 최적화 방향을 가른다 — `peaks`가 크면 **피크 계산 Web Worker 이전**(작업량 작음), `bake`가 크면 **부분 재렌더**(전략 B 전제를 건드리는 큰 작업)다. 합계만 재면 이 판단을 못 한다.
- `len`(트랙 전체 길이)을 함께 찍어 **비용이 트랙 길이에 비례**한다는 전략 B의 특성을 수치로 확인할 수 있게 했다. 정비례를 벗어나 튀는 지점이 있으면 별도 병목이다.
- 편집은 사용자 조작(드래그 종료 등)마다 1회이므로 로그가 넘치지 않는다.

⚠️ **배포 빌드에서는 콘솔을 열 수 없다** — [electron/main.js](electron/main.js) `devTools: !app.isPackaged`. 계측은 반드시 `npm start`로 한다.

### 검증

`node --check audio-engine.js` 통과. **계측 코드는 값을 읽기만 하므로 동작에 영향이 없다**(무해함은 T-1.41.4-1 판정 기준 ⑥으로 확인). 수치 자체는 사용자 시험으로 확보한다.

**High >4kHz가 세 경우 모두 −0.1** → v1.41.2에서 클리퍼를 경로에서 빼낸 것으로 **하모닉 왜곡은 종결**(수정 전 −9는 +2.5였다). 남은 노이즈는 성격이 다르다 — **LRA가 함께 무너지는 것**이 단서로, 리미터가 피크를 넘어 **음악의 다이내믹 자체**를 건드리는 구간에서 들린다. 가청 경계는 **crest 감소 −3 dB / LRA −0.3 부근**.

### 수정

- **룩어헤드 2ms → 8ms.** 게인 램프가 룩어헤드 구간에 걸쳐 있으므로, 2ms는 킥 한 주기(55Hz=18ms)보다 훨씬 짧아 **저역 파형 자체를 변형**시켰다. 8ms는 킥 반 주기 수준으로 마스터링 리미터의 통상 범위다. 샘플당 게인 이동이 **1.136% → 0.284%** 로 완만해졌다.
- **릴리즈 50ms → 80ms + 게인에 60ms 글라이드.** 글라이드는 **회복할 때만** 적용하고 **감쇠는 즉시** 반영하므로 천장은 그대로 보장된다.
- **피크 검출을 단조 deque 슬라이딩 최대값으로 교체.** 종전에는 샘플마다 버퍼 전체를 스캔했다(8ms면 샘플당 ~700회 비교가 될 뻔했다). 이제 O(1) amortised라 **룩어헤드를 4배로 늘리고도 종전보다 빠르다**.

### 검증

증분 빌드 성공 → `bin/` 교체 → 격리 프로필 부팅 정상. **시뮬레이션 6검사 통과**(소스가 기대 형태인지 정규식으로 먼저 확인):

### 진짜 원인 — 리미터 게인이 한 샘플 만에 점프

[AudioEngine.cpp](juce_skeleton/Source/AudioEngine.cpp)의 Export 리미터:

```cpp
envelope = std::max(envelope * releaseFactor, reduction);   // ← 즉시 점프
```

릴리즈는 있는데 **어택 스무딩이 없다.** 필요한 감쇠량이 나오는 즉시 게인이 **한 샘플 만에** 그 값으로 뛴다 → 파형에 계단(불연속) → **클릭**. 측정: 순간적으로 튀는 트랜지언트(스네어)에서 한 샘플에 **게인 42.2% 급락**.

증상 대조표가 전부 맞는다:

| 관찰 | 설명 |
|---|---|
| 드럼 타격에서 노이즈 | 피크가 임계를 넘는 **매 타격마다** 게인 계단 |
| −9/−12/−14 **동일** | 게인만 충분하면 어느 목표든 피크가 임계 초과 → 계단 발생. **감쇠량이 아니라 "넘었는가"가 조건** |
| Normalize OFF 정상 | `targetGain=1.0` → 피크가 −1 dBFS 아래 → 리미터 미작동 |
| −23 LUFS 정상 | 감쇠 방향 → 임계 아래 |
| 인앱 재생 정상 | 이 리미터는 **Export 전용** |

### 수정 — 룩어헤드 구간에 걸친 램프

```cpp
const float riseStep = 1.0f / (float)limiterLookahead;
...
if (reduction > envelope) envelope = std::min(reduction, envelope + riseStep);
else                      envelope = std::max(reduction, envelope * releaseFactor);
```

새 피크는 룩어헤드(2ms/88샘플) **앞서** 버퍼에 들어오므로, 샘플당 `1/limiterLookahead`씩 올리면 **피크가 출력에 도달하기 전에 게인이 제자리에 닿는다** — 계단이 램프가 된다. 하강은 종전 릴리즈 곡선을 쓰되 `std::max(reduction, …)`로 **천장은 계속 지킨다**.

## v1.41.0 — Export LUFS Normalize: 음량을 올릴 때 리미터 체인으로 교체 (2026-07-20)

> **재빌드: [electron/main.js](electron/main.js)·[version.js](version.js).** **renderer·엔진·네이티브 무변경.** 사용자 보고(T-1.38.0-3 중 발견): "−9 LUFS Export에서 드럼 쿵쿵대는 부분에 노이즈가 심하다, −14에서도 미묘하다, 인앱 재생은 정상". 사용자 결정: **체인만 수정**(UI 알림은 추가하지 않음). 시험 대기 1건([시험.md](시험.md) T-1.41.0-1).

### 원인 (v1.38.0 조사에서 확정)

`loudnorm`은 **`linear=true`를 필요 게인이 음수일 때만 지킨다.** 게인이 양수면 true peak가 `TP=-1`을 넘기므로 **Dynamic 모드로 폴백**하고, 시간에 따라 게인을 바꾸는 내장 리미터가 **트랜지언트를 찌그러뜨린다.** 요구 게인이 클수록 심해져 −9(+8.5dB) 심함 / −14(+3.5dB) 미묘함과 정확히 일치. 인앱 재생은 loudnorm을 거치지 않아 깨끗했다.

### 수정 — 방향에 따라 도구를 나눈다

- **음량을 낮출 때**(예: −23): 종전 **linear loudnorm 유지** — 이미 정확·투명하다(−23.00 정확, gain swing 0).
- **음량을 올릴 때**: **정적 게인 → look-ahead 리미터 → 측정 → 트림 1회**.
  - `volume=<G>dB, alimiter=limit=<ceil>:attack=5:release=50:level=disabled`
  - 리미터 실링은 목표 TP보다 **0.3 dB 낮게**(`LIMITER_ISP_HEADROOM_DB`) — `alimiter`는 **샘플 피크만** 보고 인터샘플 피크는 못 보므로, 이 여유가 없으면 리샘플 후 −1 dBTP를 넘길 수 있다.
  - 리미팅이 라우드니스를 조금 깎으므로 **측정 후 트림 1회**. 2회 이상은 목표에 더 가까워지지만 **다시 게인을 흔들기 시작**(측정: 3회 반복 시 흔들림 5.66→7.74dB)해서 제거하려던 문제로 돌아간다.

### 측정 결과 (실제 출고 함수 실행)

**gain swing** = 10ms 블록별 (처리후/원본) RMS 비의 5~95 퍼센타일 폭. 0이면 정적 게인(투명), 클수록 게인이 움직임 = 펌핑·왜곡.

| 목표 | 달성(이전→이후) | 목표 대비 | gain swing(이전→이후) |
|---|---|---|---|
| −23 | −23.00 → −23.00 | 0 | 0 → **0** (감쇠, 경로 무변경) |
| −14 | −13.70 → **−14.06** | +0.30 → **−0.06** | 5.64 → **2.30** |
| −9 | **−12.75** → **−9.74** | −3.75 → **−0.74** | **16.92** → **5.66** |

−9에서 게인이 **−1 ~ +15.9 dB로 요동**치던 것이 3배 줄었고, 목표 미달도 3.75 → 0.74 dB로 개선됐다.

### 남는 한계 (버그 아님)

−9 LUFS는 이 소재에서 **−9.74가 한계**다. 크레스트가 큰 믹스를 그 이상 밀면 왜곡이 늘어난다 — 물리적 제약이다. **사용자 결정으로 UI 알림은 넣지 않았으므로**, 목표 미달은 여전히 조용히 발생한다. 확인이 필요하면 `node tools/loudnorm-probe.js <파일.wav>`로 본인 음원의 달성값을 볼 수 있다.

### 검증

## v1.40.3 — Loop-Take 녹음(Repeat ON·Punch OFF)도 트랙 전체를 교체 (사용자 보고) (2026-07-20)

## v1.40.0 — Punch가 원본을 버리지 않고 Take로 보존 (사용자 요청) (2026-07-20)

> **재빌드: renderer([app.jsx](app.jsx) esbuild) + [audio-engine.js](audio-engine.js)(플레인 로드).** **네이티브 무변경.** 사용자 요청(2026-07-20): "punch-repeat로 여러 take를 녹음한 뒤 Take A·B·C 중 하나만 고를 수 있다 — **원본을 선택할 방법이 필요하다**". 사용자 결정: **단일 Punch에도 함께 적용**. 시험 대기 4건([시험.md](시험.md) T-1.40.0-1~4).

### 원인 — punch가 구간의 기존 클립을 삭제하고 있었다

`attachLoopPunchRecording`·`attachPunchRecording` 양쪽에 `if (cS >= inPt && cE <= outPt) continue;  // fully inside → removed`가 있었다. **오디오 자체는 살아 있다** — 소스는 `track.sources`에 등록된 채고 WAV도 `Recordings/`에 있다. 사라진 것은 **그것을 가리키는 클립**뿐이라, 되살리는 데 재렌더가 필요 없다.

### 수정 — 버리지 않고 take로 park

- 신규 **`_sliceActiveAtRegion(track, inPt, outPt)`**: 활성 클립을 구간 기준으로 잘라 `{untouched, outside, inside}`로 돌려준다. 두 punch 함수가 각자 갖고 있던 split/trim 산술을 **한 곳으로 합쳤다**(inactive take는 그대로 통과). 클립에서 잘린 **첫 조각은 원래 id를 재사용**(선택·Undo 안정), 나머지는 새 id. 시작점이 **이동한** 조각만 `recordedStart`/`recordedOffsetMs`를 비운다(Cal. 배지 오해 방지 — deleteRange와 동일 규칙).
- 신규 **`_parkDisplacedAsTakes(track, inside)`**: 밀려난 구간 오디오를 take에 담는다.
  - 이미 **어떤 take에 속해 있었으면 그 take를 유지** → 종전에는 **활성 take가 통째로 소멸**했다(punch를 두 번 하면 첫 punch의 take가 사라졌다). 이제 살아남는다.
  - **base(공유 오디오)에서 왔으면** `kind: "original"` take를 신설.
- **`_normalizeTrackLayout` 라벨링 분리**: `kind === "original"`은 `Original`(2개째부터 `Original 2`…)로 이름을 유지하고 **A/B/C 카운터를 소비하지 않는다** → 녹음 pass는 계속 Take A, B, C. 라이브 녹음 배지(v1.36.1)가 예측하는 letter와 일치.
- **단일 Punch의 take 없는 트랙**: punch 클립이 base(`takeId=null`, 항상 렌더)가 되면 park한 Original과 **구간에서 동시에 소리난다**. 그래서 이 경우 punch 클립에도 take를 만들어 **둘을 정상적인 대안 관계**로 두고 그것을 활성으로 삼는다.
- [app.jsx](app.jsx) `takeBase`(녹음 중 Take 배지) 갱신: 더 이상 활성 take가 소비되지 않으므로 **기존 take 전부가 앞에 남는다**. `kind === "original"`은 letter를 쓰지 않으므로 카운트에서 제외.

- 일반 녹음(Take A) 후 punch-repeat → **Take A(원본) · Take B · Take C…** — Take A를 고르면 원본으로 돌아간다.

## v1.38.0 — Phase 7 Stage 3: 클립 경계 equal-power crossfade (2026-07-20)

> **재빌드: [audio-engine.js](audio-engine.js)(플레인 로드)·[version.js](version.js).** **renderer·네이티브 무변경**(네이티브는 구워진 WAV를 받으므로 자동으로 같은 소리). 사용자 결정(2026-07-20): **고정 10ms**, 설정 UI 없음. 사용자 시험 T-1.38.0-1·2·3 **전부 통과**(2026-07-20) → [시험-아카이브.md](시험-아카이브.md).

### 설계 — 모델 불변, 렌더링만 변경

crossfade에는 경계 **너머의 오디오**가 필요하다. 클립을 트림해도 원본 WAV는 잘리지 않으므로 그 여분(**handle**)이 소스에 남아 있다. 이걸 **굽는 순간에만** 읽어 겹친다 → `clip.start`/`duration`은 **그대로**이므로 **겹침 금지 불변식·Undo 구조·네이티브 동기화가 전부 무영향**이다.

### 경계 3분류 (`_planClipEdges`)

| 경계 | 조건 | 처리 |
|---|---|---|
| **continuous** | 같은 source + offset 이어짐 + gap 없음 (=안 움직인 split) | **페이드 없음** |
| **seam** | 다른 source 또는 offset 불연속 | **equal-power crossfade** (`sin`/`cos`, 10ms) |
| **gap** | 사이가 무음 | 종전 micro-fade |

- **continuous에서 페이드를 없앤 것은 버그 수정이기도 하다** — 나눴지만 움직이지 않은 클립은 오디오가 원래 이어져 있는데도 양쪽에서 4ms씩 볼륨이 꺼졌다 켜지는 **원본에 없던 딥**이 있었다.
- handle이 부족하면 있는 만큼만 짧게(`useTail`+`useHead`), 양쪽 다 없으면 **종전 micro-fade로 폴백** → 회귀 없음.
- fade 길이는 **양쪽 클립 길이의 절반**을 넘지 않게 clamp(짧은 클립이 자기보다 긴 페이드를 먹지 않도록).

### 두 렌더 경로 공용

`_bakeTrackLayout`(재생·Export)와 `_renderClipsToSource`(Merge 렌더)가 **같은 `_planClipEdges` + `_renderDescs`** 를 쓴다 → **편집 중 들리던 소리 그대로 병합된다.** 종전에 각자 페이드를 계산하던 코드는 제거.

⚠️ **`outer` 옵션이 두 경로에서 다르다**: bake는 첫/마지막 가장자리가 타임라인의 **무음과 만나므로 `micro`**, 병합 렌더는 그 가장자리가 **병합 클립 자신의 가장자리**가 되어 bake가 다시 페이드하므로 **`none`**(안 그러면 이중 페이드). 초기 구현에서 이걸 `micro`로 통일했다가 **시뮬레이션이 회귀를 잡았다**(병합 오디오가 0에서 시작·끝남).

### 검증

`node --check` OK, 재빌드 OK, 전역 충돌 신규 0건, Electron 부팅 렌더러 에러 0. 시뮬레이션은 **파일에서 함수를 추출해 실행**하는 방식으로 3종 전부 통과 — crossfade 7케이스 22검사(continuous 평탄·equal-power `sin²+cos²=1`·handle 없음 폴백·head만 있는 경우·gap·짧은 클립 clamp·버퍼 경계), 병합 렌더 8케이스 27검사, 전략 판정 6검사. 실제 청감(경계 클릭/딥 없음)은 T-1.38.0-*(사용자).

### ① 비동기 경합 — `filePath`가 버려진 객체에 기록됨

`joinClips`가 `first.takeId || second.takeId`로 태그를 정해, **base 클립(`takeId=null`, 항상 렌더)을 Take 안으로 흡수**했다. Loop-Punch Comp 구간 경계를 걸쳐 join하면 흡수된 base 오디오가 **Take 전환 시 사라진다**. Consolidate는 거부하는데 Join은 조용히 수행하던 **불일치**(사용자 질문으로 발견). Join에도 같은 레인 가드를 적용 — ⚠️ **기존 동작 변경**.

### 한계

**v1.37.1 이전에 병합해 이미 저장한** 클립은 `filePath`가 null로 굳어 소급 복구 불가 — 다시 병합해야 한다(원본 WAV는 `Consolidated/`에 남아 있다).

- 신규 **`consolidateClips(trackId, clipIds)`**: 선택 clip을 하나로 평탄화. **같은 레인만 허용**. raw 누락 시 `null` 반환하고 **clip을 건드리지 않는다**.

## v1.35.3 — 구간의 PUNCH 표시를 Repeat 버튼과 분리 (사용자 요청) (2026-07-19)

### 로직 확인 (사용자 문의)

## ⚠️ 상시 개발 노트 — 버퍼 인덱싱 이원성 (전략 B, 반드시 숙지)

> Phase 5에서 도입된 **레이아웃 베이크(전략 B)**의 결과로 생긴 **상시 제약**이다. 완료 기록은 아카이브에 있으나, 이 노트는 이후 모든 파형/재생/피크 코드에 계속 적용되므로 여기에 남긴다.

전략 B에서 `track.buffer`는 두 가지 인덱싱 체계 중 하나이며, **`track._layoutBaked` 플래그가 이를 구분한다.** 이 둘을 혼동하면 **이동한 클립이 빈 파형으로 그려지는 버그**가 재발한다(2026-07-14 실제 발생, 수정 완료).

| 상태 | 버퍼 | 오디오 위치 | 파형 샘플링 기준 | 재생 |
|---|---|---|---|---|
| `_layoutBaked = true` (baked) | 평탄화 렌더 결과 | **타임라인 위치**(`clip.start`) | `clip.start` 기준(타임라인) | `_offset`(플레이헤드)로 그냥 재생 |
| `_layoutBaked = false` (raw/trivial) | 원본 디코드 버퍼 | 소스 원점 | `clip.offset` 기준(소스) | 동일(정렬돼 있어 무관) |

- `_ensureBaked`가 bake하면 `_layoutBaked=true`, trivial로 raw 복원하면 `false`로 설정한다.
- [ui-tracks.jsx](ui-tracks.jsx) `Waveform.bucketAt`는 `track._layoutBaked`로 분기해 `bufPos`를 **타임라인(clip.start) vs 소스(clip.offset)** 로 계산한다. peaks 배열도 `_ensureBaked`가 baked 버퍼에서 재계산하므로 타임라인 인덱싱이다.
- **새 파형/재생/피크 코드를 추가할 때**: baked 버퍼를 `clip.offset`로 접근하면 안 된다(그 위치는 침묵). 반드시 `_layoutBaked`를 확인하고 baked면 타임라인 위치로 접근.
- 재접속(`_assignDecodedToTrack`)은 raw 디코드 후 non-trivial이면 `_ensureBaked`를 호출해 재시작 후에도 이 불변식을 복원한다.
- 관련: `_rawBuffers[sourceId]`(원본 보관), `_isTrivialLayout`(bake 스킵 판정), `clip.gain`은 baked 버퍼에 반영·`clip.params.volume`/automation은 합성커브(이중적용 방지).

- **로드 순서**(studio.html): `ui-kit` → `ui-tracks` → `ui-mixer` → … → `app`. 그래서 app.jsx는 ui-tracks의 전역을 그대로 읽을 수 있다(의도적으로 쓰는 경우 주석으로 명시할 것).

## ⚠️ 상시 개발 노트 — Undo 스냅샷 정합성 (반드시 숙지)

> 2026-07-17 세션에서 이 규칙 위반으로 버그 3건(v1.23.2 · v1.23.3)이 나왔다. 새 편집 UI를 붙일 때마다 적용된다.

판정 기준은 하나 — **`getSnapshot()`이 그 상태를 담는가?**

| 상태 | 스냅샷 | `onBeforeChange`/`pushUndo` |
|---|---|---|
| `master`(volume·reverb·echo·fadeIn·fadeOut·room·roomParams·saturation·widener·exciter), `eqBands`, `tempo`, `duration`, 트랙 `params`·`clips`·`sources`·`takes` | **담음** | **필수** — 없으면 그 기능은 Undo 불가(v1.23.3 fade) |
| 플레이헤드, `loopRange`, `repeatPlayEnabled`, `masterFxBypassed` | **안 담음** | **금지** — 100% no-op 항목이 되어 **다음 Ctrl+Z를 먹고 Redo까지 지운다**(v1.23.2 OUTPUT FX / v1.23.3 Repeat 버튼) |

- **드래그 핸들러는 mousedown이 아니라 "첫 실제 변경 직전"에 1회만** 스냅샷(클릭만 했을 때 no-op 항목이 생기지 않게). 값 비교 후 같으면 조기 반환하는 것이 정석 — 각 컴포넌트의 **휠 핸들러가 원래부터 `if (nv === value) return;`으로 이 패턴**을 쓴다. 단 `Fader`·`SleekSlider`는 mousedown에서 클릭 위치로 값을 점프시키므로 그 자리의 스냅샷이 정당하다.
- **편집이 아무것도 바꾸지 않았으면 `cancelUndo(token)`** 으로 되돌린다. `pushUndo()`는 **redo 스택을 비우므로** `undoStack.pop()`만 하면 **Redo가 조용히 죽는다**(v1.23.1).
- **가드**: `undo()`가 복원 전후 `snapshotSig`를 비교해 같으면 `console.warn("[undo] no-op entry ...")`. 정상 동작 시 무출력이므로, **이 경고가 보이면 위 규칙을 어긴 핸들러가 있다는 뜻.**
