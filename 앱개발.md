## [Phase 5] Audio In 클립 편집 — 구현 계획 (전략 B 확정) — 2026-07-14

> 착수 전 코드 조사로 확인한 재생 구조 제약과 채택한 아키텍처, 단계별 구현 계획.

### 조사 결론 — 현재 재생 구조의 제약

- 웹 재생·오프라인 렌더(Export)·네이티브 모두 **"트랙당 버퍼 1개 + 합성 게인 커브"** 모델.
  - 웹: 트랙마다 `BufferSourceNode` 하나가 `t.buffer` 전체를 재생([audio-engine.js](audio-engine.js) `play`), `_sourceOffsetForTrack`가 `offset % duration`로 **버퍼 위치 = 타임라인 위치**를 전제.
  - 클립 gap/볼륨은 오디오를 자르지 않고 `autoGain`에 `_buildCompositeCurve`로 게인 커브를 씌워 표현.
  - 네이티브는 **트랙당 클립 1개**(`track.clips[0]`)에 `startSeconds` lead-in만 지원([audio-bridge.js](audio-bridge.js)) — 다중/이동 클립 표현 없음.
- **결론**: 이 모델은 `버퍼시간 == 타임라인시간`일 때만 성립. `moveClip`은 이 불변식을 깨므로 재생 경로를 바꿔야 실제로 소리가 옮겨진다.

### 채택 아키텍처 — 전략 B: 레이아웃 베이크 (2026-07-14 확정)

- `track.clips[]`가 **비파괴 진실원본**. 클립 편집 시 `clips[]`를 **평탄화한 새 `t.buffer`를 즉석 렌더(bake)** — 각 클립 소스를 타임라인 위치에 배치 + 경계 3~5ms micro fade + `clip.gain` 반영.
- 그 결과 `버퍼시간 == 타임라인시간` 불변식이 항상 성립 → **기존 단일소스 재생/Export/네이티브 loadTrack 경로를 그대로 재사용**. (보너스: start>0에 놓인 녹음의 웹 재생 오정렬도 교정.)
- 원본 소스 오디오는 `track._rawBuffers[sourceId]`에 보관(첫 편집 직전 캡처), 재베이크는 항상 원본에서 수행.
- `clip.gain`은 baker가 버퍼에 반영(트랙 volume과 분리), `clip.params.volume`·automation은 기존 합성 커브가 계속 담당(이중적용 없음, 필드 분리).
- **trivial layout**(단일 클립·start≈0·offset≈0·gain 1·full 길이·params/automation 없음)은 bake 스킵 → 기존 프로젝트 무회귀.
- 채택 근거: Q5 "오프라인 렌더 방식 권장" 철학과 정합, 이중 엔진 동기화(앱 최대 취약부)를 건드리지 않아 **저위험**. (로드맵 §5의 clip별 스케줄러(전략 A)보다 위험 낮음.)

### Stage 3 사용자 시험 버그 수정 (2026-07-14)

사용자 클립 이동 시험에서 2건 발견·수정:

- **문제1 — 클립을 뒤로 옮기면 파형이 사라지고 앞으로 다시 못 옮김**: 근본 원인은 `Waveform`이 baked(타임라인 인덱싱) 버퍼를 `clip.offset`(소스 인덱싱)로 샘플링해, 이동한 클립이 버퍼 앞쪽 침묵 구간을 그려 **빈 파형**이 되던 것. 수정: `_ensureBaked`가 `track._layoutBaked` 플래그 설정 → [ui-tracks.jsx](ui-tracks.jsx) `Waveform.bucketAt`가 baked면 **타임라인 위치로 샘플링**. 더불어 `_ensureBaked`의 duration을 `Math.max` 누적에서 **정확 재계산(`_projectClipDuration()`)**으로 바꿔 클립을 앞으로 옮기면 ruler가 **다시 줄어들도록** 함.
- **문제2 — 재시작 후 이동한 Audio In 파형이 안 보이고 ruler가 초기값**: 재접속 경로 `_assignDecodedToTrack`가 raw 디코드 버퍼만 넣고 레이아웃을 bake하지 않아, 복원된 이동 클립이 위치/파형/duration에 반영 안 됨. 수정: 재접속 시 non-trivial 레이아웃이면 `_ensureBaked` 호출(파일/무편집 트랙은 raw 유지). 녹음은 프로젝트 폴더에 `.wav`로 영속(`prepareRecordingPath`)되어 재접속으로 복원됨.
- 검증: 실제 `_ensureBaked` 대상 Node 테스트 8/8(이동 시 baked 오디오가 타임라인 위치에 존재·구위치 침묵, `_layoutBaked` 토글, duration 확장/축소), 렌더러 빌드 통과, Electron 기동 스모크 무에러.

**추가 보완 — 클립 이동 시 타임 스케일 고정 (2026-07-14)**: 사용자 요청. 클립을 뒤로 옮겨 곡 길이가 늘면 fit-to-view 모드가 `pxPerSec`를 재계산해 화면이 축소(zoom out)되던 것 → 클립 편집 핸들러(`handleMoveClip`/`trimStart`/`trimEnd`/`deleteClip`/`duplicateClip`/`pasteClip`)에서 `fitTimelineRef.current=false`로 **fit 모드 해제**. 이제 편집으로 곡이 길어져도 줌이 고정되고 레인만 넓어져 가로 스크롤로 처리(파형 스케일 유지). 최소 줌까지 축소하면 fit 모드 재활성(기존 동작).

### Q1. 클립 편집에 crossfade 구현 계획이 있는가?

- **현황(코드)**: 클립 편집 crossfade **미구현**. 문서상 계획만 존재. (검색에 잡히는 crossfade는 SoundTouch 타임스트레치 내부 overlap crossfade로 별개 DSP.)
- **계획상 위치**: crossfade는 **개발 Phase 7(Comp/Consolidate)** 소속 — "Comp 구간 경계에 equal-power crossfade 자동 생성"([제안.md](제안.md) §6 "Take 선택과 Comp", [개발-프로세스.md](개발-프로세스.md) Phase 7). **개발 Phase 5에는 crossfade 없음**, 대신 클립 경계 **micro fade 3~5ms**(click/pop 방지용 fade in/out, crossfade 아님)만 있음([개발-프로세스.md](개발-프로세스.md) Phase 5).
- **제약**: [제안.md](제안.md) §6 시간축 이동은 **v1에서 같은 트랙 내 클립 겹침 금지**(겹친 녹음은 Take Lane으로 표현)로 규정 → 일반 DAW식 "겹친 두 클립 수동 crossfade" 편집은 현재 로드맵에 없음.
- **결정 지점**: (a) 로드맵 유지 = Phase 5는 micro fade만, crossfade는 Phase 7 Comp에서 처리(권장, 겹침 금지 규칙과 정합) / (b) Phase 5에 수동 클립 crossfade(겹침 허용) 추가 = 겹침 금지·Take Lane 규칙과 충돌하므로 데이터 모델·재생 엔진 설계 변경 필요.
- ✅ **결정(2026-07-14)**: **(a) 로드맵 유지**. Phase 5는 micro fade만, crossfade는 Phase 7 Comp.

### Q2. 음정 튜닝 기능이 있는가? (일반 기능 문의)

- **① 전체 키 변경(트랜스포즈) → 있음**: **Vari Key** — Key 자동 감지(`detectedKey`) 후 반음 단위 오프셋으로 전곡 재생 피치 이동([audio-engine.js](audio-engine.js) `keyShift`/`setKey`/`setKeyShift`, `variKey`). SoundTouch RateTransposer로 재생·Export 반영. 관련: Vari BPM(템포), Keep pitch/preservePitch(템포만 변경·피치 유지). Key 설정창 개편은 v1.10.x.
- **② 오토튠식 음정 보정(피치 커렉션) → 없음**: per-note auto-tune/Melodyne류 기능은 코드에 없고 로드맵([제안.md](제안.md)·[개발-프로세스.md](개발-프로세스.md))에도 없음(로드맵은 파일 트랙·Audio In 녹음·클립·Take·Comp 범위). SoundTouch는 신호 전체 균일 transpose만 하며 음 단위 검출·재합성은 미지원. 추가 시 로드맵 밖 신규 기능(음정 검출 + PSOLA/phase vocoder 등 재합성)으로 규모 큼.
- **정리**: 키/피치 트랜스포즈(Vari Key)는 있음, 음정 보정(auto-tune)은 없음.

### Q3. clip 복사/붙여넣기 기능이 있는가?

- **현황**: 없음(코드·로드맵 모두). 현재 클립 편집은 split/join(`handleSplit`/`handleJoin`)·clip volume 기초뿐([app.jsx](app.jsx)). [제안.md](제안.md)·[개발-프로세스.md](개발-프로세스.md)에 clip copy/paste 언급 없음(문서의 "복사"=Save As 파일 복사, "복제"=피해야 할 버퍼 복제로 다른 맥락). Phase 5 명시 범위(이동·trim·delete·deleteRange·clip gain·micro fade)에도 copy/paste 없음.
- **추가 난이도 낮음**: Phase 1 source/clip 모델 덕분에 clip 복사 = **같은 `sourceId` 참조하는 새 clip을 다른 `start`에 배치**(오디오 복제 없이 비파괴). 붙여넣기 = playhead/대상 위치 삽입. Phase 5에 이동/trim/delete와 함께 얹기 좋은 후보.
- **결정 지점**: Phase 5 범위에 **clip 복사/붙여넣기(+duplicate) 추가 여부**.
- ✅ **결정(2026-07-14)**: **Phase 5 범위에 포함**. 같은 `sourceId` 참조 새 clip 배치(비파괴 복사) + playhead/대상 위치 붙여넣기 + duplicate.

### Q4. Punch Recording(원하는 구간만 다시 녹음)이 계획에 있는가?

- **현황**: 없음(코드·로드맵 모두 별도 기능 미명시). 현재 녹음은 playhead→곡 끝(또는 수동 정지)까지 기록. ([app.jsx](app.jsx)의 "punch"는 BPM 표시 애니메이션으로 무관. 문서의 "재녹음"은 트랙 전체 교체 맥락이지 구간 punch 아님.)
- **기반 요소 존재**: ① 구간 `loopRange` 이미 있음(구간 반복 재생용). ② 개발 Phase 6(루프 Take 녹음)의 `startRecording({ trackId, start, end, loopEnabled, countIn })`에 구간(start/end) 파라미터가 계획됨(단 회차별 Take/루프용이지 "구간 교체" punch와는 목적 다름). ③ Phase 5 클립 모델(비파괴 배치/trim)로 구간 클립 교체가 용이.
- **추가 시 형태**: pre-roll 재생 → in 지점 자동 punch-in → out 지점 자동 punch-out → 해당 구간 클립만 배치/교체(비파괴). Phase 4/5/6 연계 신규 항목.
- **결정 지점**: Punch Recording을 신규 로드맵 항목으로 추가할지, 추가 시 어느 Phase에.
- ✅ **결정(2026-07-14) — 개발 Phase 6에 편입**. 근거: Punch가 필요로 하는 무거운 인프라(구간 지정 녹음 `startRecording({start,end})`, count-in/pre-roll, 녹음 상태머신 `idle→armed→countIn→recording→…`, sample-accurate 경계 분할)를 **Phase 6가 전부 생성**하며, Punch는 그 위에 "회차별 Take 누적" 대신 "**단일 구간 클립 교체 + 자동 punch-in/out**"만 바꿔 얹는 Phase 6 변형이기 때문. Phase 5에 넣으면 녹음 인프라를 앞당겨야 해 비대·순서 꼬임, Phase 7 뒤로 미루면 저비용·고가치 기능 지연. → Phase 6 루프 Take 녹음 안정화 후 같은 인프라 위에서 마무리.
- **Phase 5 훅(선행 준비)**: Phase 5의 `deleteRange` + 클립 삽입을 "구간을 비우고 그 자리에 새 clip을 배치"가 깨끗하게 되도록 설계 → Phase 6에서 Punch를 붙일 때 재생 엔진 변경 없이 재사용.

### Q5. Auto-tune 설계 방향(별도 창 + 피아노 롤로 구간별 음 올림/내림) 검토 — Q2 연장

사용자 구상: 추후 **별도 window**에 **피아노 롤**을 배치하고, 그에 맞춰 **특정 구간의 음을 올리거나 내리는** 수동 피치 편집(Melodyne류 그래픽 보정). 실현 가능성 평가:

- **구조·UI·아키텍처 → 무리 없음**: 독립 Electron 창 + BroadcastChannel 동기화 패턴 확립됨([mixer-app.jsx](mixer-app.jsx)·[advanced-pan-app.jsx](advanced-pan-app.jsx)·[advanced-eq-app.jsx](advanced-eq-app.jsx)·[advanced-ambience-app.jsx](advanced-ambience-app.jsx)). 비파괴 편집은 Phase 1 source/clip 모델에 "피치 편집 메타데이터 + 렌더 캐시 source"로 정합.
- **핵심 리스크(DSP·범위)**:
  1. **모노포닉 한정 권장**(보컬/리드/베이스). 폴리포닉은 연구급 난이도로 비권장.
  2. **피치 검출(f0 tracking)** 선행 필요(YIN/pYIN 등) — 피아노 롤에 현재 음 표시용.
  3. **피치 시프트 품질**이 최대 리스크 — SoundTouch는 전체 균일 transpose용이라 보컬 구간 보정 품질(포먼트 보존) 부족. Melodyne급은 포먼트 보존 PSOLA/phase-vocoder 또는 전용 라이브러리 필요.
  4. **오프라인 렌더 방식 권장**("편집 → 새 WAV 렌더 → 클립 교체") — 웹/네이티브 두 엔진은 결과 WAV만 재생하므로 [작업원칙 6](개발-프로세스.md) 충돌 원천 회피. 실시간 시변 피치의 이중 엔진 동기화는 부담·위험 큼.
- **규모·위치**: 로드맵 밖 **대형 신규 모듈**. Phase 5(클립)·Phase 7(Comp) 이후 또는 독립 진행. "별도 창 + 피아노 롤 + 오프라인 렌더 + 모노포닉 한정"이면 단계적 구현 가능. 관건 = 피치 시프트 품질(알고리즘/라이브러리) + 모노포닉 범위 확정.
- **결론**: 구조적으로 무리 없음. 착수 전 결정 필요 = ① 모노포닉 범위, ② 피치 엔진(라이브러리/알고리즘), ③ 오프라인 렌더 확정.
- ⏸ **결정(2026-07-14) — 보류**. 로드맵 밖 대형 신규 모듈로, Phase 5/7 이후 또는 독립 진행 시 재검토(①~③ 확정 선행).
