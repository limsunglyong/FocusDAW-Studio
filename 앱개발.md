## v1.20.6 입력 리미터 게인 리덕션(GR) 미터 + 믹서 입력 미터 정렬 — 2026-07-13

> 상태: 구현 완료 · renderer/네이티브 재빌드 완료 · `bin/` 배포 완료 — 사용자 시험 대기. (라이브 입력 신호가 있어야 GR 채움이 보이므로 UI는 값 주입 렌더로 검증)

### 배경

- [개발-프로세스.md](개발-프로세스.md) Phase 4 스펙의 "입력 미터와 limiter gain reduction 미터 추가"에서 **입력 레벨 미터만 구현되고 GR(게인 리덕션) 미터는 누락**돼 있었다. 리미터(LIM, 기본 On, ceiling −1.0 dBFS)가 실제로 신호를 얼마나 누르는지 화면으로 알 수 없어 입력 게인을 맞추기 어려웠다. 이를 메인 트랙 헤더와 믹서 양쪽에 추가했다.
- 이 세션에서 Phase 4 전체를 코드와 대조 점검했다(상세: 문서 상단 **[Phase 4 점검]** 섹션). 핵심 파이프라인은 모두 반영돼 있었고, 미구현/보류 4건 중 **GR 미터를 본 버전에서 구현**했다.

### 구현 내용

1. **네이티브 GR 계산** ([AudioEngine.cpp](juce_skeleton/Source/AudioEngine.cpp)·[.h](juce_skeleton/Source/AudioEngine.h)): tanh 소프트 리미터(`ceiling * tanh(x/ceiling)`)의 실제 감쇠량을 dB로 산출한다. 파일 스코프 `kInputLimiterCeiling`(0.89125 = −1.0 dBFS)과 `limiterGrDb(prePeak)` 헬퍼로 통일. 콜백에서 **녹음 중=실측 prePeak**, **ARM만(idle)=예측(idlePeak)** 기준으로 `gainReduction` atomic을 레벨과 동일한 fast-attack/slow-release(×0.82) 방식으로 갱신. `InputRecorder::getGainReduction()`·`AudioEngine::getInputGainReduction()` 노출. `levels` 메시지에 `inputGr` 추가([WebSocketServer.cpp](juce_skeleton/Source/WebSocketServer.cpp)). 네이티브 재빌드(`juce_skeleton/build` 증분) 후 `bin/`에 배포.
2. **브리지** ([audio-bridge.js](audio-bridge.js)): `nativeState.inputGr`, `getInputGainReduction()`, `levels` 파싱 추가.
3. **메인 트랙 헤더 GR 미터** ([app.jsx](app.jsx)·[ui-tracks.jsx](ui-tracks.jsx)): 입력 게인 슬라이더의 레벨 미터 아래에 **오른쪽에서 채워지는 붉은 GR 막대(0~12 dB)** 를 추가. `inputGr` prop 배선, LIM 켜짐 + ARM일 때만 표시.
4. **믹서 GR 미터** ([ui-mixer.jsx](ui-mixer.jsx)·[mixer-app.jsx](mixer-app.jsx)·[app.jsx](app.jsx)): 믹서는 별도 창(BroadcastChannel)이라 `LEVEL_METERS` 메시지에 `inputGr`를 실어 보내고, 믹서 프록시에 `_inputGr`/`getInputGainReduction()` 수신을 추가. `InputGrMeter` 컴포넌트(세로 LED, **위→아래로 하강**, 앰버→레드)를 IN 미터 오른쪽에 배치. **채널 폭 138px를 유지**하기 위해 입력 게인 노브를 80→64px로 축소.
5. **믹서 입력 미터 정렬(후속 수정)** ([ui-mixer.jsx](ui-mixer.jsx)): GAIN dB 값의 자릿수(`+12.0`~`−20.0 dB`)에 따라 중앙 정렬 열 폭이 변해 IN/GR 미터가 좌우로 밀리던 문제를 수정. IN/GR을 한 그룹으로 **오른쪽 고정**하고, GAIN 표시를 **2행(라벨/값)** 으로 바꿔 값이 노브 폭(64px)을 넘지 않게 함. 노브는 미터 왼쪽 공간의 중앙으로 정렬. 실측: `+12.0`↔`−20.0 dB`에서 IN left·GR right 좌표 불변 확인.
6. **매뉴얼** ([ui-help.jsx](ui-help.jsx)): 섹션 4 LIM 설명에 GR 미터 사용법(레벨 미터 아래 붉은 막대, GR이 자주 크게 움직이면 게인 낮추기)을 한/영 추가.

- Exclusive로 열 수 없는 장치를 고르면 런타임은 롤백으로 복구되지만, **선택이 그대로 localStorage에 저장**됐다(`applyDeviceSetup`이 성공/실패와 무관하게 persist).

1. **실패한 선택을 저장하지 않음** ([audio-bridge.js](audio-bridge.js) `applyDeviceSetup`): 적용 전 이전 저장값을 백업하고, `setAudioInput`이 실패하면 **이전 정상값으로 복원**한다. → 잘못된 Exclusive 설정이 애초에 저장되지 않음(신규 재발 차단).

3. **장치 목록 조회 견고화** ([AudioEngine.cpp](juce_skeleton/Source/AudioEngine.cpp) `getAudioDevicesJson`): 타입별 `scanForDevices()`/`getDeviceNames()`를 `try/catch`로 개별 보호해, 한 타입(Exclusive)이 예외를 던져도 **나머지 타입 목록은 항상 반환**한다.

## v1.19.1 Exclusive 장치 오픈 실패 복구 + 오류 표면화 — 2026-07-11

> 상태: 구현 및 사용자 시험 완료 — B-233-2로 [시험-아카이브.md](시험-아카이브.md)에 이관. (v1.19.0 후속: "exclusive 선택 시 audio device fail, 출력·녹음 모두 안 됨")

### 확인된 원인

- v1.19.0에서 Exclusive 모드는 입력·출력을 같은 인터페이스로 한 번에 설정하는데, **WASAPI Exclusive는 setup의 sample rate/buffer/채널 마스크를 장치가 정확히 지원하지 않으면 오픈을 거부**한다. shared 드라이버에서 넘어온 rate나 사용자가 고른 rate가 exclusive에서 미지원이면 `setAudioDeviceSetup`이 실패했다.
- 게다가 실패 시 이전 정상 장치를 복구하지 않아 **오디오가 완전히 죽고(출력·녹음 모두 불가)**, 실제 오류 메시지도 UI에 노출되지 않았다.

### 구현 내용

1. **네이티브 setAudioInput 재시도 + 롤백** ([AudioEngine.cpp](juce_skeleton/Source/AudioEngine.cpp)):
   - 타입 전환 직전에 이전 device type·setup을 스냅샷.
   - `setAudioDeviceSetup` 실패 시 ① rate/buffer를 0(장치 기본)으로, ② 그래도 실패하면 입력 채널 마스크를 기본으로 완화해 **단계적으로 재시도**.
   - 그래도 실패하거나 입력 채널이 없으면 **이전 device type·setup으로 롤백**해 오디오가 죽지 않게 하고, JUCE의 실제 오류 사유를 담아 `"Could not open <type> device: <reason>"`을 반환.
2. **UI 오류 표면화** ([ui-dialogs.jsx](ui-dialogs.jsx)): `applyDeviceSetup` 실패 시 오류 메시지를 Audio Devices 섹션에 적색 배너로 표시("previous device was kept" 안내 포함). 성공 시 자동 해제.
3. renderer + 네이티브 `build_native.bat` 재빌드 완료.

> 참고: 이 변경으로 Exclusive가 특정 장치에서 여전히 실패할 수 있으나(장치가 그 조합을 지원하지 않는 경우), 최소한 오디오가 죽지 않고 **실패 사유가 화면에 표시**된다. 표시되는 사유를 알려주면 해당 장치에 맞춰 추가 대응한다.

3. **원자적 적용 (네이티브)**: `setAudioInput`에 `outputName` 파라미터를 추가해 **타입 전환 시 입력·출력을 한 번의 setup으로 동시 지정**한다([AudioEngine.cpp](juce_skeleton/Source/AudioEngine.cpp)·[.h](juce_skeleton/Source/AudioEngine.h)·[WebSocketServer.cpp](juce_skeleton/Source/WebSocketServer.cpp)). 출력이 stale 기본값으로 튀는 근본 버그를 제거. `outputName`이 비면 기존 출력 보존(무회귀).
4. **브리지 (audio-bridge.js)**: 통합 진입점 `applyDeviceSetup({type,inputName,outputName,sampleRate,bufferSize})` 추가. `setAudioInput`은 같은 모드의 저장 출력을 `outputName`으로 자동 동봉하고, 재연결 시에도 입력+출력을 **하나의 명령**으로 재적용한다(타입 불일치면 출력 미동봉으로 폴백). 입력 없이 출력만 바꿀 때는 `setAudioDevice`로 처리하고 stale 입력 저장을 제거.

## v1.18.11 녹음 진행바/플레이바 진행율 불일치 수정 (입력·출력 SR 차이) — 2026-07-11

> 상태: 구현 및 사용자 시험 완료 — B-228은 [시험-아카이브.md](시험-아카이브.md)로 이관. (수정요청.md 1번)

### 확인된 원인

- 녹음 중 라이브 진행바(빨간 리딩 엣지)는 `녹음샘플수 ÷ 보고된 sampleRate`로 위치를 계산한다([ui-tracks.jsx](ui-tracks.jsx) `drawLiveRecording`, [audio-bridge.js](audio-bridge.js) `recordingPeaks`).
- 그러나 `recordingPeaks` 메시지가 보고하던 sampleRate는 **init 시 출력 장치에서 캐시된 `engine.sampleRate` 멤버**였다([WebSocketServer.cpp](juce_skeleton/Source/WebSocketServer.cpp) `getCurrentSampleRate()`). `setAudioInput()`은 이 값을 갱신하지 않는데, 실제 녹음 WAV와 `endSample` 카운트는 **실제 장치 SR(`dev->getCurrentSampleRate()`)**로 이뤄진다([AudioEngine.cpp](juce_skeleton/Source/AudioEngine.cpp) `startRecording`).
- 따라서 **입력 장치 SR ≠ 초기 출력 SR**이면 라이브 진행바의 시간축이 일정 비율로 어긋난다. 캐시 SR이 실제보다 크면 진행바가 실시간보다 느리게 가고(동시 재생 시 플레이바가 더 빨라 보임), 정지 시 실제 WAV(정확한 SR)로 클립이 붙으며 파형이 확 늘어난다. (peak ring은 4096칸이라 드롭 문제는 아님.)

### 구현 내용

1. `InputRecorder`가 take 시작 시 받은 실제 녹음 SR(= WAV 헤더 rate)을 atomic 멤버 `recordingSampleRate`로 보관하고 `getRecordingSampleRate()`로 노출한다.
2. `AudioEngine::getRecordingSampleRate()`를 추가해 녹음 중에는 recorder의 SR, idle 시에는 캐시 엔진 SR을 반환한다.
3. `recordingPeaks` 메시지가 `getCurrentSampleRate()` 대신 **`getRecordingSampleRate()`**를 보고하도록 변경했다. → 라이브 진행바 SR == 최종 WAV SR이 보장되어 정지 시 늘어남/진행율 불일치가 사라진다.
4. (하드닝) `setAudioInput()`에서 입력 장치 (재)구성 후 `engine.sampleRate`를 활성 장치 SR로 갱신해, 레벨 프레임·export 기본 SR 등 다른 캐시 소비처도 최신값을 읽도록 했다.
5. 입력 SR == 출력 SR인 기존 환경은 값이 동일해 무회귀. **네이티브 엔진 변경이므로 `build_native.bat` 재빌드가 필요하다.** renderer 변경은 없다.

## v1.18.6 Audio In 짧은 트랙 반복 수정 · 믹서 색 구분 · 마스터 볼륨 테이퍼 — 2026-07-09

### 1. Audio In 짧은 트랙이 1초마다 반복되던 문제 (native)

- **원인:** 네이티브 엔진은 각 트랙(`TrackAudioSource`)을 **자기 버퍼 길이 단위로 루프**한다([AudioEngine.h `setLooping`](juce_skeleton/Source/AudioEngine.h), [AudioEngine.cpp](juce_skeleton/Source/AudioEngine.cpp) `trackSource->setLooping(loopEnabled)`). 파일 스템은 모두 같은 곡 길이라 문제가 없었지만, 1초짜리 녹음 트랙이 섞이면 그 트랙만 1초마다 되감겼다. 또한 네이티브는 클립 시작 위치(녹음을 t>0에서 시작)를 반영하지 않고 항상 0부터 재생했다.
- **수정:** `loadTrack` 명령에 `startSeconds`(클립 시작)·`songLength`(곡 길이) 파라미터를 추가하고, 네이티브 디코드 직후 버퍼를 **[앞 침묵=시작위치] + 오디오 + [뒤 침묵=곡 길이까지]** 로 패딩한다. 그러면 버퍼 길이가 곡 길이와 같아져 per-track 루프가 곡 경계에서 맞아떨어지고, 녹음 시작 위치도 올바르게 반영된다. 50ms 허용오차로 **전체 길이 스템(start=0, len≈곡)은 패딩하지 않아** 기존 재생 동작은 무회귀.
- 대상: [audio-bridge.js](audio-bridge.js)(`trackTimelinePlacement`·loadTrack 3곳), [WebSocketServer.cpp](juce_skeleton/Source/WebSocketServer.cpp), [AudioEngine.h](juce_skeleton/Source/AudioEngine.h)(LoadJob·시그니처), [AudioEngine.cpp](juce_skeleton/Source/AudioEngine.cpp)(패딩). **네이티브 재빌드 필요.**

### 2. 믹서 창 Audio In 채널 배경색 구분

- **원인:** 믹서 채널 스트립([ui-mixer.jsx `ChannelStrip`](ui-mixer.jsx))은 배경이 solo 틴트/transparent뿐이라 파일 트랙과 Audio In 트랙이 구분되지 않았다.
- **수정:** `track.kind === "audioIn"`일 때 편곡 화면 헤더(`--audio-in-track-bg`)와 동일한 **블루 틴트 배경**을 적용(믹서 테마별 `--blue` 변수 사용). renderer 재빌드만 필요.

### 3. 마스터 출력 볼륨 조절감 (dB 테이퍼)

- **원인:** `Fader`가 **위치를 게인(선형)에 직접 매핑**([ui-kit.jsx](ui-kit.jsx))해, 페이더 위쪽 절반이 0~−6 dB만 담당하고 실제 감쇠가 바닥 근처에 몰렸다. 그래서 상단 조작은 체감이 거의 없고 바닥에서만 급히 작아졌다.
- **수정:** 페이더 위치를 **dB(로그) 테이퍼**로 매핑한다. 위치 p∈[0,1] → dB∈[−60, maxDb](maxDb=20log10(max): max=1→0 dB, max=2→+6 dB), p=0은 −∞(무음). 게인 = 10^(dB/20). 이동량 대비 체감 음량이 고르게 변한다. `faderPosToGain`/`faderGainToPos` 헬퍼 추가, 0 dB 마커 위치도 테이퍼 기준으로 재계산. 마스터 VOL과 채널 볼륨 페이더 공통 적용(엔진 값 의미는 그대로라 무회귀). renderer 재빌드만 필요.

## v1.18.4 IN 리미터 실링 수식 수정 — 2026-07-09

> 상태: 구현 및 사용자 시험 완료 — B-223은 [시험-아카이브.md](시험-아카이브.md)로 이관.

### 확인된 원인

- Audio In 트랙 LIM(리미터)은 배선상 정상 적용되고 있었다. UI(LIM 버튼) → `startRecording`의 `limiter` 플래그 → 네이티브 입력 콜백([juce_skeleton/Source/AudioEngine.cpp](juce_skeleton/Source/AudioEngine.cpp) `InputRecorder::audioDeviceIOCallbackWithContext`)에서 샘플 단위로 곱해진다.
- 그러나 기존 수식 `tanh(x*1.18)/tanh(1.18)*0.89125`는 툴팁이 표방하는 "ceiling -1.0 dBFS"를 지키지 못했다.
  - `/ tanh(1.18)` 정규화로 원점 기울기가 1.271이 되어 저레벨 신호를 약 **+2.1 dB 부스트**(투명하지 않음)했다.
  - 점근선이 `0.89125/tanh(1.18) = 1.0771`, 즉 **+0.645 dBFS**여서 IN 게인을 올려 입력이 0 dBFS를 넘으면 출력이 실링은 물론 0 dBFS도 초과했다(실질적 브릭월 보호 부재).

### 구현 내용

1. 리미터 수식을 실링에 점근하는 표준형 `ceiling * tanh(x / ceiling)`(ceiling = `0.89125` = -1.0 dBFS)으로 교체했다.
   - 저레벨에서 기울기 ≈ 1(부스트 없음, 투명), 입력이 아무리 커도 출력이 -1.0 dBFS를 절대 넘지 않는다.
2. 상수 `ceiling`을 채널 루프 밖으로 hoist하고 의도를 주석으로 남겼다.
3. 네이티브 엔진 변경이므로 `build_native.bat`로 재빌드가 필요하다. renderer 변경은 없다.

### 확인된 문제

### 최근 수정 및 반영 사항 (v1.17.17)
- **Audio In 개발 Phase 3 시작 — Merge Tracks / Bounce 1차 구현**: 선택된 파일/Bounce 트랙 2개 이상을 WebAudio 오프라인 렌더로 합쳐 새 `kind: "bounce"` 트랙을 생성하는 `mergeTracks(trackIds, options)` 및 `addBounceTrack()` 흐름을 추가했다. 기존 `joinClips()`와 분리된 트랙 단위 Bounce 동작이며, 항상 전체 프로젝트 범위를 기준으로 Stereo/Mono, 원본 처리(`Keep + Mute` 기본, Keep, Delete)를 모달에서 선택할 수 있다.
- **Merge Tracks 실행 흐름 단순화**: `Loop Range` 옵션을 제거하고 `Create Bounce`를 누르면 Merge Tracks 모달을 즉시 닫은 뒤 Rendering 로딩 화면으로 전환되도록 했다.
- **Merge Tracks 이름 편집 및 액션 영역 구분**: Merge Tracks 모달에 기본 Bounce 트랙 이름이 채워진 `Track Name` 입력칸을 추가했다. 사용자가 입력한 이름은 생성되는 Bounce 트랙 이름과 WAV 파일명에 반영되며, `Cancel`/`Create Bounce` 버튼 영역 위에는 수평 구분선을 추가했다.
- **Merge Tracks 비활성 커서 정리**: 파일 트랙 선택이 없거나 1개만 선택되어 Merge 실행 조건을 만족하지 않을 때 금지 커서가 뜨지 않도록 하고, 기존 안내 말풍선만 유지한다.
- **Bounce 파일 저장 경로 추가**: `.focus` 프로젝트를 열었는지 여부와 관계없이 결과 WAV를 첫 번째 선택 트랙의 원본 오디오 파일 폴더 아래 `Bounces` 폴더에 저장하고, 저장된 파일 경로를 Bounce 트랙 source에 연결한다. 원본 파일 경로가 없는 예외적인 트랙에서는 프로젝트 경로를 fallback으로 사용한다.

  - 증상: **재생 중** 플레이바보다 **뒤쪽(이후 시점)** 에 반복 구간을 (재)설정하면, 화면 플레이바는 구간으로 들어온 것처럼 보이지만 **실제 오디오는 구간으로 진입하지 않고** 선형 재생됨. (정지 상태, 구간이 플레이바보다 앞쪽인 경우는 정상.)
  - 원인: 반복 유지 로직이 **끝(end) 경계 하나로만** 설계됨 — `handleNativeMessage`가 `offset >= loopRange.end`일 때만 `seek(start)`로 되돌림. 그래서 플레이헤드가 **구간보다 앞(offset < start)** 인 상태(=구간을 뒤쪽에 설정)는 교정할 로직이 없음. 게다가 네이티브 `getPlayhead()`는 repeat ON일 때 반환값을 `[start,end]`로 **clamp** 하여 seek 없이도 화면 플레이바만 구간 안으로 들어온 듯 보이고 오디오는 밖에서 선형 재생됨(시각/오디오 괴리). 이 wrap 로직·반복 기능은 v1.14.15에 도입된 것으로 **마우스 휠 줌(v1.15.0)과 무관**함을 git 이력으로 확인.
  - **1차 (v1.15.1) — `setRepeatPlayEnabled` read-before-set**: 위 clamp 때문에 `setRepeatPlayEnabled`가 플래그를 켠 뒤 getPlayhead를 읽으면 seek 판정이 무력화됨. → **플래그를 켜기 전에** 참(clamp 전) 플레이헤드를 읽어 구간 밖이면 `seek(loopRange.start)`. native/web 분기를 `this.getPlayhead()`/`this.seek()`로 통합, **정지 상태에서도 스냅**되도록 `isPlaying` 게이트 제거. [audio-engine.js](audio-engine.js) 엔진 메서드도 동일 적용. → **Repeat 토글 순간**을 커버.
  - **2차 (v1.15.2) — 구간 드래그 종료 시 1회 스냅**: repeat가 **이미 켜진 상태에서 구간을 드래그/이동(`setLoopRange`)** 하는 경로는 토글이 아니므로 1차로는 안 잡힘. (앞서 `handleNativeMessage` wrap에 `offset < start`를 넣어 상시 교정하려던 시도는, 드래그 중 매 위치 프레임마다 seek가 발동해 SoundTouch 그레인이 반복 재시작되며 **'징~~~' 잡음·재생 지연**을 유발 → 원복.) → 대신 브리지에 `snapPlayheadToLoop()` 신설: **드래그 종료(mouseup) 시 딱 한 번**, **raw(비-clamp) 위치**로 구간 밖인지 판정해 밖이면 `seek(loopRange.start)`. 일반 사용자 seek와 동일한 1회 seek라 잡음 없음. [ui-mixer.jsx](ui-mixer.jsx) 루프 드래그 `up` 핸들러에서 `moved && repeatPlayEnabled`일 때 호출.
  - 대상: [audio-bridge.js](audio-bridge.js)(`setRepeatPlayEnabled`, `snapPlayheadToLoop`), [audio-engine.js](audio-engine.js)(엔진 메서드), [ui-mixer.jsx](ui-mixer.jsx)(드래그 종료 훅). ui-mixer는 `build/`로 컴파일되므로 `npm run build:renderers` 재빌드 반영. 네이티브 재빌드는 불필요.

   - 믹서 INIT/SYNC 상태에 `needsAudio`를 포함했다.
   - 믹서의 Solo/Mute 버튼은 no audio 트랙에서 disabled 처리한다.
   - 믹서 메시지로 직접 solo/mute 변경이 들어와도 메인 앱과 믹서 mock DAW 양쪽에서 무시하도록 방어했다.
4. 믹서 볼륨 페이더 스케일을 메인 앱과 맞췄다.
   - `Fader`에 `scale="linear"` 옵션을 추가했다.
   - 믹서 채널 VOL과 MASTER VOL은 `max=2`, `scale="linear"`를 사용해 0dB(1.0 gain)가 중앙에 표시된다.
5. Audio Input 채널의 시각 구분을 texture 기반으로 확장했다.
   - Settings > Color Theme 아래에 Audio Input Texture 선택 메뉴를 추가했다.
   - None, Diagonal, Dots, Brushed, Edge Lines 텍스처를 제공한다.
   - 중앙은 오목하게 들어간 듯한 none 계열 배경을 유지하고, 가장자리/top/bottom 위주로 텍스처와 bevel 효과를 적용한다.
   - 어두운 color theme에서 과하게 밝아지지 않도록 배경과 선/점 색을 theme 배경에 가까운 짙은 톤으로 조정했다.
6. 상단 메뉴 우측의 중복 `+ Audio In` 버튼을 제거했다.
   - 좌측 상단의 기존 `+ Audio In` 버튼만 남겨 중복 진입점을 정리했다.
