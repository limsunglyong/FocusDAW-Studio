## v1.36.7 — 루프 녹음 중 현재 녹음 pass를 빨강으로 구분 (사용자 보고) (2026-07-19)

> **재빌드: renderer만**([ui-tracks.jsx](ui-tracks.jsx) esbuild). **엔진·네이티브 무변경.** 사용자 보고("Take B로 넘어갈 때 Take A가 안 흐려진다").

### 확인 — v1.36.6 흐림은 정상, 겹침 색이 원인

`drawLiveRecording` 로직을 떼어내 합성 2.5-pass 데이터로 렌더 검증한 결과, **이전 pass는 정상적으로 흐려짐**(헤드 오른쪽 흐림, 왼쪽 진함). 문제는 모든 pass가 **같은 구간에 wrap**되어 현재 녹음 pass가 이전 pass **위에 같은 색으로** 겹쳐 그려지는 것 — playbar **왼쪽의 밝은 부분(=지금 녹음 중인 새 take)** 을 "이전 take가 안 흐려짐"으로 오인하게 됨.

### 수정 — 현재 pass = 녹음 빨강

- 완료된 이전 pass는 **흐린 트랙색(α .2)**, **현재 녹음 중 pass는 녹음 빨강(`#df5b52`, α .95)** 으로 그린다(빨간 진행 헤드와 동일 색). DAW 표준(녹음 중 테이크=빨강)이라 "빨강=지금 녹음 / 흐린 트랙색=이전 take"로 즉시 구분. 흐림 판정(`curPass = floor(경과초/passDur)`) 로직 무변경.

## v1.36.6 — 루프 녹음 중 완료된 pass(Take A…)도 흐리게 (사용자 보고) (2026-07-19)

## v1.36.4 — 곡 끝 부근 구간에서 녹음 중 Take 배지가 안 보이던 문제 수정 (사용자 보고) (2026-07-19)

> **재빌드: renderer만**([ui-tracks.jsx](ui-tracks.jsx) esbuild). **엔진·네이티브 무변경.** 사용자 보고(음원 끝부분에서 녹음 시 REC Take 배지 미표시).

### 원인

배지를 `left: rLoop.end*pxPerSec + 6`(구간 오른쪽 바깥)에 두는데, 레인 div가 `width: laneW(=duration*pxPerSec)` + `overflow: hidden`([ui-tracks.jsx](ui-tracks.jsx)). 구간이 **프로젝트 끝**이면 `regionRight ≈ laneW`라 배지가 laneW를 넘어 **잘림**. v1.36.3에서 루프 녹음 중 duration을 더 키우지 않게 되며 오른쪽 여유가 사라져 드러남(그 전엔 duration이 자라 우측에 공간이 있었다).

### 수정 — 오른쪽에 공간 없으면 구간 안쪽으로 flip

- `fitsRight = regionRight + 6 + EST_W(≈124) <= laneW`이면 종전대로 **구간 우측 바깥**(left-align, 오른쪽으로 뻗음).
- 아니면(=곡 끝 구간) **구간 오른쪽 끝에 우측 정렬**(`translateX(-100%)`, 왼쪽=구간 안쪽으로 뻗음) → 항상 laneW 안이라 잘리지 않음. pass 계산·라벨 로직 무변경.

### 원인 (배지가 아니라 duration 증가 로직)

`recordingPeaks` 핸들러([audio-bridge.js](audio-bridge.js))가 `recordedEnd = 녹음시작(loopStart) + 경과시간`으로 **선형** 계산한 뒤 `recordedEnd >= duration-1`이면 `duration = recordedEnd+60`으로 키운다. 루프 녹음(loop-Take/loop-punch)은 연속 WAV를 **고정 구간 [loopStart, loopEnd]에 wrap**하는데도 recordedEnd는 계속 증가 → 구간이 **곡 끝 부근**(loopStart ≈ duration)이면 매 peak 배치마다 가드가 발동해 타임라인/OUTPUT FX가 우측으로 무한 확장. loop-punch는 `session.end=0`이라 `_recordingDurationLimit`이 없어 가드가 열려 있었다. **v1.36.1 배지와 무관**(배지는 고정 `rLoop.end` 위치) — v1.30.0(loop-Take)부터 잠재하던 문제가 배지로 눈에 띈 것. T-1.36.0-1은 구간이 곡 중간이라 미발현.

### 수정

증가 가드에 **`!track._recordLoop`** 추가 — 루프 녹음이면 duration을 키우지 않는다(구간은 이미 프로젝트 안). 비루프 녹음(곡 끝 너머로 실제 확장)은 종전대로 확장. loop-Take에도 동일 적용(잠재 버그 동반 수정). `node --check` OK.

## v1.36.0 — Phase 7 Stage 1: Loop-Punch Comp (구간 다중 pass 녹음 + 선택) (2026-07-19)

> **재빌드: renderer([app.jsx](app.jsx) esbuild) + [audio-engine.js](audio-engine.js)·[audio-bridge.js](audio-bridge.js)(플레인 로드).** **네이티브 무변경**(기존 루프 녹음·구간 교체 인프라 재사용). 사용자 승인(2026-07-19): 둘 다 ON 시 단일 punch → 다중 pass comp로 승격, crossfade는 보류. 시험 대기 1건([시험.md](시험.md) T-1.36.0-1). **Phase 7 착수.**

### 개요 (Punch ∩ 루프 Take)

**Punch ON + Repeat ON** 동시 상태에서 Record 시, Repeat 구간에서 **여러 pass를 연속 녹음**하고 **구간 밖(base)은 유지한 채 pass(=Take) 중 하나를 선택**. 기존 **Punch(v1.35.0) + 루프 Take(v1.30.0) + Take Lanes(v1.31.0)** 조합 — 신규 building block 없음. 캡처는 루프 Take와 동일(repeat-play ON 유지, 연속 1개 WAV, 수동 정지, `_recordLoop` wrap 프리뷰), 단일 punch의 repeat-off·자동 punch-out은 미적용.

### 트리거 매트릭스 ([app.jsx](app.jsx) `startRecordFlow`)

- 신규 `loopPunch = isPunchOn() && DAW.repeatPlayEnabled && region>0.05`.
- `punch`를 `isPunchOn() && !DAW.repeatPlayEnabled && region`로 **축소**(단일 punch = Punch만 켜짐).
- `loopTake`는 `!isPunchOn() && repeatPlayEnabled && region`.
- 결과: **둘 다 ON = Loop-Punch Comp** / Punch만 = 단일 punch / Repeat만 = 루프 Take / 둘 다 OFF = 일반. (v1.35.3에서 "둘 다 ON=단일 punch"였던 것을 comp로 승격.)
- 세션/`recordingRef`에 `loopPunch` 플래그 추가. start/end/loopStart/loopEnd·`_recordLoop`·자동정지 monitor는 `(loopTake || loopPunch)`로 공유. finalize는 `active.loopPunch`면 `attachLoopPunchRecording` 호출(loop offset shift는 루프 Take와 동일).

### 엔진 ([audio-engine.js](audio-engine.js) `attachLoopPunchRecording`)

- 연속 WAV를 **1개 source**로 등록, `passDur=loopEnd-loopStart`로 nFull+partial pass 산출(`attachLoopRecording`과 동일 회계).
- **(1) 구간 비움 + base 분리**: `[in,out]`을 **활성 클립에서** 비우고(deleteRange의 split/trim 규칙), 살아남는 **base 조각은 `takeId=null`(항상 렌더)로 전환**. 시작점이 이동하는 조각은 `recordedStart/recordedOffsetMs` null(Cal. 오해 방지). 비활성 take는 손대지 않음.
- **(2) pass별 take**: 각 pass를 `[in,out]` 한 클립짜리 take로 등록(활성=마지막 full pass). `_isActiveClip` 규칙상 **base(null) 항상 + 활성 take 구간 클립만** 렌더 → Take Lanes에서 선택 전환 시 **재-splice 없이** 구간만 교체.
- `_reindexClips`+`_ensureBaked`. 브리지 `attachLoopPunchRecording`가 재베이크된 활성 레이아웃을 native로 sync. `pushUndo`로 원본 복구.

### UI

Take Lanes가 이미 구간 스코프로 동작(`setActiveTake`/`deleteTake`/`getTakeLanes` 재사용) — **신설 없음**. Punch 버튼 title에 "Punch + Repeat = Loop-Punch Comp" 안내 추가(Repeat 상태를 라이브로 읽어 3분기: comp / 단일 punch / off).

### 검증

## v1.35.3 — 구간의 PUNCH 표시를 Repeat 버튼과 분리 (사용자 요청) (2026-07-19)

> **재빌드: renderer만**([ui-mixer.jsx](ui-mixer.jsx) esbuild). **엔진·네이티브 무변경.** 사용자 요청(v1.35.1 후속).

### 문제 (v1.35.1)

구간 버튼 하나가 `Repeat ON`/`PUNCH`/`Repeat OFF`를 겸했다. **PUNCH가 표시된 상태에서 누르면** onClick이 repeat-play를 토글해 **`Repeat ON`으로 바뀌어** 버렸다(punch 표시가 사라짐).

### 수정 — PUNCH = 별도 비interactive 텍스트

- Repeat 버튼은 **원래대로 복원**(`Repeat ON`/`Repeat OFF` 토글만; punchRole 분기 제거).
- 그 **위에 "PUNCH" 텍스트**(버튼 아님, `pointerEvents:none`)를 추가: **punch 모드 ON = 진하게(앰버·bold·glow), OFF = 흐리게**(`--faint`, opacity .5). `focusdaw-punch`를 매 렌더 읽어 라이브 반영.
- 결과 동작(사용자 명세대로): ① PUNCH 클릭 불가(=Repeat ON으로 안 바뀜) ② 구간 선택 시 Repeat 버튼 위 PUNCH 글자 ③ Repeat 버튼 누르면 Repeat ON ④ 그 상태로 Play=repeat play ⑤ PUNCH 글자는 punch 켜짐=진하게/꺼짐=흐리게.

### 로직 확인 (사용자 문의)

Punch(**녹음** 동작)와 Repeat-play(**재생** 동작)는 **독립**이라 충돌 없음. 매트릭스: **Punch ON → Record=punch**(repeat 상태 무관, 녹음 중에만 repeat-play 잠시 off 후 복원) · **Punch OFF+Repeat ON → Record=루프 Take** · 둘 다 OFF → 일반 녹음. ⚠️ 단, **Repeat+Punch 동시 켜짐에서 Record는 단일 punch**이며 "여러 pass 중 선택(loop-punch comp)"은 **Phase 7**.

## ⚠️ 상시 개발 노트 — 버퍼 인덱싱 이원성 (전략 B, 반드시 숙지)

> Phase 5에서 도입된 **레이아웃 베이크(전략 B)**의 결과로 생긴 **상시 제약**이다. 완료 기록은 아카이브에 있으나, 이 노트는 이후 모든 파형/재생/피크 코드에 계속 적용되므로 여기에 남긴다.

전략 B에서 `track.buffer`는 두 가지 인덱싱 체계 중 하나이며, **`track._layoutBaked` 플래그가 이를 구분한다.** 이 둘을 혼동하면 **이동한 클립이 빈 파형으로 그려지는 버그**가 재발한다(2026-07-14 실제 발생, 수정 완료).

| 상태 | 버퍼 | 오디오 위치 | 파형 샘플링 기준 | 재생 |
|---|---|---|---|---|
| `_layoutBaked = true` (baked) | 평탄화 렌더 결과 | **타임라인 위치**(`clip.start`) | `clip.start` 기준(타임라인) | `_offset`(플레이헤드)로 그냥 재생 |
| `_layoutBaked = false` (raw/trivial) | 원본 디코드 버퍼 | 소스 원점 | `clip.offset` 기준(소스) | 동일(정렬돼 있어 무관) |

- `_ensureBaked`가 bake하면 `_layoutBaked=true`, trivial로 raw 복원하면 `false`로 설정한다.
- [ui-tracks.jsx](ui-tracks.jsx) `Waveform.bucketAt`는 `track._layoutBaked`로 분기해 `bufPos`를 **타임라인(clip.start) vs 소스(clip.offset)** 로 계산한다. peaks 배열도 `_ensureBaked`가 baked 버퍼에서 재계산하므로 타임라인 인덱싱이다.
- **새 파형/재생/피크 코드를 추가할 때**: baked 버퍼를 `clip.offset`로 접근하면 안 된다(그 위치는 침묵). 반드시 `_layoutBaked`를 확인하고 baked면 타임라인 위치로 접근.
- 재접속(`_assignDecodedToTrack`)은 raw 디코드 후 non-trivial이면 `_ensureBaked`를 호출해 재시작 후에도 이 불변식을 복원한다.
- 관련: `_rawBuffers[sourceId]`(원본 보관), `_isTrivialLayout`(bake 스킵 판정), `clip.gain`은 baked 버퍼에 반영·`clip.params.volume`/automation은 합성커브(이중적용 방지).

## ⚠️ 상시 개발 노트 — 렌더러는 번들이 아니라 **전역 스코프 공유** (반드시 숙지)

> 2026-07-17 세션에서 **실제로 앱을 죽인** 함정이다(v1.22.0). 빌드가 잡아주지 않으므로 노트로 남긴다.

`scripts/build-renderers.js`는 esbuild **`transform`만** 하고 **번들링하지 않는다.** 결과물은 [studio.html](studio.html)에서 각각 `<script>`로 로드되므로 **모든 렌더러(`ui-kit` · `ui-tracks` · `ui-mixer` · `ui-dialogs` · `ui-help` · `app`)가 하나의 전역 스코프를 공유**한다.

- **최상단 `const`/`let` 이름이 겹치면 `Identifier 'X' has already been declared`로 앱 전체가 죽는다.** 빌드는 통과한다. (v1.22.0에서 `EMPTY_CLIP_IDS`를 app.jsx·ui-tracks.jsx 양쪽에 선언해 발생 — 정적 검사로 발견.)
- **최상단 `function`이 겹치면 오류 없이 조용히 덮인다** — 로드가 늦은 쪽이 이긴다. 더 찾기 어렵다. (실제 사례: `buildInputPortOptions`가 ui-tracks·ui-mixer에 중복 정의 → 항상 mixer 버전이 이김. 위 "미처리 발견" 참조.)
- **로드 순서**(studio.html): `ui-kit` → `ui-tracks` → `ui-mixer` → … → `app`. 그래서 app.jsx는 ui-tracks의 전역을 그대로 읽을 수 있다(의도적으로 쓰는 경우 주석으로 명시할 것).
- **새 최상단 심볼을 추가하기 전 이름 충돌을 확인할 것**:
  ```bash
  node -e "const fs=require('fs');const n=f=>[...fs.readFileSync(f,'utf8').matchAll(/^(?:const|function)\s+([A-Za-z0-9_\$]+)/gm)].map(m=>m[1]);const F=['build/app.js','build/ui-tracks.js','build/ui-kit.js','build/ui-mixer.js','build/ui-dialogs.js','build/ui-help.js'];const s={},d=[];for(const f of F)for(const x of n(f)){if(s[x])d.push(x+' ['+s[x]+' vs '+f+']');else s[x]=f}console.log(d.length?d:'없음')"
  ```

- 공용으로 쓸 심볼은 **[ui-kit.jsx](ui-kit.jsx)에 1곳만** 두는 것이 원칙(가장 먼저 로드됨).

## ⚠️ 상시 개발 노트 — Undo 스냅샷 정합성 (반드시 숙지)

> 2026-07-17 세션에서 이 규칙 위반으로 버그 3건(v1.23.2 · v1.23.3)이 나왔다. 새 편집 UI를 붙일 때마다 적용된다.

판정 기준은 하나 — **`getSnapshot()`이 그 상태를 담는가?**

| 상태 | 스냅샷 | `onBeforeChange`/`pushUndo` |
|---|---|---|
| `master`(volume·reverb·echo·fadeIn·fadeOut·room·roomParams·saturation·widener·exciter), `eqBands`, `tempo`, `duration`, 트랙 `params`·`clips`·`sources`·`takes` | **담음** | **필수** — 없으면 그 기능은 Undo 불가(v1.23.3 fade) |
| 플레이헤드, `loopRange`, `repeatPlayEnabled`, `masterFxBypassed` | **안 담음** | **금지** — 100% no-op 항목이 되어 **다음 Ctrl+Z를 먹고 Redo까지 지운다**(v1.23.2 OUTPUT FX / v1.23.3 Repeat 버튼) |

- **드래그 핸들러는 mousedown이 아니라 "첫 실제 변경 직전"에 1회만** 스냅샷(클릭만 했을 때 no-op 항목이 생기지 않게). 값 비교 후 같으면 조기 반환하는 것이 정석 — 각 컴포넌트의 **휠 핸들러가 원래부터 `if (nv === value) return;`으로 이 패턴**을 쓴다. 단 `Fader`·`SleekSlider`는 mousedown에서 클릭 위치로 값을 점프시키므로 그 자리의 스냅샷이 정당하다.
- **편집이 아무것도 바꾸지 않았으면 `cancelUndo(token)`** 으로 되돌린다. `pushUndo()`는 **redo 스택을 비우므로** `undoStack.pop()`만 하면 **Redo가 조용히 죽는다**(v1.23.1).
- **가드**: `undo()`가 복원 전후 `snapshotSig`를 비교해 같으면 `console.warn("[undo] no-op entry ...")`. 정상 동작 시 무출력이므로, **이 경고가 보이면 위 규칙을 어긴 핸들러가 있다는 뜻.**

- 🐛 **Consolidate(떨어진 클립 Join) source가 저장/재열기에서 소멸**: `_consolidateClips`가 합친 오디오를 `filePath:null`(메모리 raw)로 만드는데 `importProject`는 primary `filePath`에서만 재로드 → 재열기 시 그 clip이 primary 오디오로 폴백(엉뚱한 소리). 라이브 세션은 `_rawBuffers`가 살아 정상, **저장/재열기·네이티브 재연결에서만** 발생. 근본 수정 = Phase 7 **"Consolidate Clips가 새 WAV로 렌더"**. 그 전까지 한계 = "떨어진 클립 Join은 저장 전에만 신뢰". (상세 [앱개발-아카이브.md](앱개발-아카이브.md) 이월 항목)
- `clip.gain`은 baked WAV로 native에 정상 반영됨(문제 아님).

### 사용자 결정으로 종결한 항목 (구현 안 함)

- **clip 단위 volume / automation UI** → **전체 트랙 automation으로 대체(사용자 결정 2026-07-20)**. clip 단위 조작 UI는 신설하지 않는다. 엔진의 `setClipParam`/`clip.params`는 미사용 상태로 유지(호출 UI 없음), `clip.gain`만 baked로 반영. 이에 따라 **clip-automation의 native 미표현 문제도 함께 무효**(발동 경로 없음).
- **micro fade 시각화** → **불필요(사용자 결정 2026-07-20)**. 엔진 bake에 3~5ms로 이미 반영되어 소리에는 나타나므로 UI 표시는 하지 않는다.
- **Input source 선택(mic/line/Hi-Z)** → Skip(WASAPI 한계, ASIO 필요 — 사용자 협의).

### 향후 개발 후보

**우선순위 높음**
- **성능**: 대용량/다트랙 peak 캐시·Web Worker 디코딩·automation 곡선 memoize·스크롤/줌. 전략 B bake가 편집마다 트랙 전체를 재렌더하므로 긴 트랙 편집 반응 속도 계측 필요. 장시간 세션 메모리·Export 캐시 무효화 정책 점검.
- **장시간 녹음 실측(30분+)**: 설계상 충족(ThreadedWriter+FIFO)하나 실측 기록 없음 — 필요 시 스트레스 시험.

**중간**
- **트랙 재정렬**: 트랙 헤더 드래그로 순서 변경(클립 드래그 이동은 Phase 5 완료).
- **자동 업데이트 + CI**: `electron-updater` + GitHub Releases, `electron-builder --publish`.
- **리버브 IR 개선**: 절차적 노이즈 IR → 실측 IR 로딩/프리셋.

**검토**
- **EQ 밴드 표현 통일**: OUTPUT FX 3밴드 vs Mixer Master/Advanced EQ 9밴드.
- **VST3 / 고급 미터링**(True Peak, LUFS) 장기 조사.
- **Cloud Sync**(.focus + stem을 클라우드 동기화) 장기.
- **Auto-tune(피아노 롤 피치 편집)**: 대형 신규 모듈, 착수 전 결정 필요(모노포닉 범위·피치 엔진·오프라인 렌더). 상세 아카이브 v1.21.0 Q5.

### 보류

- **split/join 툴바 버튼**만 숨김(우클릭 메뉴·단축키 C/J는 노출). 재노출 시 [app.jsx](app.jsx) `<ToolBar tool={tool} setTool={setTool} />` 주석 해제.
- **변동 BPM / Tempo Map 분석**: 현 전체곡 비례 스케일링에서는 불필요. 비트 그리드/워프 편집 도입 시 재검토.
- **AudioWorklet 기반 실시간 DSP 재시도**: 현 JUCE + WebAudio fallback 동작으로 즉시 착수 안 함.
