## ⚠️ 상시 개발 노트 — 버퍼 인덱싱 이원성 (전략 B, 반드시 숙지)

> Phase 5에서 도입된 **레이아웃 베이크(전략 B)**의 결과로 생긴 **상시 제약**이다. 완료 기록은 아카이브에 있으나, 이 노트는 이후 모든 파형/재생/피크 코드에 계속 적용되므로 여기에 남긴다.

전략 B에서 `track.buffer`는 두 가지 인덱싱 체계 중 하나이며, **`track._layoutBaked` 플래그가 이를 구분한다.** 이 둘을 혼동하면 **이동한 클립이 빈 파형으로 그려지는 버그**가 재발한다(2026-07-14 실제 발생, 수정 완료).

| 상태 | 버퍼 | 오디오 위치 | 파형 샘플링 기준 | 재생 |
|---|---|---|---|---|
| `_layoutBaked = true` (baked) | 평탄화 렌더 결과 | **타임라인 위치**(`clip.start`) | `clip.start` 기준(타임라인) | `_offset`(플레이헤드)로 그냥 재생 |
| `_layoutBaked = false` (raw/trivial) | 원본 디코드 버퍼 | 소스 원점 | `clip.offset` 기준(소스) | 동일(정렬돼 있어 무관) |

- `_ensureBaked`가 bake하면 `_layoutBaked=true`, trivial로 raw 복원하면 `false`로 설정한다.
- [ui-tracks.jsx](ui-tracks.jsx) `Waveform.bucketAt`는 `track._layoutBaked`로 분기해 `bufPos`를 **타임라인(clip.start) vs 소스(clip.offset)** 로 계산한다. peaks 배열도 `_ensureBaked`가 baked 버퍼에서 재계산하므로 타임라인 인덱싱이다.
- **새 파형/재생/피크 코드를 추가할 때**: baked 버퍼를 `clip.offset`로 접근하면 안 된다(그 위치는 침묵). 반드시 `_layoutBaked`를 확인하고 baked면 타임라인 위치로 접근.
- 재접속(`_assignDecodedToTrack`)은 raw 디코드 후 non-trivial이면 `_ensureBaked`를 호출해 재시작 후에도 이 불변식을 복원한다.
- 관련: `_rawBuffers[sourceId]`(원본 보관), `_isTrivialLayout`(bake 스킵 판정), `clip.gain`은 baked 버퍼에 반영·`clip.params.volume`/automation은 합성커브(이중적용 방지).

## [Phase 6] 루프 Take 녹음 — 착수 전 코드 조사 결과 — 2026-07-15

> [개발-프로세스.md](개발-프로세스.md) Phase 6 스펙을 현재 코드와 대조한 결과. **완료 기준을 현 구조로는 충족할 수 없는 항목(F1)이 있어 선행 과제가 필요하다.** 아래 F1~F6은 사실 조사, Q1~Q4는 착수 전 결정 대기 문의.

### F1. 루프 구간 반복이 sample-accurate가 아니다 — **최대 리스크 · 선행 과제**

현재 Repeat(구간 반복)은 **두 엔진 모두 JS 폴링 + seek 되감기**로 구현돼 있고, **네이티브 엔진에는 loop range 개념 자체가 없다**.

| 경로 | 구현 | 정밀도 |
|---|---|---|
| 웹 | [audio-engine.js](audio-engine.js) `_loop()` — `requestAnimationFrame`으로 `raw >= loopRange.end` 감지 → `_offset = loopRange.start` 후 `_stopSources()` + `play()` **재시작** | rAF ≈ 16 ms + 소스 재시작 지연 |
| 네이티브 | [audio-bridge.js](audio-bridge.js) `startLocalTickLoop()` — `setInterval` **30 ms** 폴링으로 `getPlayhead() >= loopRange.end` 감지 → `Bridge.seek(loopRange.start)`. 추가로 `position` 이벤트 수신 시에도 동일 되감기 | 30 ms 폴링 + WebSocket seek 왕복 |
| 네이티브 C++ | `setLoop`/`isLooping`은 **곡 전체 반복**(`readerSource->setLooping`)뿐. `grep`상 `loopRange`/`loopStart`/`loopEnd` **부재** | 구간 반복 미지원 |

- 게다가 네이티브 playhead는 [audio-bridge.js](audio-bridge.js) `getPlayhead()`에서 **`Date.now()` 벽시계 외삽**으로 추정한다(`nativeState.offset + elapsed * rate`).
- **영향**: Phase 6 완료 기준 3개 — "각 Take 길이가 루프 sample 수와 일치", "반복 녹음이 길어져도 누적 밀림이 없다", "루프 경계가 오디오 블록 중간에 있어도 Take가 정확히 나뉜다" — 를 **현 구조로는 원리적으로 충족 불가**. 매 회차 길이가 폴링 지터(0~30 ms+)만큼 달라지고 회차마다 밀림이 누적된다.
- **결론**: Phase 6는 **"네이티브 엔진에 sample-accurate loop range 도입"을 Stage 0 선행 과제**로 두어야 한다. 오디오 콜백 안에서 루프 끝을 샘플 단위로 감지해 블록을 경계에서 분할·되감기하고, JS 폴링 되감기는 native 연결 시 비활성화한다. 이렇게 하면 **같은 콜백에서 녹음 버퍼도 경계 분할**할 수 있어 Take 분할이 자연히 sample-accurate가 된다. (작업원칙 6 "웹/네이티브 결과 일치"와 Phase 6 완료 기준의 관계는 Q1 참조.)

### F2. `attachRecording`이 source/clip을 통째로 교체한다 — Take 누적 불가

- [audio-engine.js](audio-engine.js) `attachRecording`은 `track.sources = [{...}]`, `track.clips = [...]`를 **배열째 재대입**한다(1 녹음 = 1 source + 1 clip). 기존 Take가 있으면 사라진다.
- Phase 6은 회차별 `Take A/B/C` **누적**이 필요 → `attachRecording`을 **append 기반**으로 재구성하고 `track.activeTakeId`(또는 clip별 `takeId`) 모델을 도입해야 한다.
- 부수: `attachRecording`은 `track.buffer = decoded.buffer`를 raw로 넣고 bake하지 않는다. Take 도입 시 `_ensureBaked` 경로와 함께 정리 필요(상시 노트 참조).

### F3. Take Lane과 전략 B 베이크 모델의 정합 문제

- 전략 B는 **트랙당 평탄화 버퍼 1개**를 전제한다. 반면 Take는 **같은 시간대에 여러 대체 후보**가 존재한다 → 그대로 bake하면 Take들이 서로 겹쳐 섞인다.
- 필요한 규칙: **활성 Take의 clip만 bake 대상**, 비활성 lane clip은 데이터로만 보존하고 bake·재생·Export에서 제외. Phase 5의 "같은 트랙 내 clip 겹침 금지"는 **활성 레인 내부에만** 적용되도록 판정 범위를 좁혀야 한다(비활성 Take는 겹쳐도 정상).
- 이 규칙이 서면 Phase 7 Comp(활성 Take 선택 + 구간별 조합)도 같은 모델 위에 얹힌다.

### F4. count-in은 이미 있다(무음·박자 무관), 없는 것은 **클릭 소리**다 — 2026-07-15 정정

> ⚠️ 최초 조사에서 "count-in 자체가 없다"고 잘못 적었다. 실제로는 **3→2→1 count-in이 이미 구현·문서화·시험까지 되어 있다.** 아래가 정확한 현황.

- **기존 count-in**([app.jsx](app.jsx) `beginRecordFlow`): 정지 상태에서 Record → 화면 중앙 **3→2→1 시각 오버레이** → 끝나면 `play`+`record` 동시 시작. 재생 중 Record는 count-in 없이 즉시 녹음. count-in 중 Record/Stop/Play 누르면 `cancelCountIn()`으로 취소.
  - 관련 상태: `recordCountRef`(interval id), `recordCount`(오버레이 숫자), `window.__recordCountdownActive`(ARM 잠금 등이 참조), `prevLoopRef`(count-in 중 Repeat off → 취소 시 복원).
  - 문서·시험 존재: [ui-help.jsx](ui-help.jsx) 매뉴얼에 "3 → 2 → 1 count-in" 명시 + 스크린샷 `manual/live-screens/49-record-countin.png`, v1.20.11 시험(카운트인 중 ARM 잠금)에도 등장.
- **없는 것 = 클릭 소리**: 앱 코드(`*.js`/`*.jsx`/`juce_skeleton/Source`)에 metronome/click 구현 **부재**(검색 결과는 전부 `third_party/JUCE`의 AAX SDK 헤더). 기존 count-in은 **완전 무음**이다.
- **한계**: `setInterval(..., 1000)`으로 도는 **1초 간격 3회 고정** → **BPM/박자와 무관**하다. 연주자가 진입 타이밍을 귀로 잡을 수 없다.
- → Q2는 "count-in 신설"이 아니라 **"기존 count-in에 클릭 소리 + 박자 개념을 얹는 확장"**이다.

### F5. 녹음은 네이티브 전용이다 (제약이자 기회)

- 웹 엔진에 `getUserMedia`/`MediaRecorder` **없음**. 녹음은 JUCE `InputRecorder`(writer thread + `ThreadedWriter` FIFO) 전용.
- 즉 **루프 Take 녹음은 native 연결이 전제**이고, 웹 fallback에서는 불가(현재 단일 녹음과 동일한 제약).
- 뒤집어 보면 native가 **recorder이자 transport**이므로, F1을 네이티브에서 해결하면 루프 경계 분할·Take 생성이 전부 한 콜백 안에서 sample-accurate하게 처리된다. 이게 F1을 선행 과제로 두는 근거이기도 하다.

### F7. 웹 엔진의 실제 역할 — "웹 버전"이 아니라 앱 내부 계층 (2026-07-15 확인)

> 배포 방침(네이티브 Electron 기준, 웹 배포판 없음) 확정에 따라 "웹 엔진을 걷어낼 수 있는가"를 조사한 결과. **결론: 출력 경로로서는 이미 포기된 상태이나, 코드는 제거 대상이 아니다.**

- **출력으로서의 웹은 이미 비활성**: 네이티브 연결·핸드오버 완료 시 [audio-bridge.js](audio-bridge.js) `activateNativeOutput()`이 `LocalDAW.setOutputMuted(true)`로 음소거한다. 들리는 소리는 전부 네이티브다. → Q1 결정은 이 현실 위에 서 있다.
- **그러나 `audio-engine.js`는 필수 계층**:
  1. **모델의 진실원본** — `window.DAW`는 `DAWProxy`이고, Bridge에 없는 모든 속성/메서드를 `LocalDAW`로 위임한다. UI 전체(`DAW.tracks`/`duration`/`loopRange`)가 여기 의존.
  2. **파형 소스** — `track.buffer`/`peaks`/`sources`/`clips`가 전부 웹 엔진에 있다. 화면 파형 = 웹 엔진 디코드 버퍼.
  3. **Phase 5 baker** — `_bakeTrackLayout`이 웹 엔진에서 Float32로 bake → 브리지가 temp WAV로 native에 `loadTrack`. **네이티브는 bake를 못 한다**(평탄화 결과 재생만).
  4. **Export fallback** — `Bridge.renderMix`는 `options.forceLocal || !this.isNative`면 `LocalDAW.renderMix`로 떨어진다. 배포본 정상 경로는 native `command: "export"`.
  5. **안전망** — `switchToLocal()`(`socket.onclose`/재시도 소진)이 웹 mute를 풀어 소리를 유지한다.
- **⚠️ `switchToLocal()` 안전망은 배포본에서도 유지**한다. 네이티브가 유일 출력인 만큼 없으면 엔진 크래시 = **무음 앱**. 코드가 이미 있어 유지 비용 0.
- 반영: [개발-프로세스.md](개발-프로세스.md) 작업원칙 6을 위 내용으로 재서술(2026-07-15).

### F6. 입력 지연 보정값이 드라이버 추정치다

- v1.20.7의 latency 표시는 WASAPI **드라이버 보고 기반 근사치**(문구에도 `driver estimate` 명시).
- Phase 6은 "입력 지연 보정값을 clip `start`에 반영"을 요구 → 근사치를 그대로 쓰면 Take마다 수 ms 어긋날 수 있다. 정확도 목표 결정 필요(Q4).

### 예상 Stage 구성 (결정 반영)

- **Stage 0 — 네이티브 sample-accurate loop range** (Q1): `setLoopRange` 명령 + 콜백 내 샘플 단위 경계 분할·되감기, native 연결 시 JS 폴링 되감기 비활성화. **Phase 6의 실질적 1단계.**
- **Stage 1 — 녹음 상태머신**: `idle → armed → countIn → recording → stopping → finalizing → armed`, 구간 지정 녹음 `startRecording({ trackId, start, end, loopEnabled, countIn })`.
- **Stage 2 — 메트로놈 클릭** (Q2): **기존 3→2→1 무음 count-in 확장** — native 클릭 생성 + On/Off UI + 모니터 전용 라우팅(녹음 파일 미포함) + 1초 고정 → BPM 박자 전환 + Help 매뉴얼·스크린샷 갱신. 기존 취소/ARM 잠금/Repeat 복원 동작 유지.
- **Stage 3 — Take 데이터 모델** (Q3): `attachRecording` append 기반 재구성(F2) + `activeTakeId` + 활성 clip만 bake + 겹침 판정 범위 축소. 회차별 `Take A/B/C` 생성, Stop 시 불완전 회차는 `Partial` 보존.
- **Stage 4 — Take Lanes UI**: 펼치기/접기, 활성 Take 선택.
- **Stage 5 — 지연 보정** (Q4): 추정치 기본 + 수동 오프셋 UI → clip `start` 반영.
  - ⏰ **착수 직전 사용자 시험 요청 필수**: **T-1.20.10-2(b)**(입력 미사용 프로필에서 시작 시 마이크 미개방). Stage 5가 **Settings ▸ Audio Devices**를 수정하는데 이 패널이 `focusdaw-audio-input` 키를 생성하는 주체([audio-bridge.js](audio-bridge.js) `AUDIO_INPUT_KEY`)라, 먼저 기준선을 잡지 않으면 이후 실패 시 원인 분리가 불가능하다. 상세 [시험.md](시험.md) "예약된 시험 알림".
- **Stage 6 — Punch Recording**: 루프 Take 안정화 후, 같은 인프라 위에 "단일 구간 클립 교체 + 자동 punch-in/out"으로 마무리(Phase 5 `deleteRange` 훅 재사용).
