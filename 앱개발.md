## 2. 새로운 앱의 목표

v1.0.0 단계의 핵심 목표는 오디오 트랙별 **시간(Tempo/BPM) 및 주파수(Pitch) 제어 기능**을 고도화하여, Stem 믹싱 중 템포 매칭과 조바꿈(Key Change) 편집을 지원하는 것입니다.

### 2.1 BPM 측정 기능 (BPM Detection)
- 임포트된 개별 오디오 파일(Stem)의 BPM을 자동으로 분석하여 검출하고 표시합니다.
- 사용자가 기준 템포를 맞출 수 있도록 수동 탭 템포(Tap Tempo) 기능을 함께 제공합니다.

### 2.2 트랙별 Pitch-preserving BPM 변경 편집 기능 (Time Stretching)
- 각 오디오 트랙의 음정(Pitch)을 원래대로 유지하면서, 재생 속도(BPM / Tempo)만 독립적으로 늘리거나 줄일 수 있는 편집 기능을 구현합니다.
- 이를 통해 서로 다른 템포의 Stem 파일들을 하나의 프로젝트 BPM에 동기화할 수 있게 합니다.

### 2.3 트랙별 BPM-preserving Pitch 변경 편집 기능 (Pitch Shifting)
- 각 오디오 트랙의 재생 속도(BPM)를 일정하게 유지하면서, 음정(Pitch)만 반음(Semitones) 단위로 조절할 수 있는 편집 기능을 구현합니다.
- 특정 Stem의 Key를 프로젝트 전체 Key에 맞추거나 리팩토링할 수 있도록 지원합니다.

## 3. 개발 방향 및 세부 계획 (Plan)

Web Audio API의 기본 `AudioBufferSourceNode.playbackRate` 조절은 템포와 피치가 동시에 변하는 한계가 있습니다. 이를 해결하기 위해 DSP(디지털 신호 처리) 알고리즘을 도입하고, 실시간성 및 오프라인 렌더링(Export) 무결성을 확보하기 위해 다음과 같이 단계별 개발 계획을 수립합니다.

### Phase 1: BPM 측정 엔진 및 UI 구현
- **알고리즘 구현**:
  - `OfflineAudioContext`를 사용하여 임포트된 오디오 데이터를 약 150Hz 이하로 로우패스 필터링(Low-pass filter)합니다.
  - 오디오 신호의 에너지 엔벨로프(Energy Envelope)를 추출하고, 피크 검출(Peak Detection) 및 자기상관 분석(Auto-correlation)을 통해 가장 지배적인 비트 간격(BPM)을 산출합니다.
- **UI 연동**:
  - 트랙 헤더 또는 트랙 상세 인스펙터 영역에 검출된 BPM을 표시합니다.
  - 프로젝트 메인 템포 입력창 옆에 수동 `TAP` 버튼을 구현하여, 사용자가 박자에 맞춰 클릭하면 BPM이 실시간 계산되도록 지원합니다.

### Phase 2: Time-stretching / Pitch-shifting 오디오 프로세서 구현
- **기술 스택 선정**:
  - **AudioWorklet**: 실시간 오디오 처리를 위해 오디오 스레드에서 동작하는 `AudioWorkletNode`를 생성하고, 여기에 DSP 처리 코드를 탑재합니다.
  - **DSP 알고리즘**:
    - **WSOLA (Waveform Similarity Overlap-Add)**: 타임 도메인에서 오디오 신호의 유사성을 기반으로 조각을 이어 붙이는 방식. 연산량이 적고 실시간 처리에 적합하며 단성음(보컬/단일 악기)에 왜곡이 적음.
    - **Phase Vocoder**: 주파수 도메인에서 FFT/IFFT를 통해 위상을 보정하며 재합성하는 방식. 다성음(드럼 믹스, 전체 Stem 믹스)에 적합함.
    - **라이브러리 이식 검토**: Pure JavaScript로 작성된 경량 DSP 라이브러리(예: `soundtouchjs` 등)의 코어를 `AudioWorkletProcessor`로 포팅하여 연산 효율과 안정성을 높입니다.
- **엔진 통합 (`audio-engine.js`)**:
  - 트랙별 신호 체인에 `AudioWorkletNode`를 삽입합니다:
    `source → timeStretchPitchShiftNode → fader → panner → masterBus`
  - 재생 속도 비율(Stretch Ratio)과 피치 시프트 양(Semitones) 파라미터를 실시간으로 노드에 매핑합니다.

### Phase 3: 트랙별 편집 UI 적용 및 내보내기(Export) 연동
- **편집 UI 구현**:
  - 트랙 헤더(M/L 크기)에 `Pitch (Semitones, -12 ~ +12)` 및 `BPM / Tempo Stretch (Ratio 0.5x ~ 2.0x)`를 조절할 수 있는 노브 또는 슬라이더 컨트롤을 추가합니다.
  - 프로젝트 기준 BPM에 맞추어 개별 트랙의 템포가 자동으로 스트레치되도록 하는 "Sync to Project BPM" 토글 스위치를 추가합니다.
- **오프라인 렌더링 연동**:
  - `renderMix()` 함수 내 `OfflineAudioContext` 구성 시 동일하게 `AudioWorkletNode`를 생성 및 동기화하여, 내보낸 MP3/WAV 파일에도 Pitch/BPM 편집 내용이 무결하게 반영되도록 보장합니다.

## 4. 개발 예상 이슈 및 대처 방안

1. **AudioWorklet 파일 로드 이슈 (Electron/브라우저 환경)**:
   - *이슈*: 무빌드 환경에서 AudioWorklet 모듈 파일을 브라우저와 Electron 앱 모두에서 절대/상대 경로로 안전하게 로드해야 합니다.
   - *대처*: `audio-engine.js` 초기화 시 `ctx.audioWorklet.addModule('path/to/processor.js')` 경로를 Electron의 `app.getAppPath()` 및 브라우저 URL 체계에 맞게 분기 처리하거나, AudioWorklet 코드를 Blob URL로 변환하여 동적으로 로드하는 방안을 고려합니다.
2. **실시간 처리 오디오 글리치(CPU 과부하)**:
   - *이슈*: 다중 트랙에 동시에 FFT 기반 Phase Vocoder를 적용할 경우, 가비지 컬렉션(GC) 및 CPU 연산 지연으로 소리가 끊기는 현상이 발생할 수 있습니다.
   - *대처*: 비교적 가벼운 WSOLA 방식을 기본 엔진으로 채택하고, 버퍼 크기를 최적화(예: 1024 또는 2048 샘플)하며, 오디오 렌더링 루프 내에서의 메모리 재할당을 최소화하여 GC 발생을 억제합니다.
3. **오프라인 렌더링 동기화**:
   - *이슈*: `OfflineAudioContext`가 동작할 때 실시간 스케줄과 AudioWorklet 내부 버퍼 간의 정렬이 어긋날 수 있습니다.
   - *대처*: 렌더 시작 시점에 AudioWorklet 내부 상태를 완전히 초기화(Clear buffer)하고, 입력 샘플 인덱스와 출력 타임스탬프를 1:1로 정렬하는 가이드를 구현합니다.

## 5. 사용자 검토 대상 항목 (아이디어 노트)

1. **Mixer 창을 외부로 분리**
   - 현재의 인앱 플로팅 믹서 창을 실제 Electron의 독립적인 `BrowserWindow`로 완전히 분리하고, 창 간 데이터 통신(IPC 또는 BroadcastChannel)을 통해 다중 모니터 사용 환경에 적합하게 대응합니다.
2. **VST3 효과 추가**
   - 외부 VST3 플러그인을 앱 내에서 로드하고 신호 체인에 결합할 수 있는 기능을 검토합니다.
     - a. Limiter
     - b. Compressor
     - c. Stereo Imager / Widener
     - d. Exciter / Enhancer
     - e. True Peak Meter + LUFS Meter
3. **JUCE 프레임워크로 핵심 오디오 엔진 변경**
   - 현재의 Web Audio API 엔진이 지닌 성능 및 정밀도 한계를 극복하기 위해, C++ 기반의 크로스플랫폼 오디오 프레임워크인 JUCE로 핵심 오디오 엔진을 재구축하여 데스크톱 네이티브 DAW 수준의 성능을 확보하는 방안을 고려합니다.
4. **Cloud Sync (Google Drive 연동)**
   - 프로젝트 파일(`.focus`) 및 소스 오디오(Stem) 파일들을 클라우드 저장소(Google Drive 등)와 연동하여 장치 간 작업 세션을 동기화하고 백업을 용이하게 합니다.

5. 도움말(help)에서 find 기능 추가할 것.

## 6. 수정요청사항 반영 계획 및 결과 (2026.06.12)

### 1) Settings 창 믹서 초기화 영역 테두리 및 배경색 제거
- **문제점**: `SettingsDialog` 내부의 `Mixer Console Window` 리셋 카드 박스에 테두리(`border`)와 `background: var(--surface)` 스타일이 잔존하여 어두운 카드 형태로 보임.
- **수정 계획 및 결과**: `ui-dialogs.jsx`의 `SettingsDialog` 리셋 영역 인라인 스타일에서 테두리(`border`)와 배경색(`background`), 모서리 깎임(`borderRadius`)을 삭제하고 `padding`을 `8px 0`으로 조절하여 창 내부에 일체형으로 녹아들도록 스타일을 다듬음.

### 2) Settings 창 믹서 초기화 미작동 버그 수정
- **문제점**: 리셋 버튼을 클릭하여 `localStorage` 및 Electron 메인 프로세스 전역 `lastMixerBounds` 캐시를 비우더라도, 이미 믹서 창이 열려 있는 상태라면 믹서 창이 닫힐 때(`beforeunload` / `close` 이벤트) 믹서 창의 현재 위치와 크기를 다시 읽어와 `localStorage` 및 `lastMixerBounds`에 **다시 덮어씌워버려** 실제 초기화가 적용되지 않음.
- **수정 계획**:
  - **Electron**: `reset-mixer-bounds` 핸들러 호출 시 `isMixerBoundsReset = true` 플래그를 설정하고, 열려 있는 `mixerWindow`가 있다면 즉시 `close()` 처리함. `mixerWindow`의 `close` 이벤트 리스너에서 `isMixerBoundsReset`가 참인 경우 bounds 기록을 스킵하고 캐시를 `null`로 강제 유도함.
  - **웹 브라우저**: `SettingsDialog` 리셋 버튼 클릭 시 `localStorage.removeItem("focusdaw-mixer-bounds")`를 수행하며, 열려 있는 팝업 객체(`window.mixerPopup`)가 있다면 즉시 `window.mixerPopup.close()`를 수행함. 팝업이 닫힐 때 트리거되는 `saveBounds` 리스너에서는 `localStorage.getItem("focusdaw-mixer-bounds")`가 이미 비어있다면(`null`인 경우) 저장을 건너뛰고 return 하도록 예외 처리하여 캐시 덮어쓰기를 원천 방어함.

### 3) 사용자 매뉴얼 내용 일치화 및 도움말 매뉴얼 F3 단축키 추가
