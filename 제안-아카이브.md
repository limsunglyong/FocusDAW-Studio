# CloudFlare(웹) 배포 시 고려사항 / 문제점 제안 (아카이브)

> 분석 기준: v1.14.4, `feature/v3`. 이 앱은 **Electron 데스크톱 앱**으로 설계되어 있어, CloudFlare Pages 같은 **정적 호스팅**에 올리면 데스크톱 전용 기능이 빠지고 브라우저(Web Audio) fallback 경로로만 동작합니다. 아래는 그때 문제가 되거나 미리 손봐야 할 부분 정리입니다.

---

## 0. 전제 — 무엇이 "사라지는가"

웹 배포 환경에서는 두 가지 축이 통째로 빠집니다.

1. **네이티브 JUCE 오디오 엔진** — `audio-bridge.js`가 `ws://localhost:8082`로 연결을 시도하는데, 이 WebSocket 서버는 사용자 PC에서 도는 데스크톱 프로세스(`juce_skeleton`)입니다. 웹 방문자에겐 존재하지 않음.
2. **`window.electronAPI`** — `electron/preload.js`가 주입하는 IPC 브리지. 파일 시스템, 프로젝트 저장/열기, ffmpeg 인코딩/정규화, OS 저장 다이얼로그, 별도 창 관리가 전부 여기에 묶여 있음.

다행히 코드 곳곳에 `if (window.electronAPI) … else …` fallback이 있어 "동작은 함". 단, 아래 항목들은 **품질이 떨어지거나 아예 막힙니다.**

---

## 1. 네이티브 엔진 연결 시도 (영향: 낮음, 확인 필요)

- `audio-bridge.js`의 `Bridge.init()`는 `window.electronAPI`가 있을 때만 `setupWebSocket()`을 호출합니다([audio-bridge.js:38](audio-bridge.js#L38)). 즉 순수 브라우저에서는 `ws://localhost:8082` 연결을 **시도조차 안 함** → 콘솔에 retry 로그가 안 뜨고, HTTPS 페이지에서의 mixed-content 차단(`ws://`)도 자동 회피됨. ✅ 잘 짜여 있음.
- 결과적으로 웹에서는 항상 `Bridge.isNative = false`(Web Audio 경로)로 동작. 의도된 동작이 맞는지 한 번 확인만 하면 됨.

## 2. 프로젝트 오디오 "재연결" 불가 (영향: **치명적**)

- `.focus` 프로젝트 파일은 오디오 샘플을 품지 않고 **파일 경로(`filePath`)만 저장**합니다. 다시 열 때 `reconnectProjectAudio()`가 경로로 파일을 읽어 복원하는데, 이 함수는 `if (!window.electronAPI) return;`로 시작합니다([app.jsx:1979](app.jsx#L1979)).
- → **웹에서는 프로젝트를 저장했다 다시 열어도 트랙 오디오가 복원되지 않습니다.** 파형/트랙 구조만 남고 소리는 안 남.
- 대응:
  - 웹 빌드에서는 프로젝트 파일에 오디오를 **함께 임베드**(WAV/Opus base64 또는 별도 번들)하거나,
  - File System Access API로 재오픈 시 사용자가 폴더를 다시 지정하게 하거나,
  - 최소한 "이 프로젝트의 오디오는 데스크톱 앱에서만 복원됩니다" 안내를 띄울 것.

## 3. Export(믹스다운) 품질 저하 (영향: 중간)

- **MP3 인코딩**: 데스크톱은 `ffmpeg-static`, 웹은 `lamejs` fallback이 있음([ui-dialogs.jsx:436](ui-dialogs.jsx#L436)). 동작은 하나 lamejs가 더 느리고 무거움.
- **라우드니스 정규화(LUFS)**: `electronAPI.processAudio`(ffmpeg)로만 수행됩니다([ui-dialogs.jsx:429](ui-dialogs.jsx#L429)). 웹에는 대체 경로가 없어 **normalize 옵션이 사실상 무동작**. UI에서 normalize 토글도 `window.electronAPI.processAudio`가 있을 때만 노출됨([ui-dialogs.jsx:510](ui-dialogs.jsx#L510)) → 웹에선 옵션이 숨겨지므로 혼란은 적지만, 기능이 빠진다는 점은 명시 필요.
- **Export 렌더링**: `OfflineAudioContext`로 메인스레드에서 렌더([audio-engine.js:1804](audio-engine.js#L1804)). 긴 곡/다수 트랙이면 탭이 멈춘 듯 보일 수 있음(아래 7번 참고).
- 저장은 웹에선 `URL.createObjectURL` + `<a download>`로 처리되어 OK.

## 4. 별도 창(믹서 / Advanced Pan·EQ·Ambience / Help) — 팝업 차단 (영향: 중간)

- 웹에서는 `window.open("mixer.html", …)` 등으로 새 창을 띄우고, 창 간 동기화는 `BroadcastChannel`로 합니다([app.jsx:1225](app.jsx#L1225), [app.jsx:1597](app.jsx#L1597)). BroadcastChannel은 같은 origin이면 잘 동작 ✅.
- 다만:
  - **팝업 차단기**에 막힐 수 있음(사용자 제스처 직후 호출이라 대체로 통과하지만 보장 안 됨). 인라인 패널 fallback 또는 안내 필요.
  - 창 위치/크기 저장(`resizeMixer`, `resetMixerBounds`, mixer bounds)은 `electronAPI` 경로라 웹에선 무시됨. 기능상 치명적이진 않음.
  - 각 팝업이 자기 `audio-engine.js`/React 번들을 다시 로드 → 첫 오픈이 무거움.

## 5. 빌드 산출물 분리 — `build/` 디렉토리 오염 (영향: **높음, 배포 위생**)

- 현재 `build/`에는 **렌더러 JS(`build/app.js`, `build/vendor/*` 등)와 CMake/JUCE 네이티브 빌드 산출물이 섞여** 있습니다(`build/JUCE/`, `*.vcxproj`, `CMakeCache.txt`, `Release/` …).
- 정적 배포 시 필요한 건 `studio.html` + `build/*.js` + `build/vendor/*` + `assets/` + 그리고 비번들 스크립트(`version.js`, `meyda.min.js`, `audio-engine.js`, `audio-bridge.js`)뿐입니다.
- `build/` 전체를 그대로 올리면:
  - 수백 MB의 불필요한 빌드 부산물 업로드,
  - 내부 빌드 경로/툴체인 정보 노출,
  - CF Pages 파일 수 제한(20,000개) 압박.
- 대응: 렌더러 출력 디렉토리를 네이티브 빌드와 분리(예: `web-dist/`)하거나, CF Pages 배포 시 명시적 include 목록으로 위 파일만 업로드. `scripts/build-renderers.js`의 `outDir`을 별도 폴더로 바꾸는 것을 권장.

## 6. 외부 의존(Google Fonts) — 오프라인/프라이버시/CSP (영향: 낮음)

- `studio.html`이 `fonts.googleapis.com`에서 Orbitron / Space Grotesk / Space Mono / Elms Sans / Tai Heritage Pro를 로드합니다([studio.html:8-13](studio.html#L8-L13)).
- 인터넷 없으면 폰트 깨짐(키/BPM 숫자 폰트 의존도 높음), GDPR류 프라이버시 이슈, CSP 작성 시 `font-src`/`style-src`에 Google 도메인 허용 필요.
- 대응: 폰트를 self-host(`assets/`에 동봉)하면 오프라인·프라이버시·CSP 모두 단순해짐.

## 7. 브라우저 런타임 제약 (영향: 중간)

- **AudioContext 자동재생 정책**: 브라우저는 첫 사용자 제스처 전까지 `AudioContext`가 `suspended`. 데스크톱 Electron보다 엄격하므로, 첫 재생 클릭에서 `ctx.resume()`이 확실히 호출되는지 확인 필요.
- **메모리**: 모든 stem을 메모리에 PCM으로 디코드. 큰 WAV 여러 개면 모바일/저사양 브라우저 탭이 OOM 위험.
- **메인스레드 블로킹**: 디코드/`OfflineAudioContext` 렌더가 메인스레드에서 돌아 긴 작업 시 UI 프리징. (현재 AudioWorklet/Worker는 미사용.)
- **MP3 import**: 브라우저 `decodeAudioData`에 의존([audio-engine.js:616](audio-engine.js#L616)) → 대부분 브라우저 OK이나 일부 코덱(예 특정 AAC/무손실)은 브라우저별 차이.

## 8. COOP/COEP — 지금은 불필요, 향후 주의 (영향: 정보)

- 현재 `SharedArrayBuffer`/`crossOriginIsolated`/`AudioWorklet`를 **쓰지 않으므로** 특별한 cross-origin-isolation 헤더가 필요 없습니다. ✅
- 단, 나중에 성능 위해 **ffmpeg.wasm**이나 SAB 기반 워커를 도입하면 `Cross-Origin-Opener-Policy: same-origin` + `Cross-Origin-Embedder-Policy: require-corp` 헤더가 필요해지고, 그러면 Google Fonts 등 외부 리소스가 CORP 위반으로 막힘 → 6번(self-host)과 연동해 미리 대비.

## 9. 보안 헤더 / CSP 권장값 (영향: 정보)

CF Pages `_headers` 파일 예시(폰트 self-host 가정):

```
/*
  Content-Security-Policy: default-src 'self'; script-src 'self' 'unsafe-inline'; style-src 'self' 'unsafe-inline'; img-src 'self' data: blob:; media-src 'self' blob:; connect-src 'self'; font-src 'self'; worker-src 'self' blob:
  X-Content-Type-Options: nosniff
  Referrer-Policy: strict-origin-when-cross-origin
```

- `'unsafe-inline'`: `studio.html`에 인라인 `<style>`/`<script>`가 있어 현재는 불가피. 장기적으로 외부 파일화 권장.
- `blob:`/`data:`: export 다운로드, 커버 아트, 오디오 URL에 필요.
- Google Fonts를 계속 쓸 경우 `style-src`/`font-src`에 `fonts.googleapis.com`/`fonts.gstatic.com` 추가.

## 10. 배포 운영 메모 (영향: 정보)

- **`.gitignore`**: `node_modules/`, `dist/` 무시 → CF Pages는 빌드 단계에서 `npm ci && npm run build:renderers` 수행하도록 설정. 단 현재 `build:renderers`는 네이티브 `build/` 폴더에 출력하므로 5번 분리 선행 필요.
- **상대경로**: `studio.html`이 모든 스크립트를 상대경로(`build/...`, `version.js` 등)로 로드 → 도메인 루트 배포면 OK. 서브패스 배포 시 `<base>` 또는 경로 조정 필요.
- **엔트리 파일명**: 진입점이 `index.html`이 아니라 `studio.html`. CF Pages 기본 라우팅상 `https://도메인/studio.html`로 접근해야 함 → `index.html`로 리네임하거나 `_redirects`로 `/ → /studio.html` 처리.
- **캐싱**: 번들 파일(`ui-help.js` ~208KB 등)은 해시 파일명이 없어 갱신 시 캐시 무효화 문제. 파일명 해싱 또는 버전 쿼리스트링 + 적절한 `Cache-Control` 권장. (CF가 brotli/gzip은 자동.)
- **데스크톱 전용 UI**: 타이틀바/메뉴/창 컨트롤이 데스크톱 가정. 모바일·터치 미대응이므로 공개 시 최소한 데스크톱 권장 안내.
